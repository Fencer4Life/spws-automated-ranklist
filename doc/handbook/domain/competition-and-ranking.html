Domain · Ranking model

Competition and ranking

The calendar is event-shaped; scoring is tournament-slot-shaped; public rankings are segmented by weapon, gender and veteran category.

Current domain modelSQL-owned scoringVerified 12 July 2026

Mental model

Competition hierarchy

Season
└── Event — one calendar occurrence, organizer, dates and lifecycle
    ├── Tournament slot — weapon × gender × V-category
    │   └── Result — canonical fencer + placement + provenance
    └── Registration — entry request and consent, separate from results

An event can exist before tournament slots. Slots are created when the competition structure or result ingestion requires them. This preserves a clean future calendar while avoiding fictional empty competitions.

Age assignment

Veteran category is season-relative

The canonical category function uses birth year and the year in which the season ends. A fencer belongs to one home category for that season: V0 30–39, V1 40–49, V2 50–59, V3 60–69 or V4 70+.

Invariant: result rows and their parent tournament slots must agree on the effective veteran category, except for explicitly tolerated unknown-birth-year cases.

Combined physical pools are normalized into category-specific tournament slots. When a joint pool shares the physical participant count, sibling slots record that shared count rather than pretending each slice fenced only its own category.

Published result

Ranking modes

Domestic ranking

Uses qualifying SPWS event families and domestic scoring configuration. Every eligible domestic result is intended to count for the correct bracket.

Extended / international ranking

Adds configured international participation for Polish fencers while applying international eligibility and minimum-participant policy.

Rolling view

For an active season, selected prior-event results may carry forward according to the season’s carry-over engine, event links and stopping rules.

Drilldown

Shows the component results and carried scores that explain a fencer’s total rather than exposing only the final number.

Responsibility

Where points are calculated

PostgreSQL functions own ranking and score calculation. The frontend reads ranking views/RPC results and renders them; Python ingestion determines which trusted results enter the model but does not become a second scoring engine.

One implementation principle: age-category and scoring policy should have one authoritative implementation at the database boundary. Python and TypeScript may display or orchestrate the result but must not silently diverge.

Across seasons

Carry-over

Seasons select a carry-over engine. The explicit model links an event to its prior counterpart through id_prior_event; legacy event-code matching remains supported for seasons configured to use it. Current-season results stop the corresponding carried contribution where defined by the ranking functions.

Carry-over windows and international toggles are season configuration, not hardcoded UI rules. Biennial event families require explicit handling rather than assuming every event repeats annually.

Admission versus scoring

Participant thresholds

Minimum participant configuration gates ingestion. A bracket below its configured threshold is skipped before it becomes ranking data; changing a threshold does not silently rewrite already committed history.

Do not conflate physical pool size, category-slice size and qualifying participant threshold. They answer different questions and are preserved separately where required.
