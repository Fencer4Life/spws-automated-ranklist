End-to-end flows

Six canonical scenarios connect user intent, automation, Python orchestration and transactional PostgreSQL behavior.

Happy path + failure boundaryPresent tenseVerified 12 July 2026

Flow 1

Browse ranklist and calendar

  1. The SPA or embedded component loads seasons and the selected public view.
  2. The frontend API queries curated Supabase views/RPCs.
  3. PostgreSQL ranking functions apply season configuration, category, eligibility and carry-over policy.
  4. The frontend renders rankings, calendar cards or score drilldown.
  5. A read failure becomes visible UI state; it never mutates domain data.

Flow 2

Register for an event

  1. The visitor opens registration from an enabled event.
  2. The registration form collects identity and required consent.
  3. A constrained public RPC validates and writes the registration.
  4. Matching logic associates the registration with a canonical fencer where possible.
  5. The public entry-list view exposes approved fields only.

Flow 3

Ingest domestic results

  1. An operator/workflow chooses a source URL, file or storage artifact.
  2. A source parser emits ParsedTournament and source rows.
  3. The rule engine plans and runs parse, validation, event resolution, identity resolution, pool handling, count checks, category assignment and commit.
  4. The bridge delegates selected plugin work to existing stage functions while maintaining the new context/fault contract.
  5. Commit persists the accepted event result set transactionally.
  6. Post-commit validation, notification and staging-report formatting run as a separate flow.
Review path: when the operator uses draft review, run-scoped draft tables and three-way comparison provide an explicit inspection step before fn_commit_event_draft.

Flow 4

Correct identity and recompute

  1. An admin confirms an alias, birth year, duplicate merge or identity correction.
  2. The database records master-data change and enqueues affected events.
  3. The recompute worker loads committed FK-linked results for an event.
  4. The rule engine reassigns effective category/counts and recommits derived state without scraping or fuzzy matching again.
  5. Post-commit checks and notifications run for the updated event.

Flow 5

Synchronize EVF and reconcile the calendar

  1. A scheduled/manual workflow fetches EVF calendar or results data.
  2. Identity keys such as EVF id/slug prevent duplicate calendar rows.
  3. CERT imports and validates source-owned event fields while preserving admin-owned fields.
  4. The calendar promotion mode compares complete active-season event sets.
  5. A guarded create/update/delete RPC reconciles PROD to CERT, resolving environment-local references by stable code.
  6. Protected results-bearing events are reported rather than deleted.

Flow 6

Release and promote

  1. CI validates database, Python, frontend and generated documentation.
  2. The release workflow builds artifacts and deploys environment-scoped schema/functions/frontend.
  3. CERT is verified before PROD actions proceed.
  4. Season, event/calendar and result promotion use separate contracts appropriate to their data.
  5. Tracking/seed exports are refreshed after successful production changes where required.
  6. Failure stops the workflow at its boundary; rollback/retry follows the runbook rather than ad hoc mutation.
