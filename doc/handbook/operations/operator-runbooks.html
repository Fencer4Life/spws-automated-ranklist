Operating principles

  • Confirm environment, season and event code before mutation.
  • Prefer event-scoped actions over bulk operations.
  • Review generated reports and divergences, not only exit status.
  • Do not advance to the next event without required human sign-off.
  • Do not force-delete results-bearing data to make parity “green.”
  • Capture workflow/run id and final outcome for handoff.

Runbook

Ingest or review one event

  1. Verify the recorded source URL/file and event identity.
  2. Run the event-scoped ingest/review command or ingest-event.yml.
  3. Inspect parse counts, pool/category handling, identity outcomes and URL/data validation.
  4. Review draft/source/CERT-reference differences where the review path is used.
  5. Resolve ambiguity with an override or identity action; rerun the same event.
  6. Commit only after the report is accepted.
  7. Verify post-commit count, score/parity and notification output.

Runbook

Synchronize EVF

  1. Run scheduled/manual EVF sync for the intended environment. The workflow reconciles calendar CERT→PROD both before and after the scrape step — the pre-scrape reconcile runs regardless of whether the scrape itself later fails, so PROD never falls further behind CERT than it already was.
  2. Review created/updated/reused event identity and duplicate warnings.
  3. Review result parity for events eligible for import/promotion.
  4. For each calendar promotion (pre- and post-scrape), inspect create/update/delete/skipped sets in its reconcile report.
  5. If a promote step refuses with an active-season mismatch, do not override it — it means CERT and PROD's bool_active seasons disagree (e.g. an admin edited a season's dt_start/dt_end on one environment but not the other). Fix the season row divergence directly, then rerun; the reconciler will pick up correctly once seasons agree.

Runbook

Repair after identity/master-data change

  1. Complete the fencer/alias/birth-year correction through the governed admin/RPC path.
  2. Confirm affected events were enqueued.
  3. Drain or observe the recompute worker: CERT and PROD both drain automatically every 15 minutes (recompute-drain.yml / recompute-drain-prod.yml, ADR-072 amendment 2026-07-14) or via manual workflow_dispatch with debounce: 0 to force an immediate check; run the documented LOCAL worker manually where appropriate.
  4. Verify affected tournament categories/counts and public ranking components.
  5. Escalate queue poison/repeated abort rather than deleting queue state blindly.

Runbook

Promote to production

  1. Confirm CI/release and CERT verification are green.
  2. Select the correct season/event/calendar promotion contract.
  3. Capture pre-operation snapshot/report where the procedure requires it.
  4. Run the GitHub workflow with exact inputs.
  5. Read the final report and verify PROD public/admin behavior.
  6. Refresh tracking/seed exports and record remaining divergence.

Runbook

Send an FTL seed bundle

  1. Complete the Gmail and GitHub Actions setup guide before the first live send.
  2. Confirm the target, exact event code, persisted organizer email, registration cutoff and current entry list.
  3. For first use, dispatch ftl-seed.yml manually to CERT and inspect the exact workflow inputs and step results.
  4. Confirm the organizer received and can import the ZIP; verify ts_ftl_sent appears in EventManager.
  5. Manual UI or Telegram send <EVENT-CODE> participants dispatches are explicit re-sends and replace the timestamp.
  6. If SMTP fails, verify no stamp exists and retry after correcting the provider/recipient issue. If SMTP succeeded but stamping failed, warn the organizer that a retry may duplicate the email.
  7. Enable repository variable ENABLE_FTL_DEADLINE_SEND=true only after pilot sign-off; disabling/removing it stops scheduled sends without removing manual delivery.
Scope: the first release sends the validated mix-all bundle only. Never paste Gmail, Supabase or Telegram secret values into logs, issues or chat.
