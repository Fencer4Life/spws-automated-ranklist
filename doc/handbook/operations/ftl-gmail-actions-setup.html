What this setup enables

The manual workflow reads an event from CERT, generates the current mix-all FTL seed ZIP in memory, sends it through Gmail over certificate-verifying TLS, and stamps tbl_event.ts_ftl_sent only after Gmail accepts the message.

ItemStored wherePurpose
SPWS Gmail addressGitHub repository secret SPWS_GMAIL_USERSMTP login and exact From address.
Google App PasswordGitHub repository secret SPWS_GMAIL_APP_PASSWORDRevocable SMTP credential; not the normal Google password.
Organizer emailCERT tbl_event.txt_organizer_email, edited through EventManagerRecipient for the selected event.
Send evidenceGitHub Actions run, organizer inbox, Telegram, and ts_ftl_sentEnd-to-end acceptance evidence.

Never use the normal Google password

Only the generated App Password belongs in SPWS_GMAIL_APP_PASSWORD. Never put either password in source files, .env, shell history, screenshots, issues, chat, or workflow logs.

The manual run sends a real email

Use an approved test inbox saved on an exact CERT event. Do not use the example event code as an assumption. Do not select PROD during the pilot.

Before touching credentials

Confirm access and choose the pilot

  • You can sign in to the exact SPWS Gmail account that should appear in the message's From header.
  • You can administer Actions secrets for Fencer4Life/spws-automated-ranklist.
  • You have an approved test inbox controlled by the project team.
  • You have selected one exact event in CERT with registrations suitable for an FTL import test.
  • The release containing 20260713000001_ftl_seed_organizer_delivery.sql and ftl-seed.yml is green.
  • The repository variable ENABLE_FTL_DEADLINE_SEND remains absent or not equal to true.

The scheduled PROD sender is separately gated. Completing this guide does not authorize enabling it.

Phase 1 · Google account

Turn on strong account protection

  1. Open Google Account → Security in a private or clearly identified browser profile.
  2. Confirm the avatar and email are the SPWS sender account. Stop if another Google account is active.
  3. Review recovery email and recovery phone details. They must belong to an authorized operator, not a temporary personal contact.
  4. Open 2-Step Verification and turn it on if it is not already enabled.
  5. Prefer a passkey, hardware security key, or Google Prompt for the operator's normal login. Keep backup access in a project-approved secure location.
  6. Run Google's Security Checkup and resolve unfamiliar sessions, devices, or third-party access before creating an SMTP credential.

Google requires 2-Step Verification before an App Password can be created. Google also recommends stronger login methods such as security keys or Google Prompt over SMS for normal account access.

Phase 2 · one-purpose SMTP credential

Create the App Password once

  1. While signed in as the SPWS sender, open Google App Passwords. Re-authenticate if Google asks.
  2. Create a new App Password with a descriptive label such as SPWS GitHub Actions FTL seed sender.
  3. Generate the password. Google displays the generated credential only once.
  4. Keep that page open temporarily. Do not save the value to a text file, password note, clipboard manager, screenshot, or chat.
  5. Proceed directly to the GitHub secret entry below. After GitHub confirms the secret was saved, close the Google page.

If “App Passwords” is missing

First confirm 2-Step Verification is on. Google says the option can also be unavailable for accounts configured only with security keys, organization-managed work/school accounts, or accounts enrolled in Advanced Protection. Do not weaken account policy blindly; confirm the account type and choose an approved SMTP identity.

Phase 3 · GitHub Actions

Add exactly two repository secrets

Preferred: GitHub web interface

  1. Open the repository's Actions secrets settings.
  2. Select New repository secret.
  3. Set the name to SPWS_GMAIL_USER. Set the value to the full SPWS Gmail address used to create the App Password. Save it.
  4. Select New repository secret again.
  5. Set the name to SPWS_GMAIL_APP_PASSWORD. Paste the generated App Password directly from Google. Save it.
  6. Confirm both names appear in the repository secret list. GitHub will not reveal their values after saving.

Alternative: GitHub CLI interactive input

Run these commands separately from the repository. At each prompt, paste only the requested value. Do not append the value to the command and do not use --body with a literal credential.

gh secret set SPWS_GMAIL_USER \
  --repo Fencer4Life/spws-automated-ranklist

gh secret set SPWS_GMAIL_APP_PASSWORD \
  --repo Fencer4Life/spws-automated-ranklist

The GitHub CLI encrypts secret values locally before sending them to GitHub. Repository secrets are read when a workflow run is queued, so update them before starting the pilot run.

Verify names without retrieving values

gh secret list \
  --repo Fencer4Life/spws-automated-ranklist \
  | rg 'SPWS_GMAIL_(USER|APP_PASSWORD)'

Expected: exactly two matching names. This command cannot display either stored value.

Phase 4 · event preparation

Prepare one exact CERT event

  1. Open the admin application in CERT. Confirm the environment indicator before editing.
  2. Open EventManager and locate the pilot by its full, exact txt_code. Record the code for the workflow input.
  3. Confirm SPWS registration is enabled and the event has the expected weapon list.
  4. Review the current entry list. The pilot should contain only data approved for the test and enough registrations to produce a non-empty mix-all bundle.
  5. Set Organizer email to the approved test inbox and save the event.
  6. Close and reopen the event to prove the email persisted. Confirm the displayed value character-for-character.
  7. Record the existing FTL bundle last sent value. It may be “Not sent yet” or an earlier timestamp; a deliberate manual resend is allowed.

Stop on any identity doubt

Do not trigger if the environment, event code, recipient, weapon list, or entries are uncertain. The workflow sends whatever is currently persisted for that CERT event.

Phase 5 · manual CERT run

Trigger the workflow once

GitHub web interface

  1. Open Actions → FTL Seed Organizer Delivery.
  2. Select Run workflow.
  3. Select branch main.
  4. Enter the exact CERT event code from the preflight.
  5. Set target to cert. Read it again before continuing.
  6. Select the final Run workflow button once. Do not click again while the run is queued or active.

GitHub CLI alternative

Replace the sentinel value before running. Leaving the sentinel unchanged should fail event lookup rather than email anyone.

EVENT_CODE='REPLACE_WITH_APPROVED_CERT_EVENT_CODE'

gh workflow run ftl-seed.yml \
  --repo Fencer4Life/spws-automated-ranklist \
  --ref main \
  -f "event_code=${EVENT_CODE}" \
  -f target=cert

Find and watch the dispatched run:

gh run list \
  --repo Fencer4Life/spws-automated-ranklist \
  --workflow ftl-seed.yml \
  --event workflow_dispatch \
  --limit 5

gh run watch RUN_ID \
  --repo Fencer4Life/spws-automated-ranklist \
  --exit-status

Replace RUN_ID with the numeric id shown by gh run list.

Phase 6 · end-to-end proof

Accept the pilot only when every boundary agrees

  • The workflow run is tied to main, the exact event code, and target cert.
  • Validate inputs passed.
  • Generate, deliver and stamp passed.
  • The approved test inbox received one message from the expected SPWS Gmail address.
  • The subject contains SPWS – pliki startowe FTL and the exact event code.
  • The message has one ZIP attachment; filenames inside match the intended event/weapons and contain no unrelated event.
  • Every XML file in the ZIP imports successfully into an FTL test installation.
  • EventManager in CERT shows a new ts_ftl_sent value after refresh.
  • The Telegram operations channel received the success confirmation.
  • The GitHub run URL, event code, timestamp, inbox result, ZIP manifest, and FTL import result are recorded as pilot evidence without secret values or participant data.

Pilot definition of done

GitHub success alone is not enough. The organizer-side inbox, ZIP contents, FTL import, database timestamp, and Telegram confirmation must all be checked.

Keep ENABLE_FTL_DEADLINE_SEND disabled until the pilot is explicitly signed off. Manual PROD delivery and scheduled PROD delivery are separate approval decisions.

For a failed run, inspect only failed logs:

gh run view RUN_ID \
  --repo Fencer4Life/spws-automated-ranklist \
  --log-failed

If a log accidentally reveals a credential, revoke it immediately before any other debugging.

Lifecycle

Rotate, revoke, or stop delivery

Routine or incident rotation

  1. Create a new App Password from the exact SPWS Gmail account.
  2. Replace SPWS_GMAIL_APP_PASSWORD in GitHub Actions secrets.
  3. Run one approved CERT pilot and complete the acceptance checklist.
  4. Revoke the old App Password in Google.

Google revokes App Passwords after the Google Account password changes. If that happens, generate a new App Password and update GitHub before the next run.

Emergency stop

  1. Confirm ENABLE_FTL_DEADLINE_SEND is absent or not true.
  2. Revoke the App Password in Google to stop SMTP authentication immediately.
  3. Delete SPWS_GMAIL_APP_PASSWORD from GitHub if the integration is being retired.
  4. Record the incident/retirement without recording the old value.
gh secret delete SPWS_GMAIL_APP_PASSWORD \
  --repo Fencer4Life/spws-automated-ranklist
