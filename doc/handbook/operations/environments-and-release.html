Environments and release

Keep development, certification and production responsibilities distinct; move software and approved data through explicit contracts.

LOCAL → CERT → PRODWorkflow controlledVerified 14 July 2026

Environment roles

Environment contract

LOCAL

Development, schema iteration, test execution and operator preparation. Safe to rebuild through repository scripts; never a source for production credentials.

CERT

Certification and review surface. Receives release candidates and curated data, supports parity checks and becomes the approved source for production promotion.

PROD

Published system. Changes are deliberate, auditable and protected from experimental/manual shortcuts.

Software delivery

Release flow

  1. Run local tests, format/type gates, documentation checks and coherence.
  2. Push the intended commit with unrelated work excluded.
  3. CI validates database, Python, frontend and generated documentation.
  4. The release workflow builds/deploys schema, functions and frontend artifacts to the intended environment.
  5. Verify CERT before allowing production continuation.
  6. Verify production health and update release tracking/seed artifacts where required.

Recovery

Failure boundaries

  • A CI failure blocks release; fix or prove transient before rerun.
  • A CERT verification failure blocks PROD.
  • A reconciliation transaction failure rolls back the unit of work; inspect the generated report/log.
  • A protected deletion becomes reported divergence, not a forced erase.
  • Do not edit deployed migrations; add a corrective migration.
  • Use documented mirror/export/snapshot scripts before risky data operations.
  • Tracking/reconcile-log commit steps (release.yml deploy-cert/deploy-prod, evf-sync.yml promote jobs) retry on a rejected push by fetching and rebasing onto origin/main, since concurrent runs commit to the same auto-generated files; a failure here after 5 attempts is a real conflict needing investigation, not routine contention.

Security

Secrets

GitHub environment secrets and Supabase deployment credentials are environment-scoped. Browser code receives public configuration only. Workflow dispatch uses trusted server-side infrastructure. Rotate any credential exposed in source, logs or chat.

Location only: documentation and agent memory may name these paths and variable names, but must never copy their values. Confirm each path remains ignored with git check-ignore before relying on it.

Definition of done

Release verification

  • All required jobs concluded successfully.
  • Expected schema migration set is deployed.
  • Frontend artifact/version/environment is correct.
  • Public ranklist/calendar and protected admin authentication load.
  • Promotion/reconciliation report shows expected convergence.
  • No unrelated local files were staged or pushed.
  • Tracking and documentation reflect the shipped state.
