Local development

Bring up the database, Python and Svelte surfaces safely, then use the same narrow-to-broad validation discipline as CI.

LOCAL onlySafe reset requiredVerified 12 July 2026

Before starting

Prerequisites

  • Python 3.11+ and repository virtual environment
  • Node/npm compatible with the frontend lockfile
  • Docker and Supabase CLI
  • Ruff and BasedPyright through development dependencies/tooling
  • Postgrestools for SQL semantic/safety checks
  • Graphify for architecture/impact analysis

First run

Install and start

source .venv/bin/activate
pip install -e ".[dev]"

cd frontend
npm ci
cd ..

supabase start
./scripts/reset-dev.sh

cd frontend
npm run dev
Never run bare supabase db reset. Use ./scripts/reset-dev.sh; it resets LOCAL and recreates the expected admin account.

Broad confidence

Full suites

supabase test db
source .venv/bin/activate && python -m pytest python/tests/ -v
cd frontend && npm test
cd frontend && npm run check
cd frontend && npm run test:e2e
bash scripts/check-coherence.sh
bash scripts/check-spec-sync.sh

Run the smallest relevant test first. Before completion, run all suites applicable to the changed behavior. Record anything intentionally not run.

Implementation discipline

TDD order

  1. Define acceptance assertions and plan-test IDs.
  2. Write tests.
  3. Run and prove RED.
  4. Implement the minimum behavior.
  5. Run and prove GREEN.
  6. Refactor while remaining green.

Common boundaries

Troubleshooting

  • Database semantic checks require the configured LOCAL instance on port 54322.
  • Svelte files are checked through svelte-check; generic TypeScript LSP alone is insufficient.
  • Integration tests may require live external APIs; distinguish them from deterministic unit tests.
  • Do not install Python packages into the system interpreter; use the virtual environment.
