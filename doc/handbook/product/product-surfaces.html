Product surfaces

One Svelte application serves public ranking and calendar journeys, event registration and authenticated administration; selected views are also packaged as embeddable web components.

Public experience

Ranklist

Filter by season, weapon, gender and age category; switch ranking mode where configured; open a fencer drilldown to understand score composition.

Calendar

Browse events chronologically with event family, location, dates, status, registration and result links.

Registration

Submit identity and consent information for an event that has registration enabled, then follow verification/matching behavior.

Entry list

View registered participants through the event’s public entry-list surface without exposing private administrative fields.

Authenticated

Administration

Admin mode is requested through the application route/query context, but access is established through Supabase Auth and TOTP MFA. Merely adding ?admin=1 does not grant data authority.

Season administration

Create and manage seasons, initialize season structure, select scoring/carry-over configuration and operate environment-aware promotion.

Event administration

Maintain events, tournament slots, result URLs, registration settings and ingestion actions.

Fencer and identity administration

Review identity candidates, aliases, estimated birth years, duplicate identities and master-data corrections.

Operational feedback

Trigger trusted workflows through server-side dispatch and surface progress/errors in the interface rather than requiring browser-console debugging.

Host integration

SPA and web components

The SPA entrypoint supports the certification/admin application. A separate custom-element build produces Shadow-DOM-isolated ranklist, calendar, registration and entry-list elements for embedding by the SPWS website.

Embedding invariant: web components must not leak styles into the host page or depend on the host page’s CSS to remain usable.

Canonical journeys

From user action to data

  1. A public visitor selects a ranking or calendar context.
  2. The frontend reads stable Supabase views/RPCs through its API layer.
  3. For registration, a dedicated RPC validates event availability, consent and identity input.
  4. For administration, the auth layer establishes an MFA-backed session before protected operations are available.
  5. Admin writes use controlled RPCs or trusted workflow dispatch rather than direct anonymous table mutation.

Product responsibility

What the frontend does not own

  • It does not decide scoring policy independently from PostgreSQL.
  • It does not store GitHub credentials in the browser.
  • It does not treat visual event status as permission to mutate lifecycle state.
  • It does not make console output the operator’s only diagnostic surface.
