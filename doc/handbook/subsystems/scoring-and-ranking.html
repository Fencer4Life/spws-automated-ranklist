Scoring and ranking

PostgreSQL owns the authoritative transformation from committed results and season configuration to published segmented rankings.

Database-owned policySeason configurationRolling carry-over

Why it exists

Purpose

Produce reproducible domestic and extended rankings, expose component scores for explanation and support season-specific policy without forking frontend or Python formulas.

Ownership

Scope and boundaries

Owns

  • Age-category lookup
  • Points and ranking functions
  • Season scoring configuration
  • Domestic/international inclusion
  • Rolling carry-over engine dispatch
  • Public ranking/score read models

Does not own

  • Source parsing
  • Fencer identity decisions
  • Event URL selection
  • UI filter state
  • Promotion authorization

Smallest useful explanation

Mental model

committed result + tournament slot + fencer + event + season config
         → effective category / eligible pool / points
         → ranking rows + score drilldown + optional carried rows

Present tense

Current flow

  1. Ingestion commits result placements to category/gender/weapon tournament slots.
  2. Ranking functions select eligible current-season results for the requested dimensions.
  3. The configured engine determines prior-event carry-over behavior.
  4. Current results and stopping rules suppress replaced carry-over where applicable.
  5. Public ranking and drilldown APIs return ordered totals and components.
  6. The Svelte frontend formats/filters the result without recalculating authoritative points.

Must remain true

Invariants

One authoritative SQL implementation owns scoring and age-category semantics.
Weapon, gender and V-category segmentation is preserved even after combined physical pools.
Season configuration is exportable/importable and round-trips without silent field loss.
Carry-over engine selection is per season; existing seasons are not silently rewritten by a new default.
Admin edits to scoring configuration are explicit, authenticated and test-covered.

Run and observe

Operations

Change scoring through the admin configuration contract, export the JSON before risky changes and compare affected rankings. Engine changes require parity comparison and explicit season selection. Never patch calculated output rows to “fix” a formula.

Evidence

Tests

pgTAP suites cover base ranking, rolling, engine dispatch/FK carry-over, cross-gender segmentation, joint pools, thresholds and configuration round-trip. Frontend tests verify filters and drilldown presentation; they do not replace SQL correctness tests.
