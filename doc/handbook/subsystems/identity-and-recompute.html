Identity and recompute

Resolve unstable source names to durable fencer identities, preserve human decisions as aliases and repair affected rankings when master data changes.

FK identityAlias memorySelf-healing recompute

Why it exists

Purpose

Source names vary by spelling, diacritics, ordering and provider. Rankings need a stable person identity, while operators need a safe path to confirm ambiguity and correct mistakes without re-scraping unrelated events.

Ownership

Scope and boundaries

Owns

  • Canonical fencer identity
  • Name folding and fuzzy candidates
  • Alias memory and match provenance
  • Estimated birth-year review
  • Merge/split identity correction
  • Affected-event recompute

Does not own

  • Raw source parsing
  • Ranking formulas
  • Registration consent policy
  • Event lifecycle
  • Foreign federation master records

Smallest useful explanation

Mental model

scraped name → exact alias / canonical comparison → fuzzy candidate
     → auto match | human confirmation | domestic auto-create | international skip
     → result stores fencer FK + original source name + match provenance

A later identity correction does not rewrite source evidence. It changes canonical links/data, queues affected events and recomputes derived category/count/score state from committed FK-linked results.

Present tense

Current flow

  1. Identity resolution receives parsed rows and intake policy.
  2. Known aliases provide deterministic matches before fuzzy scoring.
  3. Thresholds and organizer context decide automatic, review, auto-create or skip outcomes.
  4. Committed results retain original source names, confidence and method.
  5. Admin identity actions update aliases/fencer data through controlled RPCs.
  6. Database change hooks enqueue affected events; the recompute worker drains/debounces work.
  7. RECOMPUTE_DOMESTIC reloads committed rows, recalculates category/counts and commits derived state.

Must remain true

Invariants

Results identify fencers by foreign key, never by mutable display name.
Original scraped names and match provenance remain auditable after correction.
Human-confirmed aliases become reusable cross-event memory.
Domestic and international low-confidence outcomes follow different business policy.
Birth-year or merge corrections recompute all affected events, not only the currently viewed ranking.
The recompute queue and watermark are service_role-only. An admin edit still enqueues because the eight fencer-mutation RPCs are SECURITY DEFINER and fn_enqueue_affected_events is too, so the queue INSERT runs as the definer — the admin never needs a grant on the queue (Security posture).
fn_merge_fencers is not callable by an anonymous browser. It deletes a fencer row and re-points its results.
Recompute uses committed results and FKs; it does not repeat source scraping/fuzzy matching.

Run and observe

Operations

Review ambiguous identity in admin/staging reports. Prefer confirming/transferring aliases over manually editing result names. After master-data changes, monitor recompute queue/workflow and verify affected event rankings. A merge is a high-impact operation and must preserve result/registration references.

Evidence

Tests

Matcher, alias, dedup, birth-year, identity UI and recompute tests cover normalization, threshold outcomes, provenance, FK repointing, queue debounce and cross-event repair. Database tests protect alias uniqueness and merge behavior.
