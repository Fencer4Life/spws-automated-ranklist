Subsystem · Ingestion

Ingestion

Normalize heterogeneous fencing sources, apply SPWS domain policy and commit one trusted event result set with traceable provenance.

Plugin/rule orchestrationDraft + automated pathsVerified 12 July 2026

Why it exists

Purpose

Result providers disagree on files, fields, categories and identity. Ingestion creates one source-neutral contract and applies domain decisions after parsing, so provider quirks cannot silently become ranking policy.

Ownership

Scope and boundaries

Owns

  • Source parsing and IR
  • Event/tournament resolution
  • Identity dispatch
  • Pool/category/count validation
  • Draft/review artifacts and commit orchestration
  • Ingest trace, faults and report fragments

Does not own

  • Authoritative scoring formulas
  • Admin event/URL policy decisions
  • CERT→PROD release authorization
  • External provider correctness
  • Frontend navigation

Smallest useful explanation

Mental model

Source adapter → ParsedTournament IR → Rule plan → Ordered plugins
      → faults/remediation + report fragments → Commit → Post-commit flow

Plugins declare reads, writes, kind and effects. The rulebook orders them into scenarios. Context moves forward; plugins do not call one another. Recoverable domain faults remain data in the context. Infrastructure failure is an abort.

Present tense

Current flow

  1. ingest_cli.py constructs services and selects INGEST_DOMESTIC.
  2. ParseSource invokes the selected parser and creates the shared context.
  3. Validation, event resolution, identity, pool split, count, pool-round and category plugins run in declared order.
  4. Some plugins use plugins/bridge.py to call proven functions in stages.py; mapped legacy halts become faults or aborts.
  5. Commit persists accepted live state through the database boundary.
  6. POST_COMMIT validates participant count, notifies and formats staging output.

The review CLI can instead write run-scoped draft tables, compare source/CERT-reference/draft state and call fn_commit_event_draft after operator review.

Must remain true

Invariants

Every parser emits the same ParsedTournament/ParsedResult contract and stable source-row identity.
Parser facts do not override admin-canonical metadata without explicit validation.
Plugin public writes are declared and checked; order satisfies the context-key DAG.
One event commit is transactional—partial live replacement is not an acceptable result.
Draft and ingest-history tables are reachable only by service_role: RLS is enabled with no policy, and all grants are revoked from anon and authenticated. The pipeline reaches them because service_role bypasses RLS inherently, not because a policy admits it (Security posture).
Combined categories, joint pools, gender routing and participant counts retain their distinct meanings.
Source URLs are validated against parsed data on writes where policy requires it.

Run and observe

Operations

Use event-scoped CLI/workflow inputs and review the generated staging report before advancing. Failures should identify parser, plugin and fault/abort boundary. LOCAL database changes use the safe reset script, never a bare reset. Do not advance through batches without required event sign-off.

Operational commands and workflow selection are documented in operator runbooks.

Evidence

Tests

Primary evidence lives in python/tests/test_pipeline_*.py, parser tests/fixtures, draft-store/review tests and pgTAP coverage for draft/commit/ingest invariants. New behavior follows RED→GREEN TDD and carries traceable plan-test IDs.
