Frontend and automation

Svelte presents public/admin capabilities while GitHub Actions, Edge Functions, GAS and Telegram execute trusted background and operational work.

Why it exists

Purpose

Give users a coherent bilingual interface and give operators controlled automation without exposing credentials or embedding domain policy in presentation code.

Ownership

Scope and boundaries

Owns

  • Public/admin interaction and state
  • SPA/custom-element builds
  • Auth/MFA UX
  • API adaptation and diagnostics
  • Workflow definitions and triggers
  • Operational notification adapters

Does not own

  • Authoritative scoring
  • Database authorization policy
  • External provider availability
  • Secret values in client code
  • Manual domain approval decisions

Present tense

Current flow

  1. App.svelte selects ranklist, calendar or authenticated admin surfaces.
  2. lib/api.ts centralizes Supabase read/RPC calls and environment behavior.
  3. Admin auth establishes Supabase session and TOTP challenge/enrollment state.
  4. Vite builds SPA and custom-element targets from separate entrypoints.
  5. Trusted operational actions dispatch allowlisted workflows through server-side infrastructure.
  6. GitHub Actions install dependencies, execute domain commands and use environment-scoped secrets.
  7. GAS/Telegram provide an additional command/artifact adapter and return human-readable status.
  8. The EventManager seed button and Telegram send <EVENT-CODE> participants command dispatch the same ftl-seed.yml input contract; the workflow performs the actual Gmail send server-side.

Must remain true

Invariants

No GitHub PAT/service-role/Gmail secret is bundled or stored in browser-visible configuration.
Web components remain Shadow-DOM isolated and usable on the host site.
English/Polish locale state is reactive across public surfaces.
Admin UI availability follows authenticated authority, not query-string presence alone.
Operational failures are visible in UI/report/Telegram surfaces, not console-only.
Workflow dispatch requires a signed-in admin session that has completed TOTP (aal2). The allowlist is the second gate, not the first: verify_jwt = true is satisfied by the public anon key that ships in this bundle (Security posture).
Workflow allowlists, Telegram help and documentation stay synchronized.
The send button is disabled until the persisted organizer address matches the current draft.

Run and observe

Operations

Run Vitest and Svelte check after frontend changes; use Playwright for browser journeys. Inspect workflow job logs at the failing step and rerun only when evidence supports a transient failure. For FTL delivery, verify the chosen target and event code, save the organizer address before dispatch, and treat a failed stamp after SMTP acceptance as a possible duplicate-on-retry. Update dispatch allowlists, secrets reference and operator help whenever adding a workflow/command.

Evidence

Tests

Vitest covers components, state and API integration; Svelte check enforces type/template correctness; Playwright covers critical browser paths. Workflow YAML is validated through CI execution and targeted configuration checks.
