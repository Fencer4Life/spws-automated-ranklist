Events and registration

Maintain a trustworthy season calendar, create scoring slots only when needed, reconcile approved event identity across environments and accept privacy-aware registrations.

Why it exists

Purpose

Events connect public planning, result ingestion, promotion and registration. The subsystem keeps calendar identity stable while separating future event existence from tournament/result state.

Ownership

Scope and boundaries

Owns

  • Season/event/tournament lifecycle
  • Calendar identity and organizer
  • Prior-event linkage
  • Event URL/registration fields
  • Season skeleton and calendar reconciliation
  • Registration, consent and public entry list

Does not own

  • Result source correctness
  • Scoring formula
  • Fencer master identity policy
  • Email provider delivery guarantees
  • Host-site content outside components

Smallest useful explanation

Mental model

Season → Event (calendar identity and lifecycle)
             ├── Tournament slots (created when competition/results need them)
             └── Registrations (intent to participate, not results)

CERT event set ──whole-set diff by stable code──▶ PROD event set

Present tense

Current flow

  1. Admin creates/initializes a season and future events without manufacturing tournament children.
  2. Calendar synchronization imports or refreshes source-owned EVF identity while preserving admin-owned fields.
  3. Ingestion creates/fetches tournament slots when actual results require them.
  4. CERT calendar promotion computes create/update/delete differences against PROD and applies one guarded transaction.
  5. Public registration is enabled per event and records consent/version evidence through a constrained RPC.
  6. Registration identity matching and entry-list views expose the appropriate public/admin projections.
  7. Admin or Telegram dispatch generates a fresh mix-all FTL ZIP, Gmail accepts it, and only then does the service-role RPC stamp the event; a gated daily sweep handles due unstamped events.

Organizer handoff

FTL seed delivery

The first delivery release contains the validated mix-all files only. It is generate-at-send and at-least-once: manual runs are explicit re-sends, while the sweep selects only unstamped live events after cutoff and skips empty bundles. Exact-once behavior would require a durable attempt/lease model.

Must remain true

Invariants

An event may exist without tournament children.
Event codes and external identity keys remain stable; environment-local ids are resolved, not copied.
Source-owned identity may refresh, but admin-owned URLs/fees/registration settings are never silently clobbered.
PROD deletion is limited to safe planned, results-free events; protected divergence is reported.
CERT and PROD active seasons must match before whole-set reconciliation.
Registration consent evidence is captured with the version presented to the user.
FTL delivery never stamps before SMTP acceptance; SMTP errors leave ts_ftl_sent unchanged.

Run and observe

Operations

Review calendar reconciliation reports before trusting convergence. Investigate skipped deletions rather than forcing them. Validate URLs against the event they claim to represent. For organizer delivery, pilot one event on CERT, inspect the manifest/recipient and timestamp, then use manual PROD dispatch. Keep ENABLE_FTL_DEADLINE_SEND unset until that pilot is signed off. Gmail credentials stay only in GitHub Actions secrets; an App Password is independently revocable but must still be treated as a high-value credential.
