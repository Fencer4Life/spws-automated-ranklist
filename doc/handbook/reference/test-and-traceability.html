Requirement chain

Traceability chain

FR / NFR → RTM row → plan test ID → test-code marker → implementation evidence

Use stable IDs in plans and tests. If requirements or test IDs change, update the RTM, the explicit roster sentinel in scripts/check-spec-sync.sh and documented suite totals in the same change. The HTML requirements traceability matrix remains governance/reference, not newcomer reading.

Change workflow

Mandatory TDD

  1. Define acceptance tests and IDs before implementation.
  2. Write the test.
  3. Run it and prove RED for the intended reason.
  4. Implement the minimum change.
  5. Run and prove GREEN.
  6. Refactor while preserving GREEN.
  7. Run applicable full suites and coherence/documentation gates.

Repository gates

Coherence checks

  • Python/frontend versions remain synchronized when released.
  • ADR source count matches the governance registry.
  • pgTAP planned assertions match the documented total.
  • New migrations ship with relevant documentation/decision changes.
  • Generated ADR and root HTML match retained Markdown sources.
  • Canonical handbook metadata, links, map coverage and ownership are valid.

Handoff

Completion report

Report exact commands, outcomes, skipped checks and remaining risks. Never say “tests pass” when only a narrow test ran. Do not fix unrelated pre-existing failures silently; distinguish them from changes introduced by the task.
