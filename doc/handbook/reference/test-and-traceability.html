Fixture discipline

Fixtures own their data

A pgTAP fixture must not depend on which rows the seed happens to contain, nor on the date the suite runs. Two rules follow, and both are load-bearing:

  • Create fixture rows; do not borrow seed rows. Looking a fencer up by surname binds the test to seed contents and is ambiguous whenever production holds two fencers sharing a surname — the same defect fixed in export_seed.py::fencer_lookup(), where identity is surname, first name and birth year together.
  • Derive age-dependent values from the active season, never from literals. fn_age_category buckets on season end year − birth year, so a hardcoded birth year silently changes category at a season rollover. Fixtures set birth years as <active season end year> − <target age> and place each fencer mid-band (V1 at 45, V2 at 55, V3 at 65) rather than near a boundary.

Mutating statements in a test are scoped to the fixture's own season. A statement that mirrors a migration backfill must carry an explicit scope even when the unscoped form appears to work against today's seed.

A test that must guard live rules resolves the active season rather than naming a season literally — but pairs that with a config contract: one assertion pinning the exact configuration values the rest of the file's arithmetic assumes. Scoring configuration is per-season, so a literal season pin silently guards retired rules, while a bare config-following test silently changes meaning when an admin edits config. The contract assertion resolves both: a rule change produces one named failure that says which expectations to recompute. Assertions about whether the engine applies configuration compare against the configured value, not a literal, so they never drift; the literals live in the contract, in one place.

Requirement chain

Traceability chain

FR / NFR → RTM row → plan test ID → test-code marker → implementation evidence

Use stable IDs in plans and tests. If requirements or test IDs change, update the RTM, the explicit roster sentinel in scripts/check-spec-sync.sh and documented suite totals in the same change. The HTML requirements traceability matrix remains governance/reference, not newcomer reading.

Change workflow

Mandatory TDD

  1. Define acceptance tests and IDs before implementation.
  2. Write the test.
  3. Run it and prove RED for the intended reason.
  4. Implement the minimum change.
  5. Run and prove GREEN.
  6. Refactor while preserving GREEN.
  7. Run applicable full suites and coherence/documentation gates.

Repository gates

Coherence checks

  • Python/frontend versions remain synchronized when released.
  • ADR source count matches the governance registry.
  • pgTAP planned assertions match the documented total.
  • New migrations ship with relevant documentation/decision changes.
  • Generated ADR and root HTML match retained Markdown sources.
  • Canonical handbook metadata, links, map coverage and ownership are valid.

Handoff

Completion report

Report exact commands, outcomes, skipped checks and remaining risks. Never say “tests pass” when only a narrow test ran. Do not fix unrelated pre-existing failures silently; distinguish them from changes introduced by the task.
