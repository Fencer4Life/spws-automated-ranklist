Security posture

Who may reach what, why new objects are unreachable until you say otherwise, and where that is enforced.

Deny by defaultADR-083Verified 23 July 2026

Why it exists

Purpose

This page is the single owner of the system's authorization model. Before July 2026 no such page existed anywhere in the handbook or governance corpus, and documentation-map.html assigned the topic to nobody. That absence is why four months of grant drift had no home to be recorded in — the drift was detectable the whole time, but nothing in the documentation claimed responsibility for noticing.

Smallest useful explanation

Mental model

Three Postgres roles reach the database, and they are not tiers of one hierarchy — they are three different trust stories.

RoleWho it isWhat it may reach
anonEvery anonymous visitor. Its API key ships in the public frontend bundle and is not a secret.SELECT on the tables and views the public ranklist and calendar render, plus a fixed allowlist of read-only functions and the two self-registration RPCs. No write privilege on any table.
authenticatedAny holder of a Supabase Auth account. Signup is closed, so in practice: the admins.The admin surface. Reaches most tables through SECURITY DEFINER RPCs; three direct writes remain (season and event status).
service_roleThe Python pipeline and GitHub Actions workflows. Key is a repository secret and never reaches a browser.Everything. It holds BYPASSRLS, which is how it reaches the pipeline/staging tables that have RLS on and no policy at all.

The load-bearing idea: RLS and grants are two independent controls, and neither substitutes for the other. An owner-rights view ignores RLS on its base tables entirely, and a TRUNCATE grant is not filtered by RLS at all. Where the model relies on RLS alone, one mis-written policy is the whole defence.

Present tense

How access is decided

Pipeline and staging tables — tbl_result_draft, tbl_tournament_draft, tbl_event_ingest_history, tbl_tournament_ingest_history, tbl_recompute_queue, tbl_recompute_watermark — have RLS enabled and no policy, with all grants revoked from anon and authenticated. No policy is the correct amount of policy: RLS with zero policies denies everyone except roles that bypass it, and only service_role does. Nothing in frontend/src or supabase/functions references any of the six.

The other ten tables keep SELECT for anon and have lost every write privilege — INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER. They also carry RLS policies, so the policy is now the second line rather than the only one.

vw_fencer_aliases is owner-rights (owner=postgres, no security_invoker), so RLS beneath it does nothing and grants are the only control that applies. anon holds nothing; authenticated holds SELECT only. The admin UI reads aliases through fn_list_fencer_aliases.

New objects arrive ungranted. ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA public no longer grants anon or authenticated anything on new TABLES, FUNCTIONS or SEQUENCES.

The common task

Adding a new public-facing table

A new table is unreachable from the frontend until you grant it explicitly. That friction is deliberate — it forces a migration author to state intent rather than inherit it. In your migration:

CREATE TABLE tbl_example (...);

-- Decide, explicitly, one of these two:

-- (a) public-readable: the ranklist or calendar renders it
ALTER TABLE tbl_example ENABLE ROW LEVEL SECURITY;
CREATE POLICY "public read" ON tbl_example FOR SELECT TO anon, authenticated USING (true);
GRANT SELECT ON tbl_example TO anon, authenticated;

-- (b) pipeline-only: only service_role touches it
ALTER TABLE tbl_example ENABLE ROW LEVEL SECURITY;   -- and no policy, and no grant

Omitting the RLS line fails assertion 52.1 in CI. Granting a write privilege to anon fails 52.3–52.6. Adding an anon-callable function without updating the allowlist fails 52.7. You will find out before the change reaches CERT.

The anon-executable surface

Function allowlist

Nineteen functions are callable by anon, pinned as a set equality by pgTAP 52.7. Equality rather than a deny-list, because every object that produced the ADR-083 findings was created after any deny-list could have been written — a list of known-bad names would have caught none of them.

Scope: directly-callable, non-extension functions. Trigger-returning functions are excluded because Postgres refuses to call them outside a trigger context, so a grant on one is not reachable surface; btree_gist ships roughly 200 functions into public whose grants belong to the extension.

Server-side workflow dispatch

The dispatcher's authorization contract

dispatch-workflow holds a GitHub PAT and can trigger seven allowlisted workflows, including promote-season.yml and ftl-seed.yml against PROD. Its contract:

CallerResponse
No Authorization header401 missing_authorization
Undecodable bearer token401 invalid_token
Public anon key, or service_role401 not_authenticated
Token that resolves to no user401 invalid_token
Signed-in admin who has not completed TOTP403 mfa_required
Signed-in admin at aal2Dispatch proceeds; the user id is logged

verify_jwt = true is not an authorization control. It reads like one, which is how the gap survived review: the public anon key is a valid, correctly-signed JWT that ships in the frontend bundle, so it satisfied verify_jwt and reached the workflow allowlist. The role check is what closes it.

Consequence for local development: admin-auth.svelte.ts skips MFA on localhost, so a LOCAL session stays aal1 and the dispatcher answers 403. Dispatch targets real GitHub Actions and is not a LOCAL-appropriate operation.

Must remain true

Invariants

  • No table in schema public has RLS disabled.
  • anon holds no INSERT, UPDATE, DELETE or TRUNCATE on any table.
  • The set of anon-EXECUTEable functions equals the documented allowlist exactly — no more, no fewer.
  • vw_fencer_aliases grants nothing to anon and only SELECT to authenticated.
  • Role postgres grants no default privileges to anon or authenticated in schema public.
  • Every SECURITY DEFINER function carries an explicit SET search_path. Without one it is a privilege-escalation vector, because the caller controls how its unqualified identifiers resolve.
  • Signup stays closed on CERT and PROD. authenticated currently means "any account holder", so an open signup is an open admin door.

Run and observe

Operations

Check any environment on demand — read-only, safe to run at any time:

export SUPABASE_ACCESS_TOKEN=...            # already in .env
./scripts/check-security-posture.sh <project-ref> CERT

The same script runs inside release.yml's deploy-cert and deploy-prod jobs and fails the release on regression. It lives there, and not only in CI, because release.yml's build job is skipped entirely by a manual workflow_dispatch — a CI-only guard is optional in practice. It also checks the right thing: pgTAP proves what LOCAL looks like after replaying migrations, while this proves what CERT and PROD actually look like.

Failures alert to Telegram. Not e-mail: the Supabase advisory e-mail fired weekly from 6 May 2026 and every message but the last went unread. Detection was never the problem; delivery was.

Maintenance

Change triggers

  • Any new table, view, function or sequence in schema public.
  • Any GRANT, REVOKE, RLS or policy change.
  • Any change to the anon-executable allowlist — update 52_security_posture.sql (source of truth), scripts/check-security-posture.sh and this page together.
  • Any change to the dispatcher's authorization contract, or a new Edge Function.
  • Any change to signup policy or the meaning of authenticated.
