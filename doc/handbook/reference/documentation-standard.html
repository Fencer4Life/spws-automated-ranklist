Canonical page contract

Required metadata and structure

Every handbook page declares doc-id, doc-status, doc-type, doc-owner, last-verified, source globs and review triggers. Subsystem pages additionally contain purpose, boundaries, mental model, current flow, invariants, implementation map, operations, tests, related decisions and change triggers.

Present tense rule: handbook pages describe the post-change system. Do not append implementation diaries, milestone close-outs or superseded behavior.

Authority

Link, do not duplicate

  • Link to ADRs for rationale.
  • Link to formal rules for exact business statements.
  • Link to the workflow catalog for inventory and to runbooks for procedures.
  • Link to source paths/tests as evidence.
  • Do not link legacy narrative as the canonical explanation.
  • Do not maintain volatile counts in README or orientation pages unless generated.

Per change

Maintenance workflow

  1. Classify documentation impact during planning.
  2. Name owning pages through the documentation map.
  3. Update current-system prose and source references in the same change.
  4. Update/add/supersede ADRs when the change is a decision.
  5. Update runbooks/catalogs for operational impact.
  6. Move superseded prose to archive/evidence.
  7. Run renderers, check_docs.py and coherence gates.
  8. Report docs changed and checks run.

Ownership warnings remain advisory until mappings have completed two development cycles without unacceptable false positives. Structural metadata, links, map coverage and required sections are hard failures immediately.

History discipline

Archive policy

Archive only after replacement coverage is proven. Preserve files and Git history, create an archive index and leave redirects at high-value old HTML paths. Evidence remains exact and searchable but is not part of the newcomer navigation.
