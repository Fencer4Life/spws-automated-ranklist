SPWS · Deep analysis · 22 July 2026 · follow-up to the 21 July handover

The advisory understates it. Six tables is the visible part; the real defect is a hardening pattern that was never re-applied.

I verified every claim in the 21 July handover against the live LOCAL schema — all six are correct. Then I went one layer further, into views, RPC grants and the migration history, and found three things the Supabase advisor does not report and the handover did not reach: an anon-readable view that will survive the proposed fix, five anon-executable write RPCs (two of them SECURITY DEFINER, which bypass RLS entirely), and a March 2026 migration that fixed this exact bug class and was never carried forward. Nothing has been changed.

handover claims: all verified +1 view bypasses the planned fix +5 anon-executable write RPCs recurrence of ADR-016 nothing changed

00Verdict in one paragraph

The handover is accurate and safe to act on — I re-ran its queries and reproduced every number. But its §04 fix plan (“enable RLS on the six tables, optionally revoke grants”) would not close the exposure. Draft data is also reachable through vw_fencer_aliases, which is owned by postgres and not security_invoker, so it reads the draft tables with the owner’s rights and ignores whatever RLS you put underneath. Separately, five RPCs are callable by anon, and two of them are SECURITY DEFINER, meaning they bypass RLS on tables that are protected — including tbl_tournament and tbl_registration. The scope of the fix has to be “the anon attack surface of the public schema”, not “six tables”.

One correction to the handover, in its favour Its §06 open question 1 — “service-role-only, or does the admin UI need these?” — is answered. Nothing under frontend/src or supabase/functions references any of the six tables. The only frontend mentions are two comments naming vw_fencer_aliases in types.ts and birthYearEstimate.ts. The consumers are all Python: draft_store.py, recompute/worker.py, review_cli.py, alias_writeback.py, db_connector.py and three tools. A service-role-only lock on the base tables breaks nothing — but only if the alias view is handled at the same time, because that is what the admin UI actually reads.

02New finding A — the fix as written does not close the hole

One view is built on the draft tables:

view              relkind  reloptions  owner
vw_fencer_aliases   v          -       postgres

grants: anon | authenticated | service_role
        → DELETE,INSERT,REFERENCES,SELECT,TRIGGER,TRUNCATE,UPDATE  (all three)

Postgres views execute with the view owner’s privileges unless created with security_invoker = true. reloptions is empty, so this view runs as postgres and RLS on the underlying draft tables will not apply to reads through it. Proven, not inferred:

BEGIN; SET LOCAL ROLE anon;
SELECT count(*) FROM vw_fencer_aliases;   -- 380 rows returned to anon
ROLLBACK;

So if the handover’s §04 migration ships as described, an anonymous caller loses tbl_result_draft but keeps a 380-row window onto the same alias/draft data through rest/v1/vw_fencer_aliases. The Supabase advisor will still go green, because rls_disabled_in_public only inspects tables — this is a separate lint (security_definer_view) that either was not included in the mail or was filtered out of the condensed body.

Why this one needs a decision, not just a REVOKE The alias view is what the Phase‑4 admin UI reads (ADR‑050 / ADR‑058, per the comments in types.ts and birthYearEstimate.ts). Revoking anon is safe; revoking authenticated would break the identity-manager screen. The correct shape is SELECT to authenticated only, plus dropping the nonsensical INSERT/UPDATE/DELETE/TRUNCATE grants on a view. Recreating it with security_invoker = true is the more thorough option but changes its behaviour for the admin session too, so it needs the RLS policies underneath to allow authenticated reads.

03New finding B — five write RPCs are callable by anon

Every function in public that touches the six tables, plus the SECURITY DEFINER functions anon can execute:

FunctionSECURITYanon EXECUTEWhat it does / why it matters
fn_find_or_create_tournamentdefineryesWorst of the set. DEFINER means it bypasses RLS on tbl_tournament — a table the advisory considers safe. Migration 20260406000001 granted it to authenticated but never revoked the PUBLIC default.
fn_create_registrationdefineryesWrites tbl_registration bypassing its authenticated-only policy.
fn_commit_event_draftinvokeryesPromotes a draft toward real results. Invoker rights, so the final write into tbl_result is still refused by RLS for anon — but the draft-side effects land.
fn_discard_event_draftinvokeryesAnon can discard pending drafts — destructive, and it succeeds because anon already has DML on the draft tables.
fn_enqueue_affected_eventsinvokeryesAnon can push arbitrary work into tbl_recompute_queue. Note the scheduled Recompute Self-Heal Drain (PROD) workflow runs hourly and consumes that queue.
fn_refresh_active_seasondefineryesNot draft-related, but same class: a DEFINER state-changing function open to anon.

Read-only DEFINER functions also open to anon — fn_ranking_ppw, fn_ranking_kadra, fn_season_summary, fn_export_scoring_config, fn_fencer_scores_rolling and variants — are intentional and documented as such in 20260327000001. Leave them alone.

The ACL proves the mechanism

fn_find_or_create_tournament | acl={=X/postgres, postgres=X/postgres,
                                    anon=X/postgres, authenticated=X/postgres, service_role=X/postgres}
                                    ^^^^^^^^^^^
                                    the bare "=X" entry is the PUBLIC default grant

Postgres grants EXECUTE to PUBLIC on every new function. Unless a migration says REVOKE … FROM anon, PUBLIC, the function is world-callable the moment it exists.

04New finding C — this is a recurrence, and that is the real defect

This exact bug class was found and fixed in March 2026. From supabase/migrations/20260327000001_revoke_write_functions.sql:

-- ADR-016: Supabase Auth + TOTP MFA for Admin Access
-- Security fix: all SECURITY DEFINER write functions were callable by anon
-- because PostgreSQL defaults GRANT EXECUTE TO PUBLIC.

The comment describes today’s finding word for word. That migration, and 20260327000003_delete_cascade.sql after it, established the house pattern: REVOKE … FROM anon, PUBLIC then GRANT … TO authenticated. Every table and RPC created after March — the Phase‑1/2 ingest tables (May), the Phase‑5 alias RPCs (May), the CDC recompute tables (June) — skipped it. Nothing in CI or pgTAP enforces it, so the regression was silent.

The conclusion that changes the plan A migration that locks these six tables fixes this advisory and leaves the mechanism intact. The next new table or RPC will be born world-writable exactly the same way. The deliverable has to include a standing pgTAP assertion — “no table in public has RLS disabled; no non-allowlisted function is executable by anon” — with the read-only ranking functions as an explicit allowlist. That test is what makes the fix permanent, and it is cheap: it is a catalog query, and it will fail loudly the day someone adds a table.

06What I would change about the fix plan

Keep the handover’s §04 sequence — TDD-first, one migration, postgrestools, LOCAL proof by attempted anon write, LOCAL→CERT→PROD, coherence gate. Widen its content:

  1. Six tables: ENABLE ROW LEVEL SECURITY, and REVOKE ALL … FROM anon, authenticated. No policy for anon. service_role bypasses RLS inherently, so the Python pipeline is unaffected. An authenticated read policy is only needed if the admin UI later reads them directly — today it does not.
  2. vw_fencer_aliases: REVOKE ALL FROM anon; reduce authenticated to SELECT.
  3. Five RPCs: REVOKE EXECUTE … FROM anon, PUBLIC, keep authenticated where the admin UI calls them — following 20260327000001 verbatim, including its comment style.
  4. Standing pgTAP guard (the part that prevents recurrence) — one test asserting no RLS-disabled table in public, one asserting the anon-executable function set equals a named allowlist. Bump the pgTAP total in the specification.
  5. ADR — “pipeline and staging objects are service-role only; new objects must be revoked by default”. This amends rather than repeats ADR‑016; the registry entry belongs in specification Appendix C.
  6. Coherence gate — the handbook security/subsystem pages, the RTM, and the workflow catalog entry for the self-heal drain if the queue’s access contract is now stated explicitly.

07Reproduce this analysis

# views over the six tables, and whether they are security_invoker
docker exec supabase_db_SPWSranklist psql -U postgres -d postgres -c "
SELECT c.relname, coalesce(c.reloptions::text,'-'), pg_get_userbyid(c.relowner)
FROM pg_class c JOIN pg_namespace n ON n.oid=c.relnamespace
WHERE n.nspname='public' AND c.relkind IN ('v','m')
  AND EXISTS (SELECT 1 FROM pg_depend d JOIN pg_rewrite r ON r.oid=d.objid
              JOIN pg_class t ON t.oid=d.refobjid
              WHERE r.ev_class=c.oid AND t.relname LIKE 'tbl_%draft%');"

# every SECURITY DEFINER function anon may execute
docker exec supabase_db_SPWSranklist psql -U postgres -d postgres -c "
SELECT p.proname, p.prosecdef, p.proacl
FROM pg_proc p JOIN pg_namespace n ON n.oid=p.pronamespace
WHERE n.nspname='public' AND p.prokind='f' AND p.prosecdef
  AND has_function_privilege('anon', p.oid, 'EXECUTE') ORDER BY 1;"

# read-only proof that anon really reaches the data (rolled back, no writes)
docker exec supabase_db_SPWSranklist psql -U postgres -d postgres -c "
BEGIN; SET LOCAL ROLE anon;
SELECT count(*) FROM vw_fencer_aliases; SELECT count(*) FROM tbl_recompute_queue;
ROLLBACK;"

08Open questions, revised

  1. Scope of the migration: the six tables only, or the full anon surface — tables, the alias view and the five RPCs — in one change? Recommendation: one change. They share a root cause and a test, and splitting means shipping a security migration you already know is incomplete.
  2. Audit PROD before or after locking? Unchanged from the handover, and it is now more pressing: tbl_tournament and tbl_registration were writable through DEFINER RPCs, so the audit surface is larger than the six tables. Locking first is still the safer order.
  3. Recreate vw_fencer_aliases with security_invoker = true, or just fix its grants? Grants alone are the small, safe diff; security_invoker is correct but couples the view to whatever RLS the draft tables end up with.
