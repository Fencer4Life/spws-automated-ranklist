EVF → CERT → PROD · System explainer · 2026-07-14

How EVF calendar sync works now

The European Veterans Fencing calendar is scraped into CERT, and a season-scoped reconciler mirrors CERT's active-season events onto PROD's public calendar. As of the 2026-07-14 incident it converges PROD to CERT twice per run — once before the scrape and once after — and refuses to run at all when the two environments disagree about which season is active.

3 jobs · 1 scheduled run idempotent · code-keyed CUD diff fail-loud season guard

The whole thing on one page

High-level architecture

Three tiers, one direction of flow. EVF is the outside world's record of which veteran tournaments exist. CERT is SPWS's own source of truth — the scraper writes there and only there. PROD is a read replica of CERT's calendar that the public sees; nothing writes to PROD's event set directly. A single scheduled workflow drives all of it, and a reconciler — not a copy script — keeps PROD converged to CERT.

Reads down, never up

One-way convergence

EVF is never written. CERT is written only by the scraper. PROD's event set is written only by the reconciler. No step ever pushes data back up a tier.

A diff, not a copy

Reconciler, not export

fn_mirror_events_to_prod compares CERT and PROD for the active season and applies the difference — so running it twice, or a hundred times, converges to the same state.

Who owns what

Three tiers, one source of truth each

The whole design rests on each tier having exactly one writer, so there is never a question of which copy is authoritative.

TierAuthoritative forWritten by
EVF / OphardtWhich veteran tournaments exist in the world, and their detail-page slugsNobody here — it is the external source we read
CERTThe SPWS calendar: event codes, seasons, organizers, dates, statusThe EVF scraper (fn_ingest_evf_calendar), plus admins editing directly in CERT

▸ Why PROD owns nothing

Because PROD is defined as "whatever CERT's active season currently holds", the reconciler is free to delete a PROD event that no longer exists on CERT without losing information — the truth was always on CERT. That is only safe because nothing writes event data straight into PROD; the single-writer rule is what makes convergence, and deletion, sound.

The scheduled run, job by job

Promote runs twice — before and after the scrape

The older evf-sync.yml reconciled CERT→PROD only after scraping. That left a gap: if the scrape step failed, or an admin had edited CERT by hand, PROD had no way to catch up until the next fully-successful scrape. The workflow now brackets the scrape with the same reconciler on both sides, as three jobs in one run:

Job 1 — promote-calendar-pre

Runs the reconciler first, converging PROD to whatever CERT already holds, before anything new is written. This is what makes a manual CERT edit — or a season-row fix — reach PROD on the very next scheduled run, even on a day the scrape itself later errors out.

Job 2 — sync

Scrapes the EVF calendar and writes new or changed events into CERT only. It dedupes each incoming event in three escalating passes — exact EVF id, then detail-page slug (scoped to the season), then a fuzzy name/location match — before allocating a fresh SPWS event code. It needs job 1 but runs regardless of job 1's outcome: a broken PROD promote must never block fresh events from being scraped into CERT.

Job 3 — promote-calendar

Runs the reconciler again after the scrape, this time picking up whatever job 2 just wrote into CERT. This is the job that actually delivers the new events to the public calendar. It is wired so that an earlier job's failure cannot strand it — if the scrape produced new CERT events, this promote still runs to push them to PROD.

▸ Why bracket instead of promoting once, at the end

A pre-scrape promote means CERT-side changes that did not come from a scrape — an admin edit, a season fix — converge to PROD on the next run no matter what the scrape does. The post-scrape promote keeps its original job: deliver whatever was just scraped. Because fn_mirror_events_to_prod is a full create/update/delete diff keyed on the event code, running it twice in one workflow is a no-op whenever CERT and PROD already agree — never a duplicate.

The engine in the middle

How the reconciler converges PROD to CERT

Both promote jobs call the same Python entry point — promote --mode calendar — which reads the two environments, computes a difference, and applies it through one atomic RPC on PROD. Four properties make it safe to run on a schedule, unattended:

1 · Season-scoped

It only ever reconciles the active season. It reads the bool_active = TRUE season on both CERT and PROD and works entirely within it. Events in past or future seasons are never touched by calendar sync — those move only through explicit season promotion.

2 · Keyed by event code, never by raw id

CERT and PROD assign their own serial ids, so they diverge. The reconciler matches events by txt_code (the SPWS event code) and resolves every foreign key — organizer, prior-event carry-over link — by code on the PROD side, never by copying a CERT id across the boundary. This is the same cross-environment-id discipline the per-event promoter uses.

3 · A create / update / delete diff

Having read both sides, it partitions the active season's events into three sets and hands them to fn_mirror_events_to_prod in one call:

  • Create — codes present on CERT but missing on PROD.
  • Update — codes on both sides. Updates are fill-blank-only: they populate empty PROD fields from CERT but do not clobber a value PROD already has, so a deliberate PROD-side value is preserved.
  • Delete — codes on PROD but no longer on CERT. Deletes are guarded: an event PROD still has children or results for is reported as skipped rather than force-removed.

4 · Idempotent by construction

Because every action is derived from the current difference between the two environments, applying it moves PROD exactly onto CERT and no further. A second run finds no difference and does nothing. That is what lets both promote jobs — and any number of retries — run without fear of duplication.

The refusal that protects PROD

Why it fails loud on a season mismatch

Before computing any diff, the reconciler checks that CERT's active season and PROD's active season are the same season code. If they differ, it raises and stops — it does not guess.

promote_calendar: active season mismatch — CERT is on
  'SPWS-2026-2027', PROD is on 'SPWS-2025-2026'.
Bootstrap the new season on PROD via promote_season.py first.

The reason is subtle but important: the diff is computed within one season's id. If CERT had already rolled onto a new season that PROD was never bootstrapped onto, then comparing CERT's new-season events against PROD's old-season events would file every create under the wrong season and propose deleting PROD's entire outgoing season as "not on CERT". The guard turns that catastrophe into a clean, loud stop.

✓ The fix is always "make the seasons agree", never "override the guard"

This is the same guard that prevented an earlier cross-season duplication bug. When it fired on 2026-07-14 it was doing its job — the correct response was to bootstrap / sync PROD's season row so the two environments agreed, after which the guard passed on its own and the events flowed.

Why nothing showed up — 2026-07-14

Two bugs, each hiding the other

The symptom was simple: EVF published 14 new circuit stops, the scraper ran on schedule, and PROD's public calendar never changed. Investigation found two unrelated defects stacked on top of each other — fixing only one would not have been enough.

1 × A uniqueness index that should have been per-season

idx_tbl_event_evf_slug enforced one EVF detail-page slug ever, across all seasons. But EVF reuses a stop's slug for its next yearly edition (Munich 2026 and Munich 2027 share a slug). Every batch insert of new events collided on this constraint and rolled back as a whole — so even events with no real conflict were silently discarded, every run.

2 × PROD's active season had quietly fallen behind CERT's

An admin edited SPWS-2026-2027's dt_start on CERT (and shrank the prior season's dt_end to keep the ranges non-overlapping), which activated CERT's new season early. PROD's copy never got the same edit, so PROD still had the old season marked active — and the reconciler's own active-season guard correctly refused to promote across the mismatch.

Fixing #1 let CERT accumulate the events correctly. Fixing #2 let PROD's guard stand down and actually receive them. Neither fix alone would have produced a working pipeline.

Also fixed while in there

A push race in the bookkeeping commits

Both release.yml (its tracking-file commits) and evf-sync.yml (its reconcile-report commits) end with a plain git commit && git push against main. Two runs landing close together — plausible when a season fix triggers Release and a manual EVF sync in the same few minutes — would both diff origin at the same instant, and one push would be rejected.

! Non-critical, but real

This never rolled back actual data — only the trailing bookkeeping commit failed, after the real CERT→PROD reconcile had already succeeded. Both workflows now retry the push up to five times, re-fetching and rebasing onto origin/main between attempts. The committed file is always freshly generated and uniquely named, so a rebase here is never a real conflict — just a replay onto a moved tip.

Proof it is stable at rest

Two consecutive runs, back to back

Idempotency was verified directly, not assumed — the same workflow dispatched twice in a row, immediately after the season fix landed on PROD:
