The honest ceiling on automation Steps 1, 3, 4, 5 and 7 can be fully automatic. Step 6 should not be, and step 2 cannot be. Beyond that there is a hard floor: 43 of 144 fencers have no email and no phone, 9 of them inside a top 5. No amount of scripting reaches them — that part of the campaign is a human working the Facebook group and club contacts.

01Where it runs — the LOCAL Ranklist environment

Decided: the campaign runs against the local Supabase stack, in throwaway tables with throwaway code. That is a good call — it buys real queries, transactions and a mail catcher for free. It also introduces exactly one serious hazard, which the rest of this section is about.

The hazard: your local database is designed to be destroyed ./scripts/reset-dev.sh runs supabase db reset, which drops and rebuilds the database from 131 migrations and the seed dump. You run it routinely. The ranking tables survive that because they are regenerable — the campaign's declarations are not. A fencer's answer, once destroyed, cannot be recovered. So the whole question of "where do the throwaway tables live" is really the question of how far they sit from the blast radius of a reset.

Three levels of isolation

Tables in public

rejected

Alongside tbl_fencer and friends. Superficially the easiest.

  • Wiped by every db reset.
  • supabase db diff would see them and try to generate migrations.
  • Worst: they could be captured into a seed dump — putting 99 people's phone numbers into a committed file. That alone disqualifies it.

Separate schema msw

workable

Same database, own namespace. Clean separation of names, and PostgREST would expose it if you granted it — so a self-hosted response form becomes possible.

  • Still inside the database that db reset drops.
  • Still visible to diff and dump tooling; safety now depends on you remembering to exclude it.
  • Choose this only if you want the free API.

Separate database msw_tbilisi

recommended

Same Postgres 17 container on 127.0.0.1:54322, different database. One CREATE DATABASE away.

  • supabase db reset targets the postgres database only — it does not touch this one. Reset as often as you like.
  • Invisible to migrations, to db diff, and to seed dumps. The personal-data leak path is closed by construction, not by discipline.
  • Same server, same psql, same connection habits, same Python driver.
  • Cost: no PostgREST, so no free API. You do not need one — Google Forms is the collection layer.
The remaining hole, and the one line that closes it A separate database survives db reset, but it does not survive the Docker volume being pruned — supabase stop --no-backup, a Docker Desktop cleanup, a machine rebuild. So: a nightly pg_dump of msw_tbilisi into the gitignored campaign folder, kept for the four weeks the campaign runs. It is one cron line and it is the difference between an inconvenience and phoning 98 people to ask what they already told you.

What using the LOCAL stack actually buys you

  • Inbucket on localhost:54324. The Supabase stack already runs a mail catcher. You can execute the entire send — real templates, real personalised links, all 98 recipients — into Inbucket and read the messages exactly as they would arrive, without a single byte reaching a real person. This is a materially better dry run than printing to stdout, and it is the single strongest argument for the local environment over plain files.
  • Queries instead of loops. "Which categories have fewer than five confirmations, and who is next in rank" is one SQL statement against a view, not bookkeeping in Python.
  • Transactions. Recording a send and its ledger entry either both happen or neither does.
  • Studio on localhost:54323 for eyeballing and hand-fixing a row when a phone answer needs correcting — better than editing a JSONL file.

Rules for the throwaway code

  • Schema lives in a single re-runnable schema.sql in the campaign folder. Never in supabase/migrations/ — a file there becomes part of the product and would run on CERT and PROD.
  • Scripts refuse to start unless the connection string names msw_tbilisi. A guard clause, three lines, prevents the entire class of accident where a campaign script points at the ranklist database.
  • No ADR, no RTM entry, no handbook page, no tests beyond what you need to trust a send. This is deliberately outside the documented system.
  • Code lives in the gitignored campaign folder with the data, so the whole campaign is one directory you delete when it is over.

02Sending

Transport

Gmail SMTP with an app password is the shortest path and matches what python/pipeline/ftl_feed_seed_send.py already does. The Gmail API is cleaner but costs you an OAuth consent flow for no benefit at this volume. Consumer Gmail allows about 500 recipients per day; you need 98, so the cap is irrelevant — but the similarity of 98 near-identical messages is not.

Deliverability, the actual risk 98 near-identical mails sent in a burst from a personal Gmail is the classic spam-filter signature. Mitigations, all cheap: genuinely personalise the body (name, their weapons, their ranking positions), send with a randomised 20–60 second gap rather than a loop, use a plain-text or very simple HTML body, no images, no tracking pixel, and no URL shortener — a bare Google Forms link is trusted, a bit.ly link is not. Send yourself and two friendly fencers a test first and check it lands in the inbox, not Promotions.

The link

Google Forms supports pre-filled links: you build one from the form's "Get pre-filled link" and swap in per-person values as URL parameters. That gives each fencer a link that already knows who they are.

Prefill an opaque token as well as the name — a short random string per person, stored in your local ledger. The name alone is a weak key (you already have two Janusz Młynek and a Gęzikiewicz/Gręzikiewicz pair), and a token makes attribution exact even if someone edits the name field.

The multi-weapon problem, solved by prefill 29 fencers qualify in more than one weapon, and Google Forms cannot vary its answer options per respondent. The trick: give the form three fixed question groups — szpada, floret, szabla — each offering indywidualnie: tak / nie and drużyna: tak / nie, plus a nie dotyczy option. Then prefill nie dotyczy for every weapon the person is not ranked in. A one-weapon fencer sees two real questions and four already-answered ones; a three-weapon fencer sees six real questions. No branching logic, no wall of checkboxes, one response row per person.

03Collecting

Google Form writing to a Sheet is the right choice and I would not change it. It is always available, it needs no server, and people trust the interface.

Read it from Python with gspread and a cached OAuth token. Do not use the "publish to web as CSV" shortcut — that puts declarations behind an unlisted but unauthenticated URL, and this is personal data.

The Sheet is an inbox, not the source of truth Treat form responses as append-only incoming mail that the script reads and never writes to. The campaign's real state lives in a local ledger, because a large share of answers will arrive by phone, by Messenger, or via a club official — and those have no form row. If the Sheet is the source of truth, every phone answer forces you to fake a form submission. With odpowiedz as the ledger, a phone answer is just another row with zrodlo = 'telefon'.

04The throwaway schema

Moving into Postgres answers your earlier objection to CSV cleanly: nothing you read is a data file any more. The tables are machine storage, and the only human artifact is generated HTML.

The organising principle is the split between tables that can be rebuilt from source at any time, and tables that hold answers which exist nowhere else.

TableClassHoldsRebuild policy
kandydatderived 144 fencers: name, gender, V-category, match key. Truncate and reload from the 24 workbooks whenever the ranking changes.
startderived 173 rows: fencer × weapon × category × ranking position. Truncate and reload. This is what "top 5" is computed from.
kontaktderived Email and phone, plus which source they came from. Truncate and reload from the contact list.
zaproszenieappend-only Every invitation actually sent: person, wave, channel, token, timestamp. Never truncated. This is what makes re-running a send safe — the sender skips anyone already present for that wave.
odpowiedzappend-only Every answer: individual and team intent per weapon, plus the source — formularz, telefon, facebook, klub. Never truncated, never updated. A changed mind is a new row, not an edit. The irreplaceable table.

Two views carry the whole campaign

  • v_status — current declaration per fencer per start, taking the latest odpowiedz row. Everything the report renders reads from here.
  • v_kategoria — per category: confirmations, whether it is below five, whether anyone is left to invite, and who is next by ranking position. This one query drives both the backfill queue and the board report's readiness line.
Why append-only rather than a status column A single status column updated in place looks simpler and is a trap. Answers arrive through four channels, sometimes contradict each other, sometimes arrive twice, and occasionally get entered against the wrong person and need unwinding. An event log makes all of that recoverable and makes "what did we actually know on the day the board met" answerable. It costs one view.

The only file that stays a file is status.html — generated from the views, disposable, regenerated after every reconcile.

05Cascading on a refusal

Your proposal was: refusal confirmed → next fencer drawn in → email initiated automatically. I would change one word in that.

Propose, don't send Let the script detect the refusal, work out whether the category is actually short, pick the next eligible fencer and write the drafted invitation into a queue — then stop. You approve the queue with one command. Two reasons: an invitation to compete for Poland is not a message you want a cron job sending on a name-matching bug at 03:00, and it is irreversible in a way nothing else in this pipeline is. The automation still does all the work; you keep the send button. This is the same per-event sign-off rule you already apply to ingestion.

When a refusal actually triggers a backfill

  • Only if confirmations in that category have dropped below 5 — four starters plus the reserve. A refusal in a category that still has 5 green needs nothing.
  • Only if there is anyone left. 11 of 24 categories have fewer than 5 ranked fencers, and szpada K3 has none at all — for those, the pool is exhausted from the start and the report should say so rather than silently show a short list.
  • Next by ranking position, never renumbered — the design doc already locked that.

06Escalation

Assuming a 14-day window. Amber dots are the points where you act rather than the script.

Day 0 · automatic
Personalised email, 98 recipients
Deadline stated explicitly and repeated in the form header.
Day 0–2 · manual
Facebook group post + club contacts
The only route to the 43 unreachable, and it needs the most lead time, so it starts immediately rather than last.
Day 4 · automatic
Email reminder to non-responders
Same thread, short body. Typically recovers 15–20%.
Day 7 · automatic
SMS to non-responders who have a number
Different channel beats a third email. One sentence and the link. 93 numbers at ~0.12 PLN is under 12 PLN for the whole list.
Day 10 · manual
Phone calls — top 5 only, still silent
Should be 10–20 calls by this point, not 100. Answers go straight into the ledger as źródło: telefon.
Day 12 · automatic
48-hour deadline warning
The single highest-yield message in the sequence. Everyone still silent, every channel available.
Day 14 · manual
Close, freeze, report to the board
Late answers still accepted into the ledger; the report simply regenerates.
Escalate on silence, not on opens There is no reliable way to know whether a Gmail message was opened, and the only way to approximate it — a tracking pixel — is both unreliable behind image proxies and a bad look for a federation writing to its own members. Trigger every escalation off the absence of a form response, which is the signal you actually care about.

07Four things I would add

Idempotent by construction

Every sender checks zaproszenie before sending and skips anyone already invited at that wave — a unique constraint on (fencer, wave) makes it impossible rather than merely unlikely. A scheduled job that can double-send is a job that eventually will — and 98 people receiving the same invitation twice is the failure mode that costs you credibility.

Dry run as the default

The send script targets Inbucket unless given an explicit --live. Every rehearsal is a complete, realistic run you can read in a browser — and every schedule mistake is harmless until the day you mean it.

Daily digest to Telegram

You already run Telegram as the alerting surface. A one-line nightly summary — replies today, categories now short, backfills waiting for approval — means you never have to remember to open the HTML to find out whether something needs you.

Regenerate on read, not on a timer

The status page is cheap to build, so rebuild it after every reconcile rather than keeping a stale file and a separate refresh job. Fewer moving parts, and the page is never lying about how fresh it is.

08Open questions

  • Sending identity. Your personal Gmail, or an SPWS-branded address? A federation address materially improves both open rate and deliverability, and it matters more than any other single choice here.
  • Deadline. 14 days is my assumption, not a decision. It drives the whole escalation ladder.
  • Facebook group. Still unanswered from before, and it decides whether the 43 unreachable are recoverable.
  • Response form. Google Forms, or your own page served from the local stack through a tunnel? The tunnel removes the prefill workaround entirely and lands answers straight in odpowiedz — but a stranger-looking URL and an uptime dependency on your machine are real costs with this audience. I lean Forms.
  • SMS gateway. Worth ~12 PLN and an account, or do you want to send those by hand from the phone?
  • Team fee. The team question carries a cost the fencer must accept. Does the form state the amount, or only ask about willingness in principle?
