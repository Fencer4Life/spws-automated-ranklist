02My side — and one design call I'm making

Separate venv for the campaign, not the repo's The repo's .venv and pyproject.toml belong to the ranking system. The campaign is explicitly outside it — no ADR, no integration. Adding google-api-python-client and two auth libraries to the production dependency set for a throwaway campaign is the wrong trade.

So: a venv inside doc/external_files/MSW-Tbilisi-2026/, gitignored with everything else, holding openpyxl plus the Google libraries. Disposable — delete the folder in September and the ranking system never knew it existed.

Cost, stated plainly: the three existing generators currently run on the repo venv. They'd move to the campaign venv, so there's one environment for campaign work rather than two. Say if you'd rather I just add the libraries to pyproject.toml instead.

Scopes I'll request

https://www.googleapis.com/auth/gmail.send
https://www.googleapis.com/auth/spreadsheets.readonly

Send-only on Gmail — no read access to the mailbox — and read-only on Sheets. This was question 3 on the previous sheet; I'm proceeding with the minimal answer unless you say otherwise. Widening later means re-running the auth script, nothing worse.

What I'll write

  • auth_setup.py — one-time OAuth flow, writes token.json next to the credentials. Prints which account it authorised, so a wrong-account mistake is visible immediately rather than three hundred mails later.
  • send_test.py — sends exactly one mail, to spws.weterani@gmail.com itself, rendering the real wave-1 template. Proves authentication, sender display name, Polish diacritics and formatting in one shot.
Why a test mail to yourself first Encoding is where this breaks quietly. A misconfigured MIME header turns "Paweł" into "PaweÅ‚" in some clients and looks perfect in others — and diacritics have already caused one silent bug in this campaign (handover mistake #2). Seeing it land in a real inbox proves what no unit test proves.
