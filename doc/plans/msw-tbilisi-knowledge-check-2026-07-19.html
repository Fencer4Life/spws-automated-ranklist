01How responses are persisted

Google Forms is the store of record. responses.json is a derived view. No spreadsheet is linked anywhere in the design.

read_responses.py calls forms.responses().list() on each of the seven variants, paginating through nextPageToken. Answers arrive keyed by questionId, and question_map() maps each one back to the question title — never a column index, because a one-weapon variant and a three-weapon variant have different layouts and position would silently mean different things per fencer.

Each response is resolved to a fencer by resolve(), in descending order of reliability: kod → nazwisko → e-mail. The route taken is stored as matched_by. The prefilled code is a hint rather than a key — Forms has no read-only field property, so a fencer can clear or mangle it.

Why the derived file is safe to delete A changed mind arrives as a second submission and collapses via rec["submitted"] > prev["submitted"] — latest wins. Only this local view collapses; every submission remains in Google's response list, so the append-only history is intact and responses.json can be rebuilt from scratch at any time by re-running the reader.

03The header defect

Your hypothesis was wrong; your conclusion was right The suspicion was that aggregating the polls over-counts fencers. It does the opposite — aggregation is precisely what removes the duplicates. But the header is still wrong, in the denominator rather than the dedupe.

The dedupe is correct

QuantityMeaning
top-five slots92 summed over the 23 populated categories, capped at 5 each
− duplicates18 the same fencer holding a slot in a second weapon or category
distinct people74 what the report shows. Already deduplicated.

8 of the 23 populated categories hold fewer than five ranked fencers, so no reserve exists there. A 24th category exists on paper with nobody ranked in it.

The denominator is not the invited population

gen_raport.py:475 renders {n_confirmed}/{n_emailed or n_mail} under the label Potwierdziło start indywidualny, immediately beside a card reading W pierwszej piątce — 74 — zbiór zaproszeń: 4 + rezerwowy. The board reads those two cards as one story. They are two different populations.

So the board is shown 0 / 103 against a stated invitation set of 74. Two consequences:

  • The ratio cannot be read as squad readiness. 36 of the 103 were never in any top five — they are not being invited to a place, and their answers dilute the headline.
  • The 7 who matter most vanish. Fencers in a top five with no e-mail are absent from both numerator and denominator, so the one group needing a phone call is invisible in the number the board looks at.

A second-order fault in the same expression

The population changes underneath the label n_emailed or n_mail evaluates to 103 before the first send ("everyone we hold an address for") and to n_emailed afterwards ("everyone actually mailed"). If --top5 is ever used it becomes roughly 67. The same card silently means three different things depending on when it is generated and which flags were passed. A KPI whose denominator is defined by history rather than by policy cannot be compared against itself over time.

Proposed fix

  1. Denominator becomes 67 — top five and has an e-mail — fixed by policy, identical before and after the send.
  2. The 36 non-top-five respondents get their own card. They are not noise: they are self-funding volunteers, and worth counting where they cannot dilute squad readiness.
  3. 7 stays visible beside the ratio, as unreachable-and-needed.

What changed in the code

  • Two named populations. squad = top5 ∧ email and vol = ¬top5 ∧ email, with _confirmed/_declined/_waiting now taking a pool argument instead of iterating all 144.
  • The denominator no longer depends on history. n_emailed or n_mail is gone from the KPI. n_squad is fixed by policy, so the ratio is comparable against itself before the send, after it, and with or without --top5.
  • The gap is derived, not re-measured. n_gap5 = n_top5 - n_squad, so the card cannot drift out of agreement with the invitation set it sits beside.
  • A duplicate definition removed. n_mail5 computed the same quantity as n_squad under a second name; it is deleted, with a comment saying why it must not come back.
Why 74 = 67 + 7 holds exactly All seven top-five fencers without an e-mail turn out to have no phone either, and the single phone-only fencer in the roster is outside every top five. The invitation set therefore splits with no remainder, which is why the header can explain itself on its face rather than in a footnote. This is a property of today's data, not an invariant — n_gap5 is derived by subtraction precisely so that it stays correct if a phone-only fencer ever enters a top five.

Left alone deliberately

The legend still reads mamy adres e-mail — 103 z 144. That is the @ marker's own scope — every fencer in the table we hold an address for — and it is labelled precisely, so it is not the same claim the KPI was making. Also unchanged: send_wave1.py still queues all 103 by default. The report now distinguishes the two groups; the send policy is a separate decision.

05Live dashboard — no, we do not have it

Answer: nothing of this exists today raport.html is a static file, generated once, by hand. Checked on 19 July: no <meta http-equiv="refresh">, no setInterval, no fetch, no EventSource anywhere in the output; no shell wrapper in scripts/; no crontab entry and no LaunchAgent. Following progress today means running two commands and reloading the tab yourself.

What is actually there

So the data half is done and correct. What is missing is a loop, a page reload, and — the part that is easy to skip — a way for the page to say how old it is.

The failure mode that makes this worth building carefully A dashboard that silently stops updating does not look broken. It looks like nobody is replying. If the loop dies at 09:10 and you glance at the tab at 14:00, the honest reading and the broken reading are the same picture. The page must carry its own timestamp and turn visibly stale when the last refresh is older than a couple of cycles. Without that, the dashboard is worse than reloading by hand, because manual reloading at least tells you it ran.

The specific thing that will kill the loop

The OAuth token expires every 7 days — the app is in Google's Testing mode. A background loop will hit RefreshError mid-campaign, and if it logs to stderr in a terminal nobody is watching, it is invisible. The refresh failure has to surface in the report itself, as a red banner, alongside the instruction to re-run auth_setup.py. The same applies to any API error: the loop must never exit quietly, and must never overwrite a good report with a half-rendered one.

What it would take

  1. A watch.py: read responses → regenerate report → sleep 300 → repeat. Both existing scripts are importable, so this is a loop around code that already works, not new logic.
  2. Atomic writes. Write to a temp file and os.replace(). The browser will otherwise sometimes fetch a half-written file and render a broken page — a 90 KB report is not written instantaneously.
  3. Self-refresh in the page. A small script that reloads on a timer and restores scroll position. A bare <meta refresh> jumps you back to the top of a long report every cycle, which makes it unusable for reading the category tables.
  4. A freshness strip. Last-updated time, cycle count, and the loop's health — green while refreshing, red on auth failure or repeated errors.

Cost — it is cheap

One cycle is roughly 14 Forms API calls: a get and a responses.list per variant, times seven. At a five-minute cadence that is about 4 000 calls a day, comfortably inside Google's per-minute read quota, and it touches a different API from the sender — so the watcher can run while send_wave1.py is still working through the queue. That is actually the most useful window: because the report already reads send_log.jsonl, the dashboard would show the send itself progressing, not just the replies.

One rail must not be relaxed for this The watcher is read-only against Google. It must never call anything that deletes or modifies form responses. "Delete all responses" stays a manual, pre-send-only action — no automated process should be in a position to reach it.

06Open questions

  1. Should I build watch.py before wave 1 or after? My recommendation: build it before, start it before the send. It is perhaps an hour of work, it needs no new API scopes, and starting it first means the send itself is observable rather than opaque for 29 minutes. Building it afterwards means writing code while responses are already arriving.
  2. Wave 1 to all 103, or the 67 first? Still open from before. The report is honest either way — this only affects who receives mail on the first pass.
