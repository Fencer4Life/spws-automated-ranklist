Is the task clear? · What "master table" and "BY status" mean here

Confirmed against the actual schema, not assumed

tbl_fencer (supabase/migrations/20250301000001_enums_tables_indexes.sql:60) is the fencer master table. Birth year lives in int_birth_year; "BY status" is bool_birth_year_estimated — TRUE = Estimated, FALSE = confirmed/Exact (ADR-035, the Birth-Year Review tab mockup at doc/mockups/fencer-birth-year-review.html). Setting a birth year and marking it confirmed is exactly the contract of the existing, already-tested RPC fn_update_fencer_birth_year(p_fencer_id, p_birth_year, p_estimated := FALSE) (supabase/migrations/20260412000004_fn_update_fencer_birth_year.sql, pgTAP 13_fencer_birth_year.sql 13.1–13.4). This plan reuses that RPC rather than a raw UPDATE — no new SQL surface, no new pgTAP required.

Age category (V0–V4) is derived, never stored: fn_age_category(int_birth_year, season_end_year) (20250303000002_age_category_by_season.sql). "Category before/after" in the summary table below is this function evaluated at the current active season's end year, 2026 (SPWS-2025-2026, dt_end = 2026-07-15), against the old and new birth years.

How matching was done

Name matching against LOCAL's 349-row roster, three passes

  1. Exact match on normalized txt_surname txt_first_name (case-folded, whitespace-collapsed). Resolved 185 of 216 names outright, all already-correct or with a simple year change.
  2. Qualifier-stripped match for master-list entries carrying a disambiguator in parentheses — KRAWCZYK Paweł (source qualifier: "GLIWICE"), MŁYNEK Janusz (source qualifier: "szabla") — matched with the qualifier removed, then verified the qualifier actually fits (see below).
  3. Alias and typo review for anything left unmatched: checked tbl_fencer.json_name_aliases (the identity subsystem's existing alias mechanism, doc/handbook/subsystems/identity-and-recompute.html) and manually reviewed every candidate a naive fuzzy-string match surfaced — rejecting the ones that were same-shape-different-person coincidences rather than trusting the string similarity score.

All matching ran against a live dump of LOCAL's tbl_fencer (349 rows, queried directly, not assumed). The generated SQL below resolves each fencer by name at execution time (a subquery on txt_surname/txt_first_name), never by LOCAL's numeric id_fencer — those ids are per-environment SERIAL values and are not guaranteed to line up across LOCAL/CERT/PROD (the same principle the environments doc states for event/season promotion: resolve by stable identity, never assume ids match).

Category before/after below is checked across every season each fencer has a result in (SPWS-2023-2024, SPWS-2024-2025, SPWS-2025-2026 — LOCAL's full result history), not only the current season. Corrected after checking the graph + the actual identity-subsystem docs

A birth-year change is not just an UPDATE — it must recompute

The owning handbook page (doc/handbook/subsystems/identity-and-recompute.html) states an invariant my first draft of this plan missed: "Birth-year or merge corrections recompute all affected events, not only the currently viewed ranking." This is real machinery (ADR-072), not a suggestion:

  • trg_fencer_change_enqueue (20260615000001_cdc_recompute_dedup.sql) — a column-aware AFTER UPDATE ON tbl_fencer trigger. It fires whenever int_birth_year (or txt_nationality) actually changes and enqueues every event that fencer has a result in, into tbl_recompute_queue. This fires no matter how the UPDATE happens — RPC call from the admin UI, or fn_update_fencer_birth_year called from a migration script, identically. So the SQL in this plan does trigger the self-heal correctly; the gap is entirely on the drain side, below.
  • A debounced worker (python/pipeline/recompute/worker.py) drains the queue and re-derives V-cat/score per affected event (RECOMPUTE_DOMESTIC). It does not run itself — something has to invoke it.

✓ Resolved 2026-07-14 — the drain now runs on PROD too

This was a real gap when first found: .github/workflows/recompute-drain.yml's own header said "CERT only. LOCAL stays manual", and nothing drained a PROD tbl_recompute_queue. It mattered concretely, not just theoretically — the per-season category check below found that SAMECKA-NACZYŃSKA Martyna's correction genuinely re-brackets her SPWS-2024-2025 results (V0→V1), so a never-drained PROD queue entry would not have been harmless for this batch.

Amended as ADR-072's 2026-07-14 amendment: .github/workflows/recompute-drain-prod.yml now drains PROD every 15 minutes, joining the existing prod-write concurrency group so it never races a season/event-result promotion. Full investigation and resolution record: doc/audits/prod-recompute-drain-gap-2026-07-14.html. CERT's job is unchanged.
