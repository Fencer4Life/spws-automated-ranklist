SPWS · Implementation plan · 22 July 2026 · security hardening

Securing the project: the advisory is the smallest of the three holes.

Asked to go deeper than the RLS advisory, I swept auth configuration, the Edge Function, CI/CD and secret hygiene. Three findings are more severe than the six tables Supabase reported, and all three are the same failure mode the project already diagnosed once in ADR-016: a control that exists in the user interface and is never enforced by the server. This plan closes all of them in four waves. Nothing has been changed yet.

3 critical · PROD-reachable anon key → GitHub Actions dispatch self-signup → full admin MFA not enforced in SQL 6 tables + 1 view + 5 RPCs 4 areas verified clear nothing changed

00Context — why this plan exists

A Supabase advisory arrived on 21 July reporting rls_disabled_in_public on CERT and PROD. The 21 July handover traced it to six pipeline tables. My 22 July follow-up showed that the fix proposed there would not actually close the exposure, because an owner-rights view and five anonymously-callable RPCs sit outside its scope.

You then asked how to secure the project, not just the advisory. That sweep found something worse. The advisory describes six staging tables that an attacker could corrupt. It does not describe the two paths by which an anonymous caller can reach production behaviour directly: the workflow-dispatch Edge Function, and self-service account creation. Those are ranked first below because they are live today and neither requires a database migration to close.

Read before acting 1. Every claim here was read back from live LOCAL state, from source, or from gh. Nothing was inferred from the advisory text.
2. PROD was not queried. The plugin:supabase MCP server is unauthorised and this session is non-interactive, so the OAuth flow cannot run here. Authorise it via claude mcp or /mcp in an interactive session to inspect CERT/PROD directly.
3. The only probes performed against live systems were SELECTs inside a rolled-back transaction on LOCAL, and one unauthenticated GET to the hosted /auth/v1/settings endpoints, which correctly returned 401. No account was created, no write attempted, no credential handled.

01Findings, ranked

S1 — Critical, reachable on PROD today

S1-1 · The workflow-dispatch Edge Function performs no caller authorization

supabase/functions/dispatch-workflow/index.ts reads GH_DISPATCH_PAT, validates the requested workflow against a hard-coded allowlist, and dispatches it. It never inspects who is calling. Its own header comment states the intent — "Authenticated callers POST { workflow, inputs }" — that the code does not implement. There is no Authorization header read, no auth.getUser(), no role check anywhere in the handler.

The platform setting at supabase/config.toml:379-380 is verify_jwt = true, which looks like protection and is not:

[functions.dispatch-workflow]
verify_jwt = true

verify_jwt asserts only that the request carries a JWT signed by the project's secret. The anon key is precisely such a JWT — it is a signed token with "role":"anon", and it ships in the frontend bundle by design. The gate is satisfied by the very key that is public.

Consequence: anyone holding the anon key can POST to https://<ref>.supabase.co/functions/v1/dispatch-workflow and trigger any of seven workflows on main, with arbitrary inputs, using a PAT scoped to Actions read+write. The allowlist is:

WorkflowWhat an anonymous caller gains
promote-season.ymlCERT → PROD promotion, or deletion of a season skeleton on PROD (ADR-077).
ftl-seed.ymlOutbound organizer delivery (ADR-080 §5) — an action that leaves the system and reaches real people.
ingest-event.ymlRe-ingest an event from a URL with attacker-chosen source overrides and keep-rules.
phase5-event-runner.ymlDrive the staging/consolidation runner for an arbitrary event.
scrape-tournament.ymlForce scrapes — outbound requests from your runners, attacker-chosen target.
populate-urls.ymlRewrite tournament URL population.
regen-report.ymlRegenerate verdict reports; cheapest of the set, still unauthorised compute.

CORS is Access-Control-Allow-Origin: "*", so this is also callable cross-origin from any web page a signed-in admin happens to visit. This is the single largest hole in the system: it converts a deliberately public key into remote execution against production.

S1-2 · Self-service signup grants full administrative privilege

LOCAL GoTrue, read directly:

GET http://127.0.0.1:54321/auth/v1/settings
{ "external": { "email": true, ... },
  "disable_signup": false,
  "mailer_autoconfirm": true, ... }

matching supabase/config.toml:174 (enable_signup = true) and :209 ([auth.email] enable_signup = true, enable_confirmations = false). Account creation is self-service and instantly confirmed.

Now pair that with the authorization model. All seventeen policies across the ten "healthy" tables reduce to one predicate:

tbl_fencer         | Admin all fencers   | ALL    | auth.role() = 'authenticated'
tbl_result         | Admin all results   | ALL    | auth.role() = 'authenticated'
tbl_scoring_config | Admin all config    | ALL    | auth.role() = 'authenticated'
tbl_event, tbl_tournament, tbl_season, tbl_organizer, tbl_registration,
tbl_match_candidate, tbl_audit_log      — same shape

ADR-016 states the assumption in as many words: "All authenticated users have admin privileges (no role table needed — only admins ever sign in)." That assumption holds only while signup is closed. Nothing in the repository enforces or tests it; it is a dashboard toggle, and Supabase's hosted default is enabled.

Unverified on PROD — this is step zero I could not check the hosted projects: GET /auth/v1/settings on both refs returned 401 "No API key found in request", and I will not handle project keys. If CERT or PROD matches LOCAL, then anyone can register an account and immediately hold full CRUD on tbl_result, tbl_fencer and tbl_scoring_config — which is total control of the ranking. Check the dashboard under Authentication → Sign In / Providers → "Allow new users to sign up" before anything else in this plan.

S1-3 · TOTP is never enforced server-side

A grep for aal2, aal or auth.jwt() across every file in supabase/migrations/ returns nothing. MFA lives entirely in frontend/src/lib/admin-auth.svelte.ts — mfa.enroll, mfa.challenge, mfa.verify, driving an AuthStep state machine that gates the Svelte UI.

The consequence is that a session which completed only email + password (assurance level aal1) already carries auth.role() = 'authenticated', and therefore already satisfies every write policy and every authenticated function grant. The TOTP modal stops a person using the browser UI. It stops nothing using curl.

This is worth naming precisely: it is structurally the same defect as the one ADR-016 was written to eliminate. That ADR replaced a client-side password gate because "client-side gate provides no real protection". The replacement moved authentication server-side but left MFA client-side, so the same sentence is true again one layer up.

02Decisions — recommendation, pros and cons

Nine decisions shape this plan. Each carries my recommendation and the honest case against it. Where I cannot verify something from this session, I say so rather than assuming.

Decision 1 · sequencing One consolidated release, or a fast wave that closes the criticals first?
A · Hotfix wave first, then the migration, then the guards recommended
For
  • Two of the three critical items are configuration, not code — disabling signup is a dashboard toggle, and the Edge Function ships via supabase functions deploy, not release.yml. Neither needs a migration, a pgTAP cycle, or a documentation gate.
  • Closes the largest hole in hours instead of days.
  • Keeps the SQL migration small, reviewable and properly test-driven rather than rushed under pressure.
Against
  • Three changesets instead of one; the ADR and the coherence gate land after the fix rather than with it, so for a short period the handbook is behind reality.
  • Requires discipline to actually complete waves 2 and 3 once the alarm has stopped ringing.
B · One consolidated campaign
For
  • Single migration, single release, single ADR, one coherence gate — cleanest possible history.
  • Nothing can be forgotten halfway.
Against
  • PROD stays open for the whole duration, including for the two items that could have been closed with a toggle on day one. That is the decisive objection.
C · Criticals only, defer the rest
For
  • Minimum effort to remove the worst risk.
Against
  • Leaves the advisory unresolved, so the weekly e-mail keeps firing and the signal keeps being ignored — the exact dynamic that produced eleven unread warnings.
  • Leaves S3-1 in place, so the next new table repeats the whole story.
RecommendationA. The deciding factor is that the two biggest holes do not need a release train at all. Making them wait behind one buys nothing and costs exposure days.
Decision 2 · MFA depth Should TOTP become a real server-side control by requiring an aal2 JWT?
A · Require aal2 for all writes and for workflow dispatch recommended, with a precondition

Add (SELECT auth.jwt()->>'aal') = 'aal2' to the seven Admin all … policies and to the Edge Function's caller check.

For
  • Makes ADR-016's central promise true. Today "MFA-protected admin" and "anyone with a password" are the same privilege level.
  • Defends against the realistic attack — a leaked or reused admin password — which is exactly what TOTP was adopted to stop.
  • Costs nothing at runtime; it is a claim comparison.
Against
  • Lockout risk. An admin who never enrolled a TOTP factor can only ever hold aal1, and loses all write access the moment the migration lands, with no self-service recovery.
  • I cannot verify enrolment from this session — it requires the dashboard under Authentication → Users → MFA, or an authorised MCP connection.
  • Any future service that authenticates as a user rather than service_role would need MFA too.
B · aal2 on the Edge Function only
For
  • Protects the highest-consequence surface — the PAT-backed path that reaches PROD and sends outbound organizer mail.
  • Zero lockout risk for the admin UI; a stale-enrolment admin still edits data, just cannot dispatch.
  • No migration needed; ships in Wave 0.
Against
  • Direct PostgREST writes stay MFA-free, so a stolen password still rewrites the ranking.
  • Two different assurance standards in one system is a thing people forget.
C · Keep authenticated only, rely on closed signup
For
  • Least work, no lockout risk at all.
Against
  • The entire security of the ranking rests on one dashboard toggle that no test can see. That is precisely how the current situation arose.
  • TOTP remains a ceremony that any scripted caller skips.
RecommendationA, gated on B. Ship the Edge Function aal2 check in Wave 0 immediately — it has no lockout risk. Add the policy-level aal2 in Wave 1 only after confirming in the dashboard that every existing admin account has an enrolled TOTP factor. If any admin is unenrolled, enrol them first; do not ship the policy change until that is true.
Decision 3 · admin model Does authenticated keep meaning "admin", or do we add an explicit allowlist?
A · Disable signup; keep authenticated = admin recommended
For
  • Smallest possible diff — no policy rewrites, no new table, no ADR supersession.
  • Matches the decision already recorded in ADR-016; accounts stay dashboard-created.
  • Combined with Decision 2A, an attacker needs a password and a TOTP secret, so the single-toggle dependency stops being the only line of defence.
Against
  • The security of every table still depends on a dashboard setting that nothing in the repository can test, assert, or alert on. If someone re-enables signup in two years, nothing goes red.
B · Explicit tbl_admin allowlist checked by every policy
For
  • Removes the dashboard dependency entirely — a stray or re-enabled signup grants a JWT that authorises nothing.
  • Defence in depth, and testable in pgTAP.
  • Opens the door to real roles later (read-only auditor, per-weapon delegate).
Against
  • Touches all seventeen existing policies plus every authenticated function grant — a large, risky diff in the same change as a security fix.
  • Needs a new ADR partially superseding ADR-016, and a bootstrap story for the first row.
  • Real risk of locking out the admin UI during CERT rollout.
C · Allowlist for writes, authenticated for reads
For
  • Confines the rewrite to the seven Admin all … policies.
  • Most of B's benefit at roughly half the blast radius.
Against
  • Still a substantial diff riding along with an urgent fix.
  • Two authorization concepts coexisting invites mistakes in the next migration.
RecommendationA now, B as a follow-up ADR. Do not couple an authorization-model redesign to an incident response. But record B as a named, dated follow-up in the ADR — because A's weakness is real, and "we meant to" is how the first regression happened.
Decision 4 · credential exposure Rotate GH_DISPATCH_PAT?
A · Rotate it, immediately after the Edge Function fix deploys recommended
For
  • The PAT has been usable by anyone holding the anon key for as long as the function has been deployed. Treat it as exposed, because functionally it was.
  • Rotation is cheap: one fine-grained PAT, one supabase secrets set, no code change.
  • Rotating after the fix avoids a window where a new PAT sits behind the old open door.
Against
  • Brief dispatch outage during the swap; the admin UI buttons fail until the secret is set on both CERT and PROD.
  • The PAT was never transmitted to callers — only used on their behalf — so strictly it is unproven to have leaked.
B · Leave it; the value never left the server
For
  • No outage, no work.
Against
  • Confuses "the secret was not disclosed" with "the secret was not abused". Its capability was fully available to the public.
  • While auditing GitHub Actions history for unexpected dispatches, you would rather be looking at a token you have already retired.
RecommendationA. Also review the Actions run history for the seven allowlisted workflows for dispatches that no admin remembers making — that is the cheapest available tamper signal, and it needs no PROD database access.
Decision 5 · the alias view Fix vw_fencer_aliases by grants alone, or recreate it with security_invoker?
A · Grants only — REVOKE ALL FROM anon, reduce authenticated to SELECT recommended
For
  • Two statements, no behaviour change for the Phase-4 admin UI that reads it (ADR-050 / ADR-058).
  • Closes the anon read immediately and completely.
  • Also removes the nonsensical INSERT/UPDATE/DELETE/TRUNCATE grants that exist on a view today.
Against
  • The view still runs with owner rights, so it remains a standing RLS bypass for any role that can read it. If the admin model changes later, this is a place the change silently will not reach.
B · Recreate with security_invoker = true
For
  • Architecturally correct: the view stops being a privilege-escalation primitive and simply inherits whatever RLS sits beneath it.
  • Future-proof against Decision 3B.
Against
  • Couples the admin UI's behaviour to the draft tables' RLS, which this same migration is locking to service-role-only — so it needs an authenticated read policy added to all six tables that would otherwise be unnecessary.
  • Larger diff and a real chance of breaking the identity-manager screen on CERT.
RecommendationA, with the security_invoker conversion recorded as a follow-up alongside Decision 3B — they want to land together, because both are about making the authorization model uniform rather than patching its edges.
Decision 6 · tamper audit Audit PROD for evidence of abuse before locking, or after?
A · Lock first, audit immediately after recommended
For
  • The window has been open roughly eleven weeks; another day spent auditing before closing it is a poor trade.
  • Locking does not destroy evidence — it freezes it. Row counts and ts_created distributions survive intact.
  • An audit conducted while the door is still open can be invalidated by activity mid-audit.
Against
  • If an attacker is actively present, locking tips them off before you have characterised what they did.
  • tbl_*_ingest_history is itself falsifiable, so some evidence may already be unreliable — a point in favour of gathering it sooner.
B · Audit first, lock after
For
  • Captures the environment exactly as the attacker left it, including any live session.
Against
  • Extends a known-critical exposure for the convenience of forensics on a system with no attacker-independent audit trail.
RecommendationA. Widen the audit scope beyond the six tables, though: tbl_tournament and tbl_registration were writable through the two SECURITY DEFINER RPCs, and the GitHub Actions run history is a third, independent surface worth reading.
Decision 7 · the standing guard How strict should the new pgTAP security-posture test be?
A · Deny-by-default with an explicit allowlist recommended

Assert the anon-executable function set equals a named list, rather than merely excluding known-bad names.

For
  • This is the single change that makes the fix permanent. A deny-list only catches what you already knew about; an equality assertion catches the next mistake.
  • Fails loudly the day someone adds a table or an RPC without thinking about grants — which is exactly the event that produced every finding in this document.
  • Cheap: catalog queries, milliseconds.
Against
  • Every legitimate new public function requires an intentional edit to the allowlist. Some people find that friction annoying.
  • That friction is the entire point, so this is a cost to accept rather than mitigate.
B · Assert only the specific objects fixed here
For
  • No ongoing maintenance.
Against
  • Would not have caught any of today's findings had it existed in March, because every one of them involves an object that did not exist yet.
RecommendationA, and treat it as the most important deliverable in the plan. The migration fixes today; this prevents the next three years.
Decision 8 · CORS Lock the Edge Function's Access-Control-Allow-Origin to the deployed origin?
A · Restrict to the known frontend origins recommended
For
  • With *, any page an authenticated admin visits can invoke the function using their live session — a cross-site request forgery path onto workflow dispatch.
  • The set of legitimate origins is small and already known per environment.
Against
  • Needs the CERT and PROD origins to be correct, or the admin UI breaks — a config value that must move with any future domain change.
  • CORS is a browser control only; it does not stop curl. It is defence in depth behind Decision 2, never a substitute for it.
B · Keep *, rely on the new caller check alone
For
  • One less environment-specific value to get wrong.
Against
  • Leaves the CSRF-shaped path open against a logged-in admin, which the aal2 check narrows but does not fully close if the session is already aal2.
RecommendationA. Read the allowed origin from an environment secret rather than hard-coding it, so CERT and PROD differ without a code change.
Decision 9 · the extension lint Move btree_gist out of public, or accept it?
A · Attempt the move; accept formally if the dependent indexes block it recommended
For
  • Clears the last advisor lint, so a green advisor genuinely means "nothing outstanding" and the weekly e-mail regains signal value.
  • If it cannot move cleanly, a written, dated exception is still better than a permanent unexplained warning.
Against
  • ALTER EXTENSION … SET SCHEMA can fail or require dropping dependent objects; the real-world risk here is genuinely low but non-zero.
  • Lowest-value item in the plan — do not let it delay anything else.
B · Accept and document, do not touch
For
  • Zero risk to working indexes.
Against
  • A permanently non-empty advisor is an advisor people stop reading — the documented root cause of the eleven unread e-mails.
RecommendationA, scheduled last in Wave 2 so it can be abandoned without affecting anything else.

03The plan

Wave 0 — Stop the bleeding · no migration, no release train

  1. Verify signup on CERT and PROD (operator action; I have no dashboard or MCP access). Authentication → Sign In / Providers → "Allow new users to sign up". If this is on, it is the most urgent item in the document and everything else is secondary.
  2. Disable signup on both projects. Confirm existing admins still sign in.
  3. Confirm TOTP enrolment for every account under Authentication → Users → MFA. This is the precondition for Decision 2A and the gate on Wave 1's policy change.
  4. Fix dispatch-workflow — the one code change in this wave, deployed with supabase functions deploy dispatch-workflow:
    • Read the bearer token; call auth.getUser(token); reject when absent or invalid.
    • Reject when the resolved role is not authenticated — this is what closes the anon-key hole.
    • Require aal2 (Decision 2, no lockout risk at this layer).
    • Replace Access-Control-Allow-Origin: "*" with an origin read from an env secret (Decision 8).
    • Keep ALLOWED_WORKFLOWS exactly as it is — that part is sound.
    • Tests asserting: anon key → 401; valid aal1 session → 403; aal2 admin → 200; non-allowlisted workflow → 400.
  5. Rotate GH_DISPATCH_PAT on both projects (Decision 4), then read the Actions run history of the seven allowlisted workflows for dispatches nobody recognises.

Wave 1 — One SQL migration for the data plane

Single migration, following the house pattern and comment style of 20260327000001_revoke_write_functions.sql:

  • The six tables — ENABLE ROW LEVEL SECURITY, then REVOKE ALL … FROM anon, authenticated. No anon policy. Safe because nothing under frontend/src or supabase/functions references any of them — verified — and the Python consumers (draft_store.py, recompute/worker.py, review_cli.py, alias_writeback.py, db_connector.py, three tools) run as service_role, which bypasses RLS inherently.
  • vw_fencer_aliases — REVOKE ALL FROM anon; reduce authenticated to SELECT (Decision 5A).
  • The five RPCs — REVOKE EXECUTE … FROM anon, PUBLIC, re-granting to authenticated where the admin UI calls them. Leave the intentional public read-only RPCs alone: fn_ranking_ppw, fn_ranking_kadra, fn_season_summary, fn_export_scoring_config, fn_fencer_scores_rolling and their variants are documented as anon-accessible in 20260327000001 and the public ranking depends on them.
  • aal2 on the seven write policies — only if Wave 0 step 3 confirmed universal enrolment (Decision 2).
Order is not negotiable Strict RED→GREEN: the Wave 2 guard tests are written and observed failing before this migration exists. postgrestools (the plpgsql-check skill) runs on the migration before it counts as done. LOCAL verification is by attempting an anon write and being refused, never by reading the catalog — the catalog is what said everything was fine for eleven weeks.

Wave 2 — Make recurrence impossible

  • supabase/tests/52_security_posture.sql — the deliverable that matters most (Decision 7A):
    1. no table in public has relrowsecurity = false;
    2. the set of functions where has_function_privilege('anon', …, 'EXECUTE') holds equals a named allowlist;
    3. no view in public runs with owner rights over a table that has RLS enabled;
    4. anon holds no INSERT/UPDATE/DELETE/TRUNCATE on any public table.
    Bump the pgTAP total in the specification (currently 686).
  • Least-privilege permissions: blocks on the eleven workflows lacking them (S3-2), copying the pattern already used in release.yml.
  • Telegram alerting on security-posture test failure, reusing TelegramNotifier in python/pipeline/notifications.py — Telegram is the established surface, and S3-5 shows e-mail alone does not reach anyone.
  • ADR — "Server-enforced authorization: pipeline objects are service-role only; new objects are revoked by default; MFA is enforced in SQL." Amends rather than supersedes ADR-016. Registry entry in specification Appendix C. Record Decisions 3B and 5B as named, dated follow-ups so they are not quietly dropped.
  • New handbook page doc/handbook/reference/security-posture.html — the missing owner from S3-4 — registered in documentation-map.html with its change triggers, plus the standard coherence pass across affected subsystem pages and the RTM.
  • btree_gist relocation attempt, scheduled last (Decision 9).

Wave 3 — Tamper audit · operator-gated, runs in parallel

Needs PROD access I do not have. Compare row counts and ts_created distributions on the six tables, and on tbl_tournament and tbl_registration (reachable via the two SECURITY DEFINER RPCs), against what the pipeline's own ingest history claims it wrote. Read the GitHub Actions run history as an independent third surface — tbl_*_ingest_history was itself writable, so it cannot corroborate itself.
