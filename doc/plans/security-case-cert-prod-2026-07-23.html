A1 · Anyone can reach your PROD workflow dispatcher with the public key

This is the finding that matters most, and it is the one you challenged directly when you asked whether a button click needs authenticating. Here is why the button is not the point.

The button is client-side. The security boundary is the HTTPS endpoint the button calls — and an attacker calls that endpoint directly, with curl, never touching your interface. So the question is not "should a button click be authenticated". It is: should a public URL that promotes CERT to PROD and mails organizers require the caller to prove who they are?

Read what that second response means. A 400 invalid_workflow is not a rejection of the caller — it is the function saying "you are allowed in, that workflow name is just wrong." Had I sent promote-season.yml, it would have dispatched. And note the error message helpfully lists all seven valid names to an unauthenticated caller, so an attacker does not even need to guess. Identical result on CERT.

Why verify_jwt = true is not protection It asks one question: "is this a JWT signed by this project?" The anon key is such a JWT — that is its entire purpose, and it is published in your frontend bundle by design. The gate is satisfied by the very key that is public. Your handler in supabase/functions/dispatch-workflow/index.ts then never calls auth.getUser() and never inspects a role. There is no second gate.

02Part B — why the fix works

For each fix: the mechanism, why it closes this specific path, the alternatives I considered and rejected, and what could break.

B1 · Check the caller in the dispatcher

Why it closes the path: the anon key carries "role":"anon". Reading the bearer token and rejecting anything that is not a real authenticated user turns the proven 400 above into a 401. Because both your admins already have verified TOTP factors, additionally requiring aal2 costs nothing and means a stolen password alone cannot dispatch either.

Alternatives considered
  • Shorten the allowlist — drop promote-season.yml and ftl-seed.yml. Rejected: reduces the blast radius but leaves the endpoint open; the admin UI genuinely needs those buttons.
  • Move dispatch behind a GitHub OIDC / signed webhook instead of a PAT. Rejected for now: architecturally cleaner, considerably more work, and it does not fix today's hole faster than a caller check.
  • Set verify_jwt stricter in config. Rejected: no such setting exists — verify_jwt is boolean and already true. Role checking is only possible in code. This is precisely the misconception that created the bug.
  • Rely on CORS. Rejected: CORS is a browser courtesy; curl ignores it entirely. Worth tightening as depth, never as the control.

What could break: the admin UI calls this through supabase.functions.invoke(), which attaches the signed-in session automatically — so a logged-in admin is unaffected. Risk is if any automation calls it with the anon key; nothing in the repo does. Rotate GH_DISPATCH_PAT afterwards — its capability has been publicly reachable, so treat it as exposed even though its value never left the server.

B2 · Close signup; keep MFA meaningful

Why it closes the path: one setting flip on each project removes the "register → become admin" route entirely, and it is reversible in seconds. Because there is exactly one legitimate account per environment, nothing else is affected — verified, not assumed.

Alternatives considered
  • Explicit tbl_admin allowlist checked by every policy, replacing the bare authenticated test. Deferred, not rejected: strictly better — a stray signup would then grant nothing — but it rewrites all 17 policies and needs an ADR superseding part of ADR-016. Do not couple a model redesign to an incident. I would record it as a dated follow-up.
  • Leave signup open, rely on e-mail confirmation. Rejected: the barrier is "own a mailbox", and the prize is the ranking.
  • Raise password_min_length from 6. Worth doing, but not a fix — it hardens the one real account; it does nothing about registration.

B3 · RLS + REVOKE on the six, and grants on the view

Why it closes the path — and why it cannot break ingestion. This is the question you raised when you said the pipeline is triggered from many places. It does not matter how many places trigger it, because every one of them authenticates as service_role, and service_role bypasses RLS inherently. Verified across the workflows:

ingest.yml:28             SUPABASE_KEY: ${{ secrets.SUPABASE_CERT_SERVICE_ROLE_KEY }}
recompute-drain-prod.yml:61  SUPABASE_KEY: ${{ secrets.SUPABASE_PROD_SERVICE_ROLE_KEY }}
ftl-seed.yml:57 / :95     SUPABASE_KEY: ${{ …SERVICE_ROLE_KEY }}
phase5-event-runner.yml:58, ingest-event.yml:76, regen-report.yml:58, evf-parity-sweep.yml:45 …

So revoking anon is invisible to ingestion. The only readers that lose access are the ones that should never have had it.

The view needs separate treatment, because RLS underneath will not touch it: revoke anon outright and reduce authenticated to SELECT. It stays working for the Phase-4 admin UI that legitimately reads it.

Alternatives considered
  • Recreate the view with security_invoker = true. Architecturally the right answer. Deferred: it couples the admin UI to whatever RLS the draft tables end up with, so it would force an authenticated read policy onto all six tables that is otherwise unnecessary. Pair it with the tbl_admin work.
  • RLS only, skip the REVOKE (what the advisory implies). Rejected: leaves TRUNCATE nominally granted and does nothing about the view — the advisor would go green while fencer names stayed readable.
  • Add an anon read policy for the queue so dashboards could read it. Rejected: nothing needs it; no frontend file references any of the six.

B4 · The test that stops this recurring

Everything above fixes today. This is the only item that fixes tomorrow, and it is why I keep returning to it: this exact bug class was already fixed once, in 20260327000001_revoke_write_functions.sql (March 2026, ADR-016), whose own comment reads "all SECURITY DEFINER write functions were callable by the anon role because PostgreSQL defaults to GRANT EXECUTE TO PUBLIC". Every object created afterwards skipped the pattern, and nothing noticed for four months.

A pgTAP file asserting deny-by-default with an explicit allowlist — no RLS-disabled table in public; the anon-executable function set equals a named list; no owner-rights view over an RLS table; no anon write grants — goes red the day someone adds a table without thinking. A deny-list would not have caught any of today's findings, because every one of them involves an object that did not exist when the list would have been written.

Direct answer to a fair question Fix A closes every hole proved in Part A. 04Will it break the ranklist? The blast radius

Correction — I overstated this, and then checked further I first concluded that revoking authenticated would break admin fencer editing, because the trigger on tbl_fencer runs with the caller's privileges. That conclusion was wrong. On checking the actual call path, every fencer write from the admin UI goes through a SECURITY DEFINER RPC, so the trigger runs as the function owner and the queue INSERT succeeds regardless of authenticated grants.

This closes the question. Every fencer mutation the admin UI performs goes through a SECURITY DEFINER function, which executes as the function's owner. The trg_fencer_change_enqueue trigger therefore fires inside the definer's security context, and its INSERT into tbl_recompute_queue succeeds no matter what authenticated has been revoked. Note also anon EXECUTE = False on all of them — the March 2026 work (ADR-016 / 20260327000003) revoked these correctly. It is only objects created after March that drifted, which is exactly the pattern §05 explains.

Cheap hardening, no longer a blocker The remaining exposure is latent, not live: the moment anyone adds a direct .from('tbl_fencer').update(…) to the frontend, or any other authenticated-context write to tbl_fencer, the revoked queue grant would break it — and the failure would look like an unrelated permission error on a fencer edit.

Making fn_enqueue_affected_events SECURITY DEFINER removes that trap permanently, and it is the pattern already used by fn_audit_log (DEFINER on three triggers, writing to the RLS-protected tbl_audit_log). Recommended, one line, but not a prerequisite.

05Option B — fix the default privileges

What the problem actually is

Nobody granted the six tables anything. The schema did it automatically, on creation:

Proof · pg_default_acl on CERT
grantor   schema   type   acl
postgres  public   r      {postgres=arwdDxtm/postgres,
                           anon=arwdDxtm/postgres,          ← every new TABLE
                           authenticated=arwdDxtm/postgres,
                           service_role=arwdDxtm/postgres}
postgres  public   f      {…, anon=X/postgres, …}           ← every new FUNCTION

  a=INSERT  r=SELECT  w=UPDATE  d=DELETE  D=TRUNCATE  x=REFERENCES  t=TRIGGER

There is a standing ALTER DEFAULT PRIVILEGES on public handing full DML plus TRUNCATE to anon on every table created, and EXECUTE to anon on every function created. This is the Supabase default and it is the mechanism behind every finding in Part A: the six tables, the view's absurd INSERT/DELETE grants, and the five anon-callable RPCs. None of it was a decision.

The change

ALTER DEFAULT PRIVILEGES IN SCHEMA public
  REVOKE ALL ON TABLES    FROM anon, authenticated;
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  REVOKE EXECUTE ON FUNCTIONS FROM anon, authenticated;

-- thereafter, every public-facing object is granted explicitly:
GRANT SELECT ON vw_calendar TO anon, authenticated;   -- as migrations already do

Will it break the ranklist?

No — and this is a property of the statement, not an opinion ALTER DEFAULT PRIVILEGES affects only objects created after it runs. It does not touch a single existing table, view, function or grant. Every current frontend read keeps working because those objects already exist with their grants intact. The blast radius on existing functionality is precisely zero.

The cost lands later, and it is the intended cost: the next migration that creates a public-facing table must add an explicit GRANT SELECT … TO anon, or the frontend gets a permission error. That failure is loud, immediate, and caught in CI — the opposite of today's failure, which is silent and discovered by an advisory e-mail four months later.

Alternatives considered
  • Leave defaults, rely on the pgTAP guard. The guard catches the same mistakes, but after someone writes them, and only if the suite runs before deploy. Defence in depth says do both; if forced to choose one, the default-privilege fix prevents while the test detects. Verdict: do both, they are not substitutes.
  • Revoke from authenticated too, or only from anon? Revoking only anon is the softer option and still kills the internet-facing half. Revoking both is stricter and matches the fact that authenticated means "admin" here. Verdict: both, since RLS remains the real gate for admin access.
  • Do nothing and document it. Rejected: this is the generator of every finding above. Leaving it is choosing to have this conversation again.

Effort: two statements in the same migration as fix A. Risk: near zero today; a known, loud cost on future migrations. Leverage: the highest of any item in this document.

06Option C — separate the planes

The idea

In Supabase, public is not a neutral namespace — it is the internet-facing API surface. Everything in it is a REST endpoint by default. Option C says internal machinery should not live there at all:

A fresh schema carries no ALTER DEFAULT PRIVILEGES entry for anon, so tables created in it are inaccessible unless someone deliberately grants access. The secure state becomes the default state structurally, not by remembering to revoke.

Will it break the ranklist?

This is where the honest answer is "it could, and it needs real care." Unlike B, this moves live tables that CERT is actively ingesting into. Concretely:

H5 — the write proof, and why it matters most

Until now every probe had been a read. A read-only exposure is a disclosure problem. This one is an integrity problem, and it is the difference between "someone can see the queue" and "someone can change what the ranking engine does".

Proof · PROD · anon INSERT with a deliberately invalid payload
$ curl -X POST ".../rest/v1/tbl_recompute_queue" -H "apikey: <anon key>" -d '{}'

{"code":"23502",
 "message":"null value in column \"id_event\" of relation
            \"tbl_recompute_queue\" violates not-null constraint",
 "details":"Failing row contains (122, null, PENDING, 2026-07-23 …, null)."}

$ curl -X PATCH ".../rest/v1/tbl_recompute_queue?id_queue=eq.-999999" \
       -H "apikey: <anon key>" -d '{"enum_status":"DONE"}'
HTTP 204

Read the error code. 23502 is a NOT-NULL constraint violation — a data error. A permission failure would have been 42501 permission denied for table tbl_recompute_queue. Postgres accepted the write, built the row, assigned it id 122, and rejected it only because I deliberately omitted id_event. Supply that column and the insert succeeds.

The PATCH returning 204 No Content is the same story for UPDATE: success, zero rows matched, because id_queue = -999999 does not exist. Permission was never the obstacle.

So the watermark scenario in §A3 is not theoretical. A single authenticated-by-nobody PATCH against tbl_recompute_watermark would move the CDC timestamp and make the hourly PROD drain skip work silently.

Disclosure — one non-zero side effect I have been saying "nothing was written". That remains true of table data: both probes failed or matched nothing, and no row exists as a result. But the INSERT attempt did call nextval() before hitting the constraint, and sequence advancement is not transactional — so the tbl_recompute_queue id sequence advanced by one on CERT (to 297) and one on PROD (to 122). The practical effect is a one-number gap in future ids, which is normal Postgres behaviour and affects nothing. I am recording it because I claimed read-only and this is the one place that claim needed an asterisk.

11How to stop this happening again

The finding that should worry you is not the six tables. It is that nobody made a mistake — the schema handed out the grants automatically, and no process anywhere would have noticed. Prevention has to attack that, in three layers: remove the cause, detect if it returns, and make sure a human hears about it.

Layer 1 — Remove the cause

Postgres applies ALTER DEFAULT PRIVILEGES per creating role, so the fix only works if it targets the role that actually creates your objects. Verified:

SELECT pg_get_userbyid(relowner), count(*) FROM pg_class … WHERE nspname='public'
  →  owner=postgres   count=23        ← every table and view, no exceptions

Migrations run as postgres, and all 23 public objects are owned by it. So revoking the postgres defaults covers everything the migration path can ever create:

ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA public
  REVOKE ALL     ON TABLES    FROM anon, authenticated;
ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA public
  REVOKE EXECUTE ON FUNCTIONS FROM anon, authenticated;
ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA public
  REVOKE ALL     ON SEQUENCES FROM anon, authenticated;

All three object types are needed — check C5 found anon on r (tables), f (functions) and S (sequences). After this, a new table is unreachable until someone writes an explicit GRANT SELECT … TO anon — a visible line in a migration diff that a reviewer can actually see and question.

Layer 2 — Detect it if it comes back

Verified: this gate is real, not decorative I checked whether a test would actually block anything before recommending one. ci.yml:24 runs supabase test db in a job named "pgTAP (database)", and release.yml is triggered by workflow_run: workflows: [CI], gated on github.event.workflow_run.conclusion == 'success'. A failing pgTAP test therefore stops the release. The guard has teeth.

Layer 1 can still be undone — by an explicit GRANT in a future migration, by a platform change re-adding defaults, or by restoring a dump. So the assertion must be deny-by-default equality, not a list of known-bad names. A deny-list would not have caught a single finding in this document, because every offending object was created after such a list would have been written.

-- supabase/tests/52_security_posture.sql
1. every table in public has relrowsecurity = true
2. { functions where has_function_privilege('anon', …) }  EQUALS  a named allowlist
3. no view in public is owner-rights over an RLS-enabled table
4. anon holds no INSERT / UPDATE / DELETE / TRUNCATE on any public table
5. pg_default_acl for role postgres in public grants nothing to anon/authenticated

Assertion 5 is the one that guards Layer 1 itself. Without it, someone could silently restore the default privileges and every other assertion would keep passing until the next table appeared.

Gap found while verifying the gate release.yml's job condition is:
if: github.event_name == 'workflow_dispatch' || github.event.workflow_run.conclusion == 'success'

The first clause is unconditional, so a manually dispatched release skips the CI gate entirely — pgTAP included. That is convenient during an incident, and it is also exactly how a guard gets bypassed on the day it matters most. Either require a typed confirmation input for manual dispatch, or re-run the posture test inside the deploy job so it cannot be routed around.

Layer 3 — Make sure a human hears it

The advisory e-mail fired weekly from 6 May, and every message but the last was unread. The detection already existed; the delivery failed. Route security-posture failures to Telegram through the existing TelegramNotifier in python/pipeline/notifications.py — the channel this project actually reacts to — rather than adding another mailbox rule nobody reads.
