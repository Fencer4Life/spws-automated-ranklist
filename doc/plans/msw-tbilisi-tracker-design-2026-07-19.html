Live response tracker for the board report

Turn raport.html into a dashboard that refreshes every five minutes, started and stopped on request — and that can never quietly pretend nobody is replying when it has actually stopped working.

2026-07-19 branch: docs/msw-tbilisi-report-mockup option B · localhost, in-place swap local only · 127.0.0.1 not yet built

01Context

Once wave 1 goes out, about 103 fencers will reply over roughly two weeks. Today raport.html is a static file: seeing new replies means running two scripts by hand and reloading the tab.

The data half already works and does not need redesigning. read_responses.py pulls declarations from all seven form variants; gen_raport.py renders the report from those plus send_log.jsonl. What is missing is a loop, a way for the page to update itself, and — the part that is easy to skip — a way for the page to say how old it is.

The failure this design exists to prevent A tracker that silently stops does not look broken. It looks like nobody is replying. If the loop dies at 09:10 and you glance at the tab at 14:00, the honest reading and the broken reading are the same picture. Every structural choice below follows from refusing to let those two states render alike.

Decisions already closed

DecisionDetail
audienceYou, this Mac, only. Not shared with the board. The report holds 144 fencers' names, addresses and ranking positions; anything the board could open means hosting, authentication, and a deployment surface this one-off campaign was scoped not to have. A snapshot sent at a moment you choose is the honest substitute for a live link.
mechanismOption B — the watcher also serves the page on a localhost port, and content is swapped in place. No reload flash, no lost scroll, and no file:// caching inconsistencies.
granularityWhole-page swap, not per-cell patching. Per-cell updates would need a stable id on every fencer-weapon cell plus a second JSON contract to keep in step with the renderer — the same duplication that was just removed from the KPI counters.
alertingNone. The strip is the whole alerting surface. A stalled tracker loses visibility, not data: responses accumulate in Google regardless and the next good cycle catches up completely.
token expiryShow its age instead of alerting. The 7-day expiry is predictable, so it should be visible days out rather than discovered afterwards.
separationOwn process, own port, own venv. No contact with the ranklist LOCAL stack — the campaign's "no schema, no pipeline, no integration" rule holds.

02Two findings in the code that drive the design

1 · gauth.creds() exits the process on every failure path gauth.py lines 47, 49, 52, 59 and 63 all call sys.exit() — missing token, missing scope, refresh failure. That is exactly right for one-shot CLI scripts and fatal for a long-running loop. Worse, SystemExit derives from BaseException, so a except Exception around it would not catch it.
2 · gen_raport.py has no main() It is a top-to-bottom module script — the body runs once at import and never again. import gen_raport inside a loop cannot work, and importlib.reload() on a module built entirely from module-level globals is a trap rather than a fix.
One decision solves both Run each refresh cycle as a subprocess. A sys.exit becomes a non-zero exit code the parent reads; the module-level script runs cleanly every time; and a crash in one cycle cannot corrupt the watcher. Critically, no existing script's control flow is modified — which matters because gauth.py sits on the send path and wave 1 has not gone out yet.

03Architecture

tracker.py serve
├── refresh thread — every 5 min
│     subprocess: read_responses.py      → responses.json
│     subprocess: gen_raport.py --live   → raport.html   (atomic replace)
│     success → bump generation, stamp last_ok
│     failure → capture stderr verbatim into state
│
├── state.json — rewritten every cycle, success or failure
│
└── HTTP server, 127.0.0.1:8765
      GET /            → raport.html
      GET /state.json  → liveness, generation, token age, last error

The page polls /state.json every 30 seconds — local, free, no Google calls — and swaps in fresh report content only when generation increases.

Why two channels rather than one If the only signal were "did the HTML change", a failed cycle would be indistinguishable from a quiet one — precisely the failure this design exists to prevent. state.json is rewritten on every cycle including failures, so the strip stays truthful even while the content is frozen. And if the tracker process dies outright, the fetch itself fails and the page reports offline — the one case no server-side flag can ever cover, because nothing would be left running to set it.

04Components

scripts/tracker.py — new

CommandBehaviour
startRefuses if a live PID is in tracker.pid — two watchers writing the same file would interleave. Otherwise spawns serve detached and prints the URL.
stopSIGTERM, wait, clear the PID file. Succeeds quietly if it was not running, so it is safe to call twice.
statusRunning or not · last successful cycle · cycle count · last error · token age.
serveThe foreground process the other three manage.
Bound to 127.0.0.1 only The report carries 144 fencers' names, e-mail addresses and ranking positions. It must not be reachable from the LAN. Binding to 0.0.0.0 would publish personal data to every device on the network. No-cache headers on both routes.

scripts/gen_raport.py — modified

  1. Atomic write. Line 574 writes straight to raport.html. Write raport.html.tmp then os.replace() — the server serves this file while the renderer may be rewriting it, and a 90 KB page is not written instantaneously.
  2. A --live flag. Only when passed, emit the freshness strip and its script, as siblings outside <div class="page">. Without the flag the output stays byte-for-byte what it is today, so the static report remains clean for sending to the board.

The swap targets .page only, so the strip and its JS survive every update with their own state intact.

The freshness strip

Fixed top-right pill, always visible while scrolling — a dashboard you glance at must not require scrolling to the top to learn whether it is alive.

StateTriggerShows
greenlast cycle succeeded, under 2 cycles ago time of last refresh · token age
amberno successful cycle in over 2 cycles how long it has been stale
redlast cycle failed the subprocess's own stderr, verbatim
grey/state.json fetch failed tracker offline

gauth's messages are already actionable — "Refresh failed (…) — re-run auth_setup.py." — so the red state surfaces them rather than inventing its own wording on top.

Token age

The strip shows the refresh token's age and turns amber from day 6. The point is to see the expiry coming, not to discover it afterwards.

token.json's mtime is useless here — gauth rewrites the file on every hourly access-token refresh — and its expiry field describes only the one-hour access token, not the 7-day refresh token. Instead tracker.py keeps a sidecar token_granted.json holding a hash of the refresh token plus when it was first seen. A changed hash means a new consent happened, so the clock resets. First run seeds from token.json's st_birthtime.

Credential handling This is entirely contained in tracker.py; auth_setup.py is not touched. The token itself is never logged, printed, or written anywhere — only its hash.

05Verification

The campaign scripts are throwaway and carry no test suite, so this is verified by driving it rather than by unit tests. Each step states its expected result; if one does not match, stop rather than continuing.

1 · it starts tracker.py start, open http://127.0.0.1:8765/.
Report renders, strip green, token age shown.
2 · it refuses to double-start tracker.py start again.
Refuses; no second watcher spawned; the original keeps running.
3 · it is not on the network Load the page on 127.0.0.1, then try the Mac's LAN address.
Works on loopback, refused on the LAN address.
4 · new responses appear Submit a test form response, wait one cycle.
Cells turn green with no page flash and no loss of scroll position.
5 · death is visible Kill the tracker process outright.
Strip goes grey, offline, within 30 seconds. This is the test that matters most — it is the failure mode the whole design targets.
6 · auth failure is legible Temporarily move token.json aside.
Strip goes red carrying gauth's own message. Restore it; recovery on the next cycle without a restart.
7 · it stops cleanly tracker.py stop, then status.
Reports not running; port released; tracker.pid gone.
8 · the static report is unchanged Run gen_raport.py with no flag.
Output identical to today's static report — no strip, no script.

Then Ruff, and delete any test submissions from every form before the real wave.

06Out of scope

  • No launchd or cron. Start and stop stay manual, on request.
  • No notifications. The strip is the whole alerting surface.
  • No change to send policy, form content, or send_wave1.py.
  • The tracker is read-only against Google. It must never be able to reach "Delete all responses" — that stays a manual, pre-send-only action, and no automated process should be in a position to perform it.
