01The four corrections to the plan

All four were found by live catalog introspection on LOCAL and confirmed against the frontend and Python sources. None were guesses; each is stated with the evidence that settles it.

Correction 1 · the plan's Block 4 would have broken the public registration form

The plan instructs REVOKE EXECUTE ON FUNCTION fn_create_registration(...) FROM anon, PUBLIC, annotating the compensating grant with "api.ts calls it". That reads the call site backwards. api.ts does call it — from frontend/register.html, which is served to anonymous visitors, not from the admin app.

Three independent sources agree it must stay anon-callable:

  • api.ts:407 comment: "The sole public write path (FR-122). anon has no INSERT policy on tbl_registration; this SECURITY DEFINER RPC upserts…"
  • Existing pgTAP 49.14 asserts fn_create_registration succeeds as anon, paired with 49.13 asserting direct anon INSERT is denied. The revoke would have turned a passing test red.
  • ADR-079 / FR-122 designed it this way deliberately: SECURITY DEFINER exists precisely so anon can register without holding INSERT on tbl_registration.

Action taken: fn_create_registration and its lookup partner fn_match_registration_fencer are retained in the 52.7 allowlist. Proven working after the migration — a live anon call returned registration id 1 (§02).

Correction 2 · the write-RPC surface is thirteen functions, not five

Finding F5 named five anon-executable write RPCs. Introspection of every non-extension, directly-callable function found fifteen volatile anon-executable functions; two are the legitimate registration path above, leaving thirteen to revoke. The nine the plan did not name include:

  • fn_merge_fencers — destructive. ADR-071 dedup: deletes a fencer row and re-points its results and aliases onto a survivor.
  • fn_refresh_active_season — SECURITY DEFINER, UPDATEs tbl_season.bool_active, i.e. moves the whole system's active season.
  • fn_update_fencer_aliases, fn_set_event_source_override, fn_dry_run_event_draft, fn_backfill_id_prior_event, fn_backfill_joint_pool_split, _fn_create_skeleton_children, _resolve_event_prefix.

Why this could not be skipped: the plan's Step 3 says to derive the live list, remove the five, and pin the remainder as the allowlist. Done literally, that writes fn_merge_fencers into the test as approved anon-callable surface — baking the vulnerability in as intended behaviour. All thirteen have zero frontend callers (Python/service_role only), and authenticated keeps its own explicit proacl entry, so no re-GRANT was needed anywhere.

Correction 3 · anon held DML on all sixteen tables, not the six

The plan's Block 2 revokes on the six RLS-disabled tables. Its own assertion 52.3, however, reads "anon holds no INSERT on any public table" — and the RED run showed anon holding INSERT, UPDATE, DELETE and TRUNCATE on all sixteen, including tbl_result, tbl_fencer and tbl_scoring_config.

On the ten RLS-enabled tables those writes were already refused by policy — but RLS was the only thing refusing them. TRUNCATE is the sharp edge: it is not filtered by RLS at all, so that grant defeated row level security outright on all ten.

Action taken: a new Block 3 revokes INSERT, UPDATE, DELETE, TRUNCATE, REFERENCES, TRIGGER from anon on the other ten while retaining SELECT — the public ranklist reads six of them directly. authenticated is untouched (api.ts:279, :296, :528 are direct authenticated writes).

Correction 4 · two smaller items

A second default-ACL grantor exists. pg_default_acl carries an identical set of anon defaults on schema public for supabase_admin as well as postgres. The plan's FOR ROLE postgres is still correct and sufficient — migrations run as postgres and all public objects are owner=postgres — and on hosted Supabase the postgres role cannot alter another role's defaults anyway. It is left alone deliberately rather than attempted-and-failed at deploy time, and recorded as a residual in ADR-083.

The migration needed a lock_timeout. postgrestools flagged lint/safety/lockTimeoutWarning, which the repo's own 20250301000002_rls_policies.sql does not trip because it sets one. Added SET LOCAL lock_timeout = '2s', matching that convention: every statement from Block 2 onward takes ACCESS EXCLUSIVE, and failing fast beats queueing behind a live recompute drain and blocking every reader that arrives afterwards.

Correction 5 · Step 7 does not ship outside the release pipeline

The plan says the Edge Function "ships outside release.yml" via a manual supabase functions deploy. It does not. release.yml already carries Deploy Edge Functions to CERT and to PROD steps that deploy dispatch-workflow on every release.

This is better than the manual path — the deploy is ordered after the migration, auditable, and cannot drift between environments. The dispatcher therefore ships in Step 13 along with everything else, and the plan's separate manual-deploy instruction is redundant.

Note: a manual supabase functions deploy was attempted and refused by this environment's permission classifier. It turned out not to be needed.

Independent confirmation of Corrections 2 and 3, against the live environments scripts/check-security-posture.sh — written for Step 8, and pointed at CERT and PROD before any of this shipped — reports identically on both: 6 tables RLS-off, 16 tables where anon holds write (the plan said six), 1 anon-readable alias view, 24 default-ACL grants, and exactly the 13 functions named in Correction 2 (the plan said five). That is not my analysis agreeing with itself — it is a separately-written tool, querying the real databases, arriving at the same list.

Test 52.7 asserts set equality, not absence of known-bad names — a deny-list would have caught none of these findings, since every offending object postdates any list that could have been written. Nineteen names survive:

Scoped to directly-callable, non-extension functions. Trigger-returning functions are excluded because Postgres refuses to call them outside a trigger context, so a grant on one is not reachable surface; btree_gist ships roughly 200 functions into public whose grants belong to the extension, not to this project.

The self-guard is the part that stops this recurring 52.10 and 52.11 assert that role postgres grants no default privileges to anon or authenticated in schema public. Without them, someone restores the defaults and every other assertion in the file keeps passing until the next table happens to be created — which is exactly how four months of drift went unnoticed.

07PAT rotation — skipped, with reasoning

The plan and handover both treated rotating GH_DISPATCH_PAT as required. On inspection that is weaker than it sounds, and the operator's decision to skip it is sound.

What was exposed was the PAT's capability, not its value. The token appears exactly once in the function — the Authorization header of a server-side fetch to api.github.com. It is never in a response body: the success path returns only workflow/inputs/runs_url, and gh_dispatch_failed returns GitHub's response text. An attacker could make the PAT act; there was no path by which they could read it. Rotation defeats someone who holds a secret, and nobody could have obtained this one this way.

The question that actually mattered was whether the open door was used — and that has been answered. All 48 workflow_dispatch runs across all seven allowlisted workflows were triggered by Fencer4Life. Zero unrecognised dispatches, including both promote-season.yml runs (14 July, both failed). No evidence of exploitation.

Rotation becomes worth doing if the PAT is ever pasted elsewhere, logged or shared, or on ordinary short-expiry hygiene — not because of this finding.
