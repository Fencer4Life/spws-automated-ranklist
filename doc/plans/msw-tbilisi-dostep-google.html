00The limit I found — read first

Apps Script cannot carry this campaign A @gmail.com account is a consumer account, not Google Workspace. The MailApp/GmailApp quota for a consumer account is 100 recipients per day — not the 1,500 that Workspace gets. Gmail proper (the web UI, SMTP, the Gmail API) allows 500 per day.

Wave 1 is roughly 103 mails, and the contact-sourcing requests are another 34 — about 137 total. Through Apps Script that does not fit in a single day. Through the Gmail API it fits with room to spare.

Caveat: 103 and 34 come from the previous session and I have not verified them — see question 4.

Sources: Apps Script — Quotas (consumer 100/day, Workspace 1,500/day) and Gmail sending limits (500/day). Both checked today against Google's documentation rather than quoted from memory — quoting from memory was mistake #1 last session.

01What I will not do, whatever the instruction

Passwords and credentials Do not give me the password to this account and do not paste it into chat. I don't enter passwords, 2FA codes or keys into any field, and I don't create accounts. If a password lands in this conversation it will have to be rotated.

Every option below is built so that you authorise, in Google's own consent screen, and I get access indirectly — either through code running on your Mac or through your already-logged-in browser. I never see the credentials.

Why A rather than B

  • The quota. 500 a day instead of 100. Wave 1 goes out in one day.
  • The data is on your Mac. The 270-contact list lives in doc/external_files/, outside git, deliberately. Apps Script would mean uploading it to Google's cloud. Option A leaves the data where it already is.
  • The database is local too. Handover section 03 fixes a separate msw_tbilisi database on port 54322. Python-side campaign logic sits next to that database; Apps Script would be cut off from it.
  • The code stays in the repo and under version control. Apps Script lives in someone else's editor.
Option D is rejected, but not because it's bad An app password works and gives the same 500/day. It loses because A gives the same thing without creating a durable secret that could leak into the repo — and mistake #5 last session was nearly committing personal data. Fewer secrets, less surface.

03What option A actually requires from you

One-off, about 10 minutes. You click all of it; I can walk you through it step by step, or — if you enable Claude in Chrome — point at where to click. I don't create accounts and I don't accept terms on your behalf.

  1. Signed in as spws.weterani@gmail.com, create a new project in the Google Cloud Console.
  2. Enable two APIs: Gmail API and Google Sheets API.
  3. OAuth consent screen: type External, status Testing, add spws.weterani@gmail.com as a test user. Publishing the app and Google verification are not needed — the account authorises itself.
  4. Create credentials of type OAuth client ID → Desktop app and download the JSON.
  5. Put that file in doc/external_files/MSW-Tbilisi-2026/ — the directory is gitignored, so it cannot reach the repo. Do not paste its contents into chat. The script reads it from the path; I never need to see it.
  6. Run my authorisation script once — it opens a browser, you click Allow, the token is written locally.

04Decision sheet

QUESTION 1 · access route Option A (Gmail API locally), B (Apps Script), C (Claude in Chrome) or D (app password)? These combine: A for sending and reading responses, C for one-off setup and building the form.
Recommendation: A as the backbone, C for setup. Why: 500/day instead of 100, the data stays on your Mac, the code stays in the repo.
QUESTION 2 · building the form I can build the form via Apps Script (a script that creates it programmatically and prints each field's entry.* prefill ID), or by clicking through it with you in Chrome.
Recommendation: Apps Script, once, for the build only. Why: the 100-mail quota is irrelevant to form creation, and the script prints the prefill IDs that are tedious and error-prone to harvest by hand. Apps Script does not come back for sending.
QUESTION 3 · scope of permissions Request gmail.send only (send, no read access to the mailbox), or also gmail.readonly so bounces and replies can be detected automatically?
Recommendation: start with gmail.send + spreadsheets.readonly. Why: minimum scope, and bounces can be eyeballed manually at this volume. Widen it later if it proves necessary.
QUESTION 4 · the numbers The handover says you overwrote the workbooks with end-of-season data on 2026-07-18 — the same day the figures 144 / 173 / 41 / 8 / 34 were produced. I can't tell whether they predate or postdate the overwrite. Should I re-run the generators against doc/external_files/Sezon_2025-2026/XLS/ to confirm?
Recommendation: yes, before anything is sent. Why: it's the only way to know how many people are actually in the campaign — and whether 500/day is enough. I need your explicit go-ahead to read those specific files.
