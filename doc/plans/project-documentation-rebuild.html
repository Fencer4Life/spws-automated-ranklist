Documentation design

Seven principles for the replacement

1 · Present tense firstCanonical pages explain what the system does now. Evolution belongs in ADRs and the archive.
2 · Top down, then sidewaysEvery page starts with purpose and boundaries before code, schema or operational detail.
3 · One fact, one ownerOther pages link to the owning document rather than copying mutable facts.
4 · Progressive disclosureA newcomer can stop after the domain tour; maintainers can continue into subsystem contracts and reference.
5 · History is reachable, not intrusiveArchived material remains searchable and linked, but never participates in the default read path.
6 · Source-linked truthArchitecture claims name the current source files, database objects, workflows and tests that prove them.
7 · Validation is part of deliveryBroken links, missing metadata, stale source maps and unindexed canonical pages fail a documentation check.
HTML as the human surfaceEvery new human-facing document is HTML, responsive and printable, with stable anchors and embedded machine metadata.

One shape for every current page

Canonical document contract

Every page under doc/handbook/ should carry machine-readable metadata in HTML and follow a predictable human structure.

Required metadata
doc-id · title · status=current · doc-type · owner
last-verified · source-globs · related-docs · review-triggers

Required human sections
1. Purpose — why this subsystem exists
2. Scope and boundaries — what it owns and explicitly does not own
3. Mental model — the smallest useful explanation
4. Current flow — present-tense sequence with failure path
5. Invariants — rules that implementation must preserve
6. Implementation map — source files, database objects, workflows, UI surfaces
7. Operations — how to observe, validate and recover it
8. Tests and evidence — checks that prove the description
9. Related decisions — ADR links, without copying ADR narratives
10. Change triggers — exactly which code/config changes require review

History is prohibited inside these sections except for a short link such as “Decision rationale: ADR-073.” Superseded behavior belongs only in the ADR or archive.

The rule added to every development plan

Documentation coherence gate

Documentation coherence gate complete

This single box may be checked only when every binary condition below is satisfied.

  1. Impact classified: the plan says whether domain behavior, product behavior, architecture, operations, configuration or reference contracts change.
  2. Owners named: every affected canonical HTML document is listed by path—or “no documentation impact” is justified with evidence.
  3. Current truth updated: affected handbook pages describe the post-change system in present tense; no implementation diary is appended.
  4. Maps synchronized: architecture flows, source paths, database objects, workflow catalog and last-verified metadata are updated where relevant.
  5. Decision trail updated: a new/superseding ADR is linked when the change introduces a tradeoff, constraint, deferral or pattern.
  6. Operations synchronized: commands, secrets, environment behavior, release steps, observability and recovery are updated when affected.
  7. Validation passed: the documentation checker passes links, metadata, index coverage, source references and changed-area ownership.
  8. Archive discipline applied: superseded prose is moved to archive/evidence, not left beside the new current description.

Machine-development practical

Checks that make the structure sustainable

  • Canonical index coverage: every current handbook page appears exactly once in the documentation map.
  • Metadata contract: required meta fields exist, status is valid, IDs are unique and dates parse.
  • Link integrity: relative pages, anchors, source files, database references and workflow links resolve.
  • Required-section contract: subsystem pages contain boundaries, invariants, implementation map, operations, tests and change triggers.
  • Ownership map: source globs embedded in the documentation map identify which pages must be assessed when code changes.
  • Changed-area report: CI reports watched source areas whose owning documents did not change; enforcement can begin as warning and graduate to hard fail after tuning.
  • Archive isolation: archived/evidence pages cannot be linked as the canonical target from the handbook index.
  • Freshness report, not fake freshness: last-verified is updated only after inspecting the named source paths, never automatically on render.

The machine-readable map should be embedded as <script type="application/json"> inside the HTML documentation map. Humans see one document; tooling reads the same ownership data without creating a second documentation format.

Approved decisions

Decisions locked for implementation

1 · Canonical location

Recommendation: use doc/handbook/ for current-system documentation. It communicates durable guidance better than current/ and leaves doc/ available for decisions, governance, work, evidence and archive.

2 · Language

Recommendation: author the developer handbook in English, retain Polish domain terms in the glossary, and add Polish summaries only where operators—not developers—are the primary audience.

3 · Archive depth

Recommendation: physically move clear history/plans and reclassify staging/audits as evidence, but keep source Markdown beside generated HTML until the new handbook has passed parity and link checks.

4 · Documentation-change enforcement

Recommendation: launch source-ownership checks as CI warnings for two development cycles, measure false positives, then promote reliable mappings to hard failures.

5 · First vertical slice

Recommendation: build the newcomer path plus one complete subsystem—ingestion—as the template. It exercises domain terminology, architecture, source mapping, workflows, database contracts, operations, tests and the legacy bridge.

Definition of success

Acceptance criteria for the documentation rebuild

  • A developer unfamiliar with fencing can explain the business entities and ranking purpose after the orientation path.
  • A developer can trace each canonical user/system flow from actor to UI, Python/workflow, Supabase objects and observable result.
  • No current-system page requires reading development history or a milestone plan to understand present behavior.
  • Every canonical fact has one owning page, and duplicate pages link rather than copy.
  • Every subsystem page names boundaries, invariants, source paths, operations, tests, related ADRs and review triggers.
  • Historical designs and evidence remain reachable but are absent from the default newcomer path.
  • The plan checkbox is present in the standard development-plan template and documentation CI passes.
  • README becomes a short gateway and contains no manually maintained volatile counts.
