04Traps — read before writing any code

Trap 1 · the wrong conclusion the last session reached

It concluded that REVOKE … FROM authenticated would break admin fencer editing, because trg_fencer_change_enqueue on tbl_fencer is SECURITY INVOKER and inserts into tbl_recompute_queue.

That conclusion is WRONG. All eight admin fencer-mutation RPCs (fn_update_fencer_birth_year and friends) are SECURITY DEFINER, so the trigger fires in the definer's context and the queue INSERT succeeds regardless of grants. The frontend's only direct tbl_fencer access is a SELECT (api.ts:573). The REVOKE is safe. Making fn_enqueue_affected_events DEFINER is cheap hardening against a future direct write, not a prerequisite.

Trap 2 · REVOKE FROM anon is not enough

The grant on those functions comes from the PUBLIC default — the bare =X/postgres entry in proacl. You must REVOKE EXECUTE … FROM anon, PUBLIC. Revoking from anon alone leaves the function callable.

Trap 3 · ALTER DEFAULT PRIVILEGES needs the role qualifier

It applies per creating role. Without FOR ROLE postgres it targets the executing role's defaults and silently achieves nothing. All 23 public tables/views are owner=postgres, so postgres is the correct target. And you need all three object types — TABLES, FUNCTIONS and SEQUENCES; anon was granted on all three.

Trap 4 · LOCAL lies about this problem

On LOCAL the staging tables are empty, which makes the whole finding look trivial. CERT is the staging environment and it is populated — 256 draft results, 50 draft tournaments, 125 queued recompute jobs. PROD's tbl_recompute_queue holds 46 rows and the watermark is a single live row. Never reason about severity from LOCAL.

Trap 5 · graphify does not cover this area

graphify affected python/pipeline/draft_store.py returns "No affected nodes found"; recompute/worker.py has no node match. Do not waste turns there, and do not claim the graph corroborated anything. Live catalog introspection (triggers, FKs, proacl, pg_default_acl) is the tool that actually works here, and it is what every finding above rests on.

Two more, smaller · Every SECURITY DEFINER function must carry an explicit SET search_path, or it becomes a privilege-escalation vector. postgrestools flags this; do not suppress it.
· Several of the five RPCs are overloaded — get exact signatures from pg_get_function_identity_arguments before writing the REVOKE, or it will not match.
