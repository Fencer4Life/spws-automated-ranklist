04What I got wrong earlier

I described this file as carrying "the same latent hazard" as 03_views_api and deferred it to a background chip on the grounds that it passes. Two errors in that:

  • I reasoned about the surname lookup because that was the pattern fresh in my mind from the file I had just fixed, and stopped there. The surname issue is real but latent and the least important thing in this file. The season pin is the actual defect, and I would have found it by reading line 37 — six lines above the lookups I did read.
  • I treated "passes" as "safe". For a test whose entire job is to guard the scoring formulas, passing against a retired configuration is the specific way it fails at its job. Green was the thing that should have prompted the check, not ended it.

05Recommended order

  1. Decide what the scoring test should pin to. This is a genuine product question, not a mechanical fix — see the single decision below.
  2. Add coverage for the active season's rules: MSW ×1.2 and the pooled PEW+MEW+MSW best-3 bucket. Today nothing asserts either.
  3. Make the fixture own its fencers, per the convention now in test-and-traceability.html. Mechanical once step 1 is settled.
  4. Replace 2.13 with a real round trip: export → import into a scratch season → re-export → compare.
  5. Raise the graph's SQL blindness separately. Either teach graphify to parse SQL, or amend pre-analysis-check to state plainly that the graph does not cover supabase/**/*.sql and that DB introspection is the authority there. Right now the skill implies coverage it does not have.
Resolved — decision taken 2026-07-19: follow the active season The fixture now resolves bool_active instead of naming SPWS-2024-2025, so these tests always guard the rules actually in force. Repointing it produced exactly one failure — test 9.85, the MSW multiplier — which is the empirical proof of the finding above: that assertion had been guarding a retired 2.0 against a live 1.2.

How the drift risk was handled

Following the active season reintroduces the hazard just fixed in 03_views_api: a config-following test can silently change meaning when an admin edits scoring config. Two measures prevent that, and both were verified rather than assumed:

  • New test 2.0 — a config contract. It pins the exact active-season values this file's hand-derived arithmetic depends on (MP 50, podium 3/2/1, PPW 1.0, MPW 1.2, PSW 2.0, MSW 1.2, 5 rounds). A rule change fails here first, by name.
  • Multiplier assertions read the configured value rather than a literal, so they test that the engine honours config — which cannot drift. The literals live in 2.0, in one place.

Verified by simulation: setting the active season's num_msw_multiplier to 1.75 produced one failure — test 2.0, naming the cause — while test 9.85 still passed, correctly observing that the engine had applied 1.75. That is the intended behavior: engine correctness stays drift-proof, and the rule change surfaces loudly instead of silently. The multiplier was restored and the suite re-verified.
