How to create a fully-enabled admin user — and why it takes four accounts to add two people.

Written to be followed top to bottom without prior knowledge. Every place you can go wrong is flagged at the moment you would go wrong, not in a footnote afterwards.

2 people → 4 accounts Dashboard only — signup is closed "Auto Confirm User" MUST be ticked TOTP enrolled in the app, not the Dashboard ~10 min per person

00Read this page first — it will save you an hour

Three facts about this system are not obvious, and each one has caused a wasted half-hour for somebody. If you read nothing else, read these.

Fact 1 — CERT and PROD are two completely separate systems They are two different Supabase projects with two different user databases. An account created on CERT does not exist on PROD. There is no sync, no copy, no "promote user" button.

So: two new people means four accounts. You will do the same procedure four times. This is the single most common source of "it works for me but not in production".
Fact 2 — there is no "admin" role to assign. The account IS the admin. This system has no roles table and no permission levels. Per ADR-016, every account that can log in has full administrative power — edit fencers, edit results, change scoring config, promote to production, trigger workflows.

There is nothing to tick to "make them an admin", and nothing to tick to limit them. Creating the account is the whole grant. Only create accounts for people you trust with production data.
Fact 3 — you must tick "Auto Confirm User", or the account will not work This project has no custom email server configured (smtp_host is unset), so it falls back to Supabase's built-in sender, which is heavily rate-limited and frequently does not arrive. The account also requires email confirmation before login (mailer_autoconfirm: false).

Put together: if you do not tick Auto Confirm User when creating the account, the new admin will be unable to log in and will be waiting for an email that may never come. Step 3 covers exactly where this tickbox is.

What you will do, visually

The procedure
  1. Gather what you need (5 min)
  2. Open the right project — and be certain which one it is
  3. Create the account (this is where the tickbox is)
  4. The new admin logs in and enrolls TOTP
  5. Verify it actually worked
  6. Repeat for the other environment, then the other person

01Before you start

You needDetail
Supabase Dashboard accessLogged in as the owner of the Fencer4Life organisation.
An e-mail address per personA real, working address they control. It becomes their login. It does not need to receive anything if you follow Step 3 correctly, but it must be unique per environment.
A password per personYou will set an initial one. The system's minimum is only 6 characters, which is far too weak for an account that holds full production power — use 16+ characters from a password manager.
The new admin, availableStep 4 has to be done by them, on their own device, because it pairs their authenticator app. You cannot do it for them and hand it over.
An authenticator app on their phoneGoogle Authenticator, Microsoft Authenticator, 1Password, Authy, Bitwarden — any TOTP app.
Decide this before you create anything Does this person need PROD, or only CERT? CERT is the staging environment — mistakes there are cheap and recoverable. PROD is the live ranking that fencers see.

If someone only needs to practise, prepare events, or learn the system, give them CERT only and skip the PROD half entirely. You can always add PROD later using the same steps. Adding it now because it feels tidier is how a training account ends up able to overwrite the live ranking.

What you should see A table headed Users with exactly one existing row — an account created 26 March 2026, e-mail beginning sp…@gmail.com. If you see zero rows or several, stop and re-check which project you are in.

03Create the account

STEP 3.1Start the "Add user" flow

Top-right of the Users table, click the green Add user button, then choose Create new user from the small dropdown.

Do not choose "Send invitation" The other option in that dropdown sends an invite e-mail. It will not work here — this project has no mail server configured, so the invite depends on Supabase's built-in sender, which is rate-limited to a handful of messages per hour and frequently never arrives. Always use Create new user.
STEP 3.2Fill the form — and tick the box

A small dialog appears with three controls. This is the part that goes wrong:

newadmin@example.com

Auto Confirm User?

Create user
This tickbox is the whole ballgame "Auto Confirm User" must be ON. It is off by default.

Leave it off and Supabase marks the account as unconfirmed and tries to e-mail a confirmation link. With no mail server configured that message is unreliable at best, and until it is clicked the new admin simply cannot log in — they will get an error that looks like a wrong password, and you will both spend twenty minutes resetting a password that was never the problem.

Enter the person's e-mail, set a strong password, tick the box, click Create user.

What you should see The dialog closes and a second row appears in the Users table. Its Last sign in column reads never — that is correct and expected at this stage.
Hand the password over safely Send it through a password manager's share feature, or a signal/whatsapp message you both delete — not e-mail, and not in a document. Ask them to change it after first login. Anyone holding this password holds the whole production database.

04The new admin logs in and enrols TOTP

This step is NOT in the Supabase Dashboard This is the part people hunt for and cannot find. There is no MFA screen in the Dashboard. Two-factor enrolment happens inside the SPWS app itself, automatically, the first time the new account signs in. You do not configure it anywhere in advance.
STEP 4.1Send them the right URL

The admin interface is the normal public site with ?admin=1 appended:

https://fencer4life.github.io/spws-automated-ranklist/?admin=1

Without ?admin=1 they will only see the public ranking and will report that "there is no login button" — because there isn't one. The sign-in modal only appears when that parameter is present.

Which environment does this URL hit? The deployed app can point at either project. Look at the small CT / PD toggle at the bottom of the ranking view — CT = CERT, PD = PROD. Whichever is highlighted is the database being used, and therefore the account list being checked. If they are logging in to CERT, that toggle must read CT.
STEP 4.2They sign in and scan the QR code

They enter the e-mail and password you created. The app checks for an existing authenticator and finds none, so it immediately shows an enrolment screen with a QR code and a text secret.

They open their authenticator app, add a new entry, scan the QR (or type the text secret if the camera is awkward), then enter the 6-digit code the app displays to confirm the pairing.

Tell them to save the recovery information NOW Have them store the text secret in a password manager before clicking confirm, or keep the authenticator entry backed up (1Password, Bitwarden and Authy all sync; Google Authenticator historically did not).

There is no self-service recovery. If they lose the phone and the secret, the only fix is manual deletion of the factor from the database by you — Step 7. Thirty seconds of care here saves that entirely.
What they should see The modal closes and the admin interface unlocks — the ranking view gains admin controls and the hamburger menu shows management screens (Seasons, Events, Identity). If the screen simply reappears, the 6-digit code was entered too late; codes rotate every 30 seconds. Wait for a fresh one and retry.
STEP 4.3Every login afterwards

From now on their login is: e-mail + password, then the current 6-digit code. No QR, no enrolment. Sessions last 1 hour, and the app signs them out after 59 minutes of inactivity — that is deliberate (ADR-016), not a fault.

05Verify it actually worked

Do not take "I logged in fine" as proof. Logging in is possible at aal1 — one factor — and an account stuck there can read everything but cannot dispatch workflows. Check the factor exists.

The quick visual check — have them try a real action

Ask them to open the Events screen and click a dispatch button such as Re-ingest event or Stage event on CERT.

ResultMeaning
✓ TriggeredFully enabled. TOTP enrolment completed and the session reached aal2. Done.
403 / mfa_requiredThey are signed in but at aal1 — enrolment never completed. Have them sign out fully and sign in again; the enrolment screen will reappear.
401Not recognised as a signed-in user at all. Session expired, or they are on the wrong environment (CT/PD toggle).

The definitive check — query the database

Run this from the repository root. It reads only; it changes nothing.

export SUPABASE_ACCESS_TOKEN=$(grep -E '^SUPABASE_ACCESS_TOKEN=' .env | cut -d= -f2- | tr -d '"'"' ')

for pair in sdomfjncmfydlkygzpgw:CERT ywgymtgcyturldazcpmw:PROD; do
  ref="${pair%%:*}"; label="${pair##*:}"
  echo "===== $label ====="
  curl -s -X POST \
    -H "Authorization: Bearer $SUPABASE_ACCESS_TOKEN" \
    -H "Content-Type: application/json" \
    -d '{"query":"SELECT u.email, u.created_at::date AS created, (u.email_confirmed_at IS NOT NULL) AS confirmed, COALESCE(u.last_sign_in_at::date::text,''never'') AS last_sign_in, (SELECT count(*) FROM auth.mfa_factors f WHERE f.user_id=u.id AND f.status=''verified'') AS totp FROM auth.users u ORDER BY u.created_at;"}' \
    "https://api.supabase.com/v1/projects/$ref/database/query"
  echo
done
What a fully-enabled admin looks like confirmed: true  ·  totp: 1  ·  last_sign_in showing a real date.

Any account showing totp: 0 is not finished — it can sign in and read, but cannot dispatch workflows. confirmed: false means Step 3.2's tickbox was missed.

Resetting a lost authenticator

Deleting the verified factor makes the app treat the next login as a first login, so enrolment starts over with a fresh QR code. Replace the e-mail and the project ref before running it.

# CERT ref: sdomfjncmfydlkygzpgw   ·   PROD ref: ywgymtgcyturldazcpmw
curl -s -X POST \
  -H "Authorization: Bearer $SUPABASE_ACCESS_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"query":"DELETE FROM auth.mfa_factors WHERE user_id = (SELECT id FROM auth.users WHERE email = ''person@example.com'');"}' \
  "https://api.supabase.com/v1/projects/<ref>/database/query"

Then have them sign in again and re-enrol. Verify with the Step 5 query that totp is back to 1.

Removing an admin entirely

Dashboard → Users → the row's ⋯ menu → Delete user. Do it in both projects — deleting on CERT leaves their PROD access fully intact, which is exactly the failure you do not want when someone leaves.

08Why it is built this way

ConstraintReasonDecision
Self-signup is disabledAny account that registers is a full administrator, so an open signup form is an open admin door. Closed on both environments on 23 July 2026.ADR-083
TOTP is mandatoryA password alone protects the entire ranking and every fencer's personal data.ADR-016
Workflow dispatch requires aal2The dispatcher can trigger production promotion. It now verifies the caller and requires completed MFA.ADR-083
Every account is a full adminA known limitation, not a design goal — authenticated currently means "any account holder". An explicit admin allowlist is a recorded follow-up; closed signup is what bounds the risk today.ADR-016, ADR-083

Every configuration value quoted here was read from the live CERT and PROD auth configuration on that date, not assumed: disable_signup: true, mailer_autoconfirm: false, smtp_host unset, password_min_length: 6, TOTP enrol and verify both enabled, jwt_exp: 3600.
