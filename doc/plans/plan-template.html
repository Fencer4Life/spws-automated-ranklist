CERT ingest · CERT → PROD · PROD data · Implementation Plan

Kill the PROD duplicate-and-mistag bug with a real reconciler

Promotion only ever adds — divergence from CERT is permanent and the organizer is guessed. Rewrite it as Create / Update / Delete over the whole active-season set, harden the CERT ingest that mints the duplicates, and repair the live PROD rows — end to end.

Plan — not started migration 20260711000001 retires fn_import_evf_events repairs 6 PROD rows

Why

The same "it's always EVF" mistake, in two layers

Two code paths carry the duplicate-and-mistag disease. The CERT ingest mints duplicate rows; the CERT→PROD promotion faithfully mirrors them and mis-tags the organizer. This plan fixes both, then cleans the rows already on PROD.

1 × Promotion hardcodes the organizer

WHERE txt_code = 'SPWS', never verified — every promoted EVF event is mis-tagged SPWS on PROD. Confirmed on 7 Samorin rows.

2 × Promotion is insert-or-refresh — no DELETE

It can add divergence from CERT but never remove it. CERT minted 7 Samorin rows, they promoted, CERT collapsed to 1 — and the 6 PROD orphans had no route to removal. Manual surgery was the only option.

3 ~ Promotion filters to PEW/MEW

The only incremental CERT→PROD path excludes domestic PPW/MPW. An event added after go-live has no route to PROD at all.

4 × Ingest mints on blank location — the upstream source

fn_allocate_evf_event_code matches by physical location; a venue-less future event skips every match step and gets a brand-new code each run. This is the engine that produced the 7 CERT duplicates in the first place.

! The one operation that can destroy data

Guard: delete only if enum_status = 'PLANNED' and zero tbl_result rows across its tournaments. Transactional — empty PLANNED tournaments first, then the event.

A results-bearing event absent from CERT is a CERT anomaly to alert on (Telegram), never erased. Once an event has results its status leaves PLANNED, so it is structurally untouchable here.

Workstream 2 · in scope — the live PROD mess

Delete the 6 dead Samorin rows & fix the survivor — as reconciler CRUD, then prove it

This is not a manual side-quest — it is the reconciler's first PROD run, exercised and verified. No hand-run DELETE.

  • The 6 orphans (PROD id_event 114–120 minus the survivor; PLANNED, zero results, absent from CERT's now-deduped set) fall into the DELETE branch → removed with their empty PLANNED tournament children, transactionally.
  • The survivor (CERT's single Samorin, matched by txt_code) lands in the UPDATE branch → its id_organizer is overwritten SPWS → EVF from CERT. The mis-tag is fixed by the same pass, no special-case.

▸ Validity test, after

A verification step (below, step 5) asserts on live PROD: exactly one Samorin event remains, it is EVF-tagged, its child tournaments are consistent, and the 6 orphan codes + their tournaments are gone. Self-heal, proven — not assumed.

Workstream 3 · in scope — get to the bottom of it

Close the ingest blank-location blind spot at the SQL layer

Root cause, traced through fn_allocate_evf_event_code (migration 20260429000001) for a venue-less Samorin scrape:

fn_normalize_city_key('', country) → loc_key = ''
  Step A  current-season slot reuse   IF v_loc_key <> '' → SKIPPED
  Step B  prior-season city carry     IF v_loc_key <> '' → SKIPPED
  Step C  next-free PEW{N+1}          → fresh code, every single run

So each daily scrape of a blank-location event minted a new PEW code → a new row. That is the engine behind the original 7 CERT duplicates. The shipped Python _find_existing_match slug-dedup stops the re-send, so the allocator is no longer re-hit — but the SQL layer still has the hole: if Python ever misses, fn_ingest_evf_calendar mints again.

Fix — mirror the Python ladder in SQL: identity before location

In fn_ingest_evf_calendar, before calling the allocator: if a current-season event already carries this id_evf_event or txt_evf_slug, reuse its code and UPDATE in place (CURRENT_SLOT_REUSE). Only on no identity match → the location-based allocator. Same order both layers now enforce — evf_id → slug → location. The allocator's location-gating becomes a safe backstop that only runs when identity genuinely can't decide; a blank-location re-scrape never reaches Step C.

Workstream 4 · consolidation (Option B)

  • fn_promote_season_skeleton (migration 20260628000002): remove the event-INSERT block (~88–98) + event sequence advance. Keep season row + scoring_config — scoring_config travels with the season (mid-season config edits stay manual on PROD).
