FTL seed delivery · ADR-080 §5 · Implementation Plan

Email the seed files to the organizer — as the SPWS Gmail, no domain

The "deliver" arrow of the §5 pipeline was logged as blocked on a new SMTP capability / Resend + an eu.org domain. It never needed any of that. SPWS already runs a dedicated Gmail — the one that scrapes results back in. We send as that same account over verified TLS, straight from the Python pipeline. No domain, no Resend, no eu.org.

Locked for this implementation

  • Payload: the current exporter emits mix-all files only. Predicted DE-bracket generation remains deferred and is never represented as complete.
  • Reliability: delivery is at-least-once. Manual UI/Telegram runs are deliberate re-sends and replace ts_ftl_sent; the deadline sweep selects only unstamped events.
  • Partial failure: SMTP acceptance precedes the stamp. A stamp failure may cause a duplicate retry; exact-once delivery would require a durable delivery-attempt/lease model and is out of scope.
  • Rollout: manual CERT first. The PROD schedule is present but disabled unless repository variable ENABLE_FTL_DEADLINE_SEND=true.
  • Secrets: app-password authentication is accepted as a pragmatic mailbox credential, without claiming protocol-limited account scope. Logs never print credentials or full recipient addresses.

Scenario walk-throughs

Happy path, retry path and adjacent failure

Manual CERT pilot

The admin saves the organizer address, clicks send, the trusted Edge Function dispatches ftl-seed.yml to CERT, Python resolves event + season metadata, generates a non-empty mix-all ZIP in memory, Gmail accepts the message, fn_mark_ftl_sent stamps the event, and the UI refresh exposes the timestamp.

Deliberate resend

The same UI or Telegram action may run again after a late registration. It regenerates the ZIP from current declarations and updates ts_ftl_sent. This is intentional, not deduplicated.

Deadline sweep

The gated PROD schedule considers only SPWS-registration events whose cutoff date has passed, whose event has not ended or been cancelled/completed, whose organizer address exists, and whose ts_ftl_sent is NULL. Empty registration bundles are skipped safely.

SMTP accepted, stamp fails

The workflow fails and reports the ambiguity without logging the recipient or secret. A later retry can duplicate the email; the operator-facing runbook names that at-least-once boundary.

Why

The blocker was the sender identity, not the email

ADR-080 §5 defines send_seed_to_organizer(event): generate FIE-XML seed files from an event's declared registrations, email the zip to tbl_event.txt_organizer_email, stamp ts_ftl_sent. The recorded blocker was “a new email/SMTP capability” that spiralled into a free eu.org domain + Cloudflare + Resend + a multi-day manual approval — heavy machinery bolted onto “send one email.”

✗ The over-engineering

Resend's only verification mode is full-domain DNS — so a provider choice forced a domain-acquisition project. The eu.org request 20260704215524-arf-56662 is not needed for organizer delivery and can be dropped for this purpose.

✓ The account already exists in the stack

The SPWS Gmail is already bound to doc/gas/Code.gs, which reads FTL results back in via GmailApp.search(...) (checkEmailForResults). Sending as that same address means the organizer's reply-with-results lands exactly where ingestion already watches — send-loop and receive-loop share one identity.

Decision — send from Python via Gmail SMTP (your call, this session)

Not Apps Script GmailApp. Rationale: one synchronous run; the zip is never stored (honors ADR-080 “generate-at-send, never a pre-stored attachment”); it matches the existing “Python does the work in a workflow, the UI/Telegram just dispatch” pattern (phase5-event-runner.yml, ingest-event.yml). Cost: one app-password GitHub secret, created by you — Claude never handles the credential.

Credential & transport security — the one place a secret flows

🔐 Four layers of protection

  • At rest: the app password lives only as the GitHub Actions encrypted secret SPWS_GMAIL_APP_PASSWORD (libsodium sealed-box, decrypted into the runner only at job time) — never committed, never in a tracked .env, never in this plan, the docs, or memory. Only its name ever appears.
  • In the runner: injected as an env var; GitHub auto-masks any registered secret in Actions logs (***). Read once via os.environ[...] at the entrypoint; lives in process memory only for the run.
  • On the wire: SMTP AUTH only base64-encodes credentials — base64 is not encryption, so TLS is the real protection. SMTP_SSL/465 is encrypted from the first byte, closing the STARTTLS-stripping window. If 587+STARTTLS is ever used instead, login() MUST run after starttls(context=ssl.create_default_context()). An explicit default context verifies the server cert + hostname (blocks MITM).
  • Blast radius: a Gmail App Password is independently revocable and does not expose the primary account password, but it is still a high-value account credential. Keep it only in GitHub Actions secrets, scope repository access tightly, redact failures, and rotate it on suspected exposure.
import ssl, smtplib
from email.message import EmailMessage

def send_via_gmail_smtp(msg: EmailMessage, smtp_user: str, app_password: str) -> None:
    ctx = ssl.create_default_context()          # verifies cert + hostname
    with smtplib.SMTP_SSL("smtp.gmail.com", 465, context=ctx) as s:
        s.login(smtp_user, app_password)         # only over an encrypted channel
        s.send_message(msg)
    # never log app_password; never echo it in an exception
