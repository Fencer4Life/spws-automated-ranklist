00Cold-start briefing — read this first

If you are a new session with no memory of the investigation, this section is enough to execute the rest safely.

What is wrong, in four sentences

  1. Six public tables have RLS disabled and anon holds full DML plus TRUNCATE. Verified on CERT and PROD; anon INSERT/UPDATE proven live (error 23502, a data error — not 42501 permission denied).
  2. vw_fencer_aliases is an owner-rights view (owner=postgres, reloptions=NONE) serving 365 fencer identities to anon on PROD, and RLS on the base tables will not stop it.
  3. The dispatch-workflow Edge Function never checks its caller; verify_jwt=true is satisfied by the public anon key, so anyone can dispatch promote-season.yml or ftl-seed.yml against PROD.
  4. Root cause: ALTER DEFAULT PRIVILEGES on public grants anon everything on every new table, function and sequence. Nobody made a mistake — the schema did it automatically.

Full evidence, with re-runnable commands: security-case-cert-prod-2026-07-23.html. Read it if any claim below needs justifying.

Scope decision — A and B, not C

One scope refinement you must understand before starting Earlier drafts put aal2 (MFA) on the RLS write policies inside fix A. It is removed, deliberately. STEP 5Write the migration (fix A + fix B in one file)

File: supabase/migrations/20260723000001_adr083_deny_by_default_grants.sql (match the existing YYYYMMDD######_snake_name.sql convention). Comment header in the house style of 20260327000001_revoke_write_functions.sql — that migration fixed this exact bug class in March 2026 and its comment explains the mechanism; mirror it.

Block 1 — the DEFINER hardening (must come before the REVOKEs)

-- fn_enqueue_affected_events runs from trg_fencer_change_enqueue on tbl_fencer.
-- Today the admin path reaches it only via SECURITY DEFINER RPCs, so the queue
-- INSERT succeeds. Making it DEFINER removes the latent trap: any future
-- authenticated-context write to tbl_fencer would otherwise fail after the
-- REVOKE below, surfacing as a confusing permission error on a fencer edit.
-- Mirrors fn_audit_log, which is DEFINER on three triggers for the same reason.
CREATE OR REPLACE FUNCTION fn_enqueue_affected_events(...)
  ... SECURITY DEFINER SET search_path = public ...;

Always pair SECURITY DEFINER with an explicit SET search_path — a DEFINER function without it is a privilege-escalation vector. postgrestools will flag this; do not suppress it.

Block 2 — fix A: the six tables

ALTER TABLE tbl_result_draft              ENABLE ROW LEVEL SECURITY;
ALTER TABLE tbl_tournament_draft          ENABLE ROW LEVEL SECURITY;
ALTER TABLE tbl_event_ingest_history      ENABLE ROW LEVEL SECURITY;
ALTER TABLE tbl_tournament_ingest_history ENABLE ROW LEVEL SECURITY;
ALTER TABLE tbl_recompute_queue           ENABLE ROW LEVEL SECURITY;
ALTER TABLE tbl_recompute_watermark       ENABLE ROW LEVEL SECURITY;

REVOKE ALL ON tbl_result_draft, tbl_tournament_draft,
              tbl_event_ingest_history, tbl_tournament_ingest_history,
              tbl_recompute_queue, tbl_recompute_watermark
  FROM anon, authenticated;

No policy is created. RLS with zero policies denies everything, and service_role bypasses RLS inherently — which is how the entire Python pipeline reaches these tables. Verified: 11 workflows pass SUPABASE_*_SERVICE_ROLE_KEY, and no file under frontend/src or supabase/functions references any of the six.

Block 3 — fix A: the alias view

REVOKE ALL ON vw_fencer_aliases FROM anon;
REVOKE ALL ON vw_fencer_aliases FROM authenticated;
GRANT SELECT ON vw_fencer_aliases TO authenticated;

The view is owner-rights, so RLS beneath it does nothing — grants are the only control. The admin UI reads aliases through fn_list_fencer_aliases, and SELECT for authenticated preserves any direct read.

Block 4 — fix A: the five RPCs

REVOKE EXECUTE ON FUNCTION fn_find_or_create_tournament(...) FROM anon, PUBLIC;
REVOKE EXECUTE ON FUNCTION fn_create_registration(...)        FROM anon, PUBLIC;
REVOKE EXECUTE ON FUNCTION fn_commit_event_draft(UUID)        FROM anon, PUBLIC;
REVOKE EXECUTE ON FUNCTION fn_discard_event_draft(UUID)       FROM anon, PUBLIC;
REVOKE EXECUTE ON FUNCTION fn_enqueue_affected_events(...)    FROM anon, PUBLIC;

GRANT EXECUTE ON FUNCTION fn_create_registration(...) TO authenticated;   -- api.ts calls it
Two traps here 1. REVOKE … FROM anon alone is not enough — the grant comes from the PUBLIC default (the bare =X/postgres ACL entry). You must revoke from PUBLIC too.
2. Get the exact argument signatures from the catalog first; several of these are overloaded. Use \df+ fn_name or query pg_get_function_identity_arguments.

Do not touch the intentionally-public read-only functions (fn_ranking_ppw, fn_ranking_kadra, fn_season_summary, fn_export_scoring_config, fn_fencer_scores_rolling and variants) — the public ranking depends on them and 20260327000001 documents them as deliberate.

Block 5 — fix B: kill the generator

-- Root cause. Every object created in public was auto-granted to anon:
--   pg_default_acl → {anon=arwdDxtm/postgres, authenticated=arwdDxtm/postgres, ...}
-- Verified: all 23 public tables/views are owned by postgres, so targeting
-- FOR ROLE postgres covers everything the migration path can create.
ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA public
  REVOKE ALL     ON TABLES    FROM anon, authenticated;
ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA public
  REVOKE EXECUTE ON FUNCTIONS FROM anon, authenticated;
ALTER DEFAULT PRIVILEGES FOR ROLE postgres IN SCHEMA public
  REVOKE ALL     ON SEQUENCES FROM anon, authenticated;

All three object types are required — the live check found anon on r (tables), f (functions) and S (sequences). FOR ROLE postgres is what makes it bite; without the qualifier it targets the executing role's defaults and silently does nothing useful.

Blast radius: zero on existing objects. ALTER DEFAULT PRIVILEGES affects only objects created after it runs. This is why B is safe to ship alongside A.

STEP 10ADR-083 (Markdown source, generated HTML)

Create doc/adr/083-server-enforced-authorization.md. Follow the structure of doc/adr/016-supabase-auth-totp-mfa.md — that ADR is this one's direct ancestor.

Title: "Server-enforced authorization: deny-by-default grants and a standing posture gate".
Status: Accepted. Date: 2026-07-23. Relates to: ADR-016 (which this amends, not supersedes), ADR-004, ADR-041 (edge-function dispatch), ADR-072 (the recompute queue), ADR-078 (GDPR — anon could read fencer identities).

Context must state the finding plainly: the March 2026 REVOKE pattern (20260327000001) was never carried into the May/June work, because ALTER DEFAULT PRIVILEGES grants anon everything on every new object. Nobody made a mistake — which is exactly why a convention could not have caught it.

Decision: (1) pipeline/staging objects are service-role only; (2) new objects are ungranted by default; (3) the anon-executable surface is an explicit allowlist enforced by pgTAP; (4) the workflow dispatcher authenticates its caller and requires aal2.

Alternatives considered — record all of these with reasoning, because they were genuinely evaluated:

  • Option C, separate pipeline schema. Rejected for now: the pipeline is a PostgREST client (db_connector.py:683 uses create_client; draft_store.py:13 forbids psycopg at runtime), so the schema would have to be exposed to PostgREST anyway and security would fall back to grants — the same place Option B already puts it, for a fraction of the risk.
  • aal2 on RLS write policies. Rejected: the eight admin fencer RPCs are SECURITY DEFINER and bypass RLS, so it would protect three low-risk direct-write call sites while adding a lockout surface. Follow-up: aal2 checks inside the DEFINER RPCs.
  • Explicit tbl_admin allowlist replacing bare authenticated. Deferred: strictly better, but rewrites all 17 policies and belongs in its own change, not an incident response. Record as a dated follow-up.
  • Recreating vw_fencer_aliases with security_invoker. Deferred: correct, but couples the admin UI to the draft tables' RLS and would force an otherwise unnecessary authenticated read policy onto all six.
  • Deny-list instead of allowlist in the pgTAP guard. Rejected: would have caught none of these findings.
