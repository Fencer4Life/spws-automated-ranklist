Why exactly these six

RLS was established once, in 20250301000002_rls_policies.sql, covering the tables that existed then. All six offenders were created later — the Phase-1/2 ingest pipeline (May) and the CDC recompute work (June) — and no subsequent migration ever enabled RLS on them. Confirmed: none of the six appears in any migration containing ROW LEVEL SECURITY or CREATE POLICY. This is drift, not a decision.

03Why it is genuinely exploitable

RLS being off only matters if the API roles can reach the table. They can — fully:

table                          grantee         privileges
tbl_event_ingest_history       anon            DELETE,INSERT,REFERENCES,SELECT,TRIGGER,TRUNCATE,UPDATE
tbl_recompute_queue            anon            DELETE,INSERT,REFERENCES,SELECT,TRIGGER,TRUNCATE,UPDATE
tbl_recompute_watermark        anon            DELETE,INSERT,REFERENCES,SELECT,TRIGGER,TRUNCATE,UPDATE
tbl_result_draft               anon            DELETE,INSERT,REFERENCES,SELECT,TRIGGER,TRUNCATE,UPDATE
tbl_tournament_draft           anon            DELETE,INSERT,REFERENCES,SELECT,TRIGGER,TRUNCATE,UPDATE
tbl_tournament_ingest_history  anon            DELETE,INSERT,REFERENCES,SELECT,TRIGGER,TRUNCATE,UPDATE
                               (identical for authenticated and service_role)
The anon key is public by design It is embedded in the frontend bundle — that is normal and not the problem. The problem is that RLS is the only thing standing between that public key and these tables, and it is switched off. With no policy, PostgREST will happily serve a DELETE or TRUNCATE. This is write access, not merely a data leak.

Concrete blast radius, worst case

  • tbl_recompute_queue / tbl_recompute_watermark — flood or wipe the queue, or move the watermark, and the incremental recompute engine either stalls or silently skips work. Ranking correctness depends on these.
  • tbl_result_draft / tbl_tournament_draft — inject rows into the staging area that feeds ingestion, i.e. an untrusted path toward real results.
  • tbl_*_ingest_history — falsify or erase the traceability record, which is exactly the evidence you would need to detect the other three.

Not yet established, and worth doing first: whether anything has actually been tampered with. Row counts and ts_created ranges on PROD versus what the pipeline expects would answer it. I did not query PROD.

04The fix, and the pattern to copy

The house pattern is already in the repo — 20250301000002_rls_policies.sql:

ALTER TABLE tbl_registration ENABLE ROW LEVEL SECURITY;
CREATE POLICY "Admin all registrations" ON tbl_registration
  FOR ALL USING (auth.role() = 'authenticated')
  WITH CHECK (auth.role() = 'authenticated');

These six are internal pipeline tables with no legitimate anonymous use, so the likely correct shape is stricter than the read-only public pattern used for ranking data: enable RLS, and grant access to service_role only (the workflows), with no anon policy at all. Confirm that intent with the operator before writing it — if any frontend view reads a draft or queue table, a blanket lock will break it.

Sequence — the project's rules apply in full

  1. Check the frontend first. Grep the Svelte app for these table names. If the admin UI reads them through the anon key, an authenticated policy is required rather than service-role-only.
  2. RED first. Add pgTAP coverage asserting RLS is enabled and that anon cannot select/insert/delete on all six. Watch it fail. The suite is the contract, and the pgTAP total in the spec must be bumped.
  3. Write one migration enabling RLS + policies on all six. Consider also REVOKEing the surplus anon/authenticated grants — RLS alone leaves TRUNCATE rights nominally in place.
  4. Run postgrestools on the changed SQL — mandatory after any SQL edit in this repo (the plpgsql-check skill).
  5. Verify on LOCAL by actually attempting an anon write and seeing it refused, not merely by reading the catalog.
  6. Ship LOCAL → CERT → PROD via release.yml; the task is not done until the workflow is green and PROD is verified. Then confirm the advisor clears in the dashboard.
  7. Documentation coherence gate — this changes the security posture, so the handbook security/subsystem pages and the RTM need the pass, and an ADR is likely warranted for "pipeline tables are service-role only".
Blocker for whoever picks this up main CI was red earlier in this campaign's history on stale test expectations, and the Koszyk migration may still be uncommitted. Check git status and CI before adding a migration — shipping a security fix on top of a red pipeline will stall it. Also: the Supabase MCP server is unauthorised in this session, so I could not query CERT/PROD directly; everything above is LOCAL plus the e-mail. Authorise it via claude mcp or /mcp in an interactive session to inspect PROD.

05Reproduce the investigation

# tables in public with RLS off and no policies
docker exec supabase_db_SPWSranklist psql -U postgres -d postgres -c "
SELECT c.relname, c.relrowsecurity,
  (SELECT count(*) FROM pg_policies p
    WHERE p.schemaname='public' AND p.tablename=c.relname) AS policies
FROM pg_class c JOIN pg_namespace n ON n.oid=c.relnamespace
WHERE n.nspname='public' AND c.relkind='r'
ORDER BY c.relrowsecurity, c.relname;"

# what anon can actually do on them
docker exec supabase_db_SPWSranklist psql -U postgres -d postgres -c "
SELECT table_name, grantee, string_agg(privilege_type,',' ORDER BY privilege_type)
FROM information_schema.role_table_grants
WHERE table_schema='public' AND grantee IN ('anon','authenticated','service_role')
GROUP BY table_name, grantee ORDER BY table_name, grantee;"

06Open questions for the operator

  1. Service-role-only, or does the admin UI need these? Decides the policy shape. This is the one real fork.
  2. Do we audit PROD for tampering before or after locking the door? Locking first is safer; auditing first preserves the evidence unchanged.
  3. Revoke the grants as well as enabling RLS? Defence in depth versus a larger diff.
