01Cause A — the season rolled over

All three 03_views_api failures reduce to a single fact. The seed of 2026-06-28 predated the 14 July season-boundary incident fix (20260714000002). The refreshed seed carries the corrected boundaries, so the active season changed:

fn_age_category buckets on season_end_year − birth_year, with V1 at 40–49 and V2 at 50–59. Moving the end year from 2026 to 2027 ages every fencer by one, and exactly one fixture fencer crosses a boundary:

That single crossing explains two of the three failures, because fn_ranking_ppw filters by the fencer's birth-year category, not the tournament's:

  • Test 3 — V2 gains FORAJTER, so the count is 4, not 3.
  • Test 12 — V1 loses its only member, so the ranking is empty and the >= 1 assertion fails.

Test 4 fails for a related but distinct reason. It passes the literal season code 'SPWS-2025-2026'. The setup block clears tournament data for bool_active = TRUE only — which is now the 2026-2027 season — so 2025-2026 is left holding real seed data and returns 24 fencers. The assertion's stated intent ("explicit season parameter returns the same result as the implicit one") is no longer what it tests.

The defect underneath The fixture borrows six real seed fencers by surname (SELECT id_fencer INTO … WHERE txt_surname = 'DUDEK') and depends on their real birth years landing in specific categories. It is therefore hostage to both the seed and the calendar. This would have broken at the next season rollover regardless of any seed refresh — the refresh only brought the date forward. Bumping 3 to 4 re-arms it for next July.

Secondary hazard, same lines: those lookups are unqualified by birth year, the exact pattern just fixed in export_seed.py::fencer_lookup(). If PROD ever holds two fencers sharing one of these six surnames, SELECT … INTO silently binds the first row.

02Cause B — an unscoped backfill in test 50.7

Subtest 50.7 mirrors the migration's slug-backfill statement to assert its tie-break rule (lowest id_event per URL group wins). Its WHERE clause selects every EVF event in the database with a non-empty url_event and a NULL txt_evf_slug — not just the seven it created. An inline comment claims the statement is "scoped implicitly … since it's the only EVF row set sharing this url_event", which was true of the old seed and is no longer true.

The refreshed seed contains real EVF events that are still unslugged. The ranked CTE de-duplicates only within the NULL-slug set; it never checks against rows that already carry a slug. A seed row therefore wins rn = 1 and collides with an existing slug in season 3, violating idx_tbl_event_evf_slug and aborting the file — which is why it reports "planned 8, ran 6" rather than a clean failure.

This is the only unscoped mutation in the file; subtests 50.1–50.6 and 50.8 all operate on explicitly captured id_event values and are unaffected.
