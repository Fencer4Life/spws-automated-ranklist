02Sending wave 1

Never without the operator saying so, in that session --send is the only thing that mails anyone. Approval given on 19 July does not carry into the next session, and approval to rehearse is not approval to send. 
$V $S/send_wave1.py                       # dry run, rewrites the preview, sends nothing
$V $S/send_wave1.py --send --limit 1      # one real fencer, proves log + resume
$V $S/send_wave1.py --send                # the remaining 101, ~29 min

Re-running is safe: everyone already in send_log.jsonl is skipped, so a crash mid-run costs nothing. Tokens are deterministic, so a resend reissues the same code and never creates a second identity for one person.

Rehearsing instead

$V $S/send_wave1.py --send --redirect a.atanassow@protonmail.com --only "TK,PRZYSTAJKO Daniel"

--redirect delivers fully rendered production messages to one address and deliberately writes no send log — a rehearsal that marked people as contacted would silently drop them from the real wave, and nothing downstream would ever reveal it. Subjects are stamped with fencer and variant so seven of them cannot be confused.

03Monitoring the wave

The tracker reads send_log.jsonl as well as the responses, so during the run the report shows the send itself progressing — amber for mailed-and-waiting — not just the replies. Start it before sending, and the 29 minutes are observable rather than opaque.

StripMeans
greenrefreshed recently; shows cycle count and token age
amberno successful cycle for more than two cycles
redlast cycle failed — carries the subprocess's own stderr
greytracker offline — the process is gone
Why the strip exists at all A tracker that silently stops does not look broken — it looks like nobody is replying. Liveness comes from /state.json, which is rewritten on every cycle including failed ones; content is swapped only when the generation advances. If the process dies outright the fetch itself fails and the page says so, which no server-side flag could ever report.

Interval: 60s during the wave, 300s afterwards. It was run at 3s during testing — that is a test cadence only, roughly 4,300 Google reads a day. A cycle takes ~25s regardless, because it reads seven forms.

05Traps — all of these have already bitten

  • A pass that validated itself. Never check scopes against token.json; ask Google's tokeninfo endpoint. preflight.py does.
  • The one message a human proofread was hand-written. send_test.py hard-coded a correct vocative while production generated a wrong one, so the test structurally could not reveal the bug. Both now call one function.
  • Two parsers for one roster. gen_raport.py had its own copy and kept listing a fencer the sender had already dropped. Deleted. gen_uczestnicy.py, gen_prosba.py, gen_merge.py still carry theirs — prefer roster.py for anything new.
  • A threshold built from the wrong quantity. Staleness was measured against the configured interval, but a cycle takes ~25s, so the strip called a healthy tracker stale. Only a 3-second test exposed it; at 300s it was accidentally correct.
  • innerHTML swaps lose scroll. The document collapses for an instant and the browser clamps scrollY to 0. Height is pinned across the swap.
  • A fallback that does not announce itself. read_responses.py records matched_by; all 8 test submissions matched by kod. If it ever says nazwisko or e-mail, that is a real defect.
