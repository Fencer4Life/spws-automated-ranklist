Admin Auth — Supabase Auth + TOTP MFA

Replaces client-side password gate (ADR-004). Server-enforced authentication with two-factor verification. Multiple admins supported.

Before (ADR-004)

  • Password in HTML attribute
  • Client-side comparison only
  • anon can call write RPCs
  • Trivially bypassable via DevTools

After (Supabase Auth + TOTP)

  • Credentials never in client bundle
  • Server verifies JWT on every request
  • Write RPCs require authenticated role
  • Email + password + rotating 6-digit code
  • Multiple admins, each with own 2FA
Flow: User visits ?admin=1 → Sign-in (email + password) → TOTP challenge (6-digit code) → Authenticated admin session. First-time only: MFA enrollment (scan QR code) happens between sign-in and TOTP challenge.
Multi-admin: Each admin has their own email + password + authenticator app. No limit on Supabase free tier (50k MAU). Independent sessions.
Step 1 — Sign In (email + password)
User navigates to ?admin=1. The app detects no active session and shows the sign-in modal.
This replaces the old single-password modal. Credentials are sent to Supabase GoTrue (server-side), never compared client-side.

Step 2b — TOTP Challenge (every login)
On every subsequent login, after email+password succeeds, Supabase returns a partial session requiring MFA verification. The app shows this screen. The user opens their authenticator app, reads the current 6-digit code, and enters it. Code rotates every 30 seconds.

Step 3 — Authenticated Admin Session
After successful TOTP verification, Supabase issues a full authenticated JWT. The admin UI unlocks: floating toolbar (bottom-left), admin sidebar section (orange items), and all write RPCs are now authorized. Client-side inactivity timer: 59 minutes. On timeout → signOut() → JWT destroyed → sign-in modal reappears. JWT expiry set to 1h server-side, so the 59-min client timeout fires first — no token refresh logic needed.

Technical Flow — What Happens Under the Hood
1. Sign-in: supabase.auth.signInWithPassword({ email, password })
    → GoTrue verifies credentials, returns partial session (MFA required)

2. Check MFA enrollment: supabase.auth.mfa.listFactors()
    → If empty → enrollment flow (Step 2a)
    → If TOTP factor exists → challenge flow (Step 2b)

3a. Enrollment:
    supabase.auth.mfa.enroll({ factorType: 'totp' }) → returns QR URI + secret
    supabase.auth.mfa.challengeAndVerify({ factorId, code }) → activates factor + issues JWT

3b. Challenge:
    supabase.auth.mfa.challenge({ factorId }) → returns challengeId
    supabase.auth.mfa.verify({ factorId, challengeId, code }) → issues full JWT

4. Server-side enforcement:
    REVOKE EXECUTE ON FUNCTION fn_import_scoring_config FROM anon, PUBLIC;
    PostgREST checks JWT → role = authenticated → function executes
    Without JWT (or anon JWT) → permission denied

5. Session management (no refresh needed):
    JWT expiry = 1h (Supabase dashboard setting)
    Client inactivity timer = 59 min (fires before JWT expires)
    On timeout or "Wyloguj" → supabase.auth.signOut() → JWT destroyed
    No onAuthStateChange refresh listener — single-use session, simple code

6. Multiple admins:
    Each admin created in Supabase Dashboard → Auth → Users
    Each enrolls their own TOTP factor independently
    All authenticated users get admin privileges (no role table needed)
    Supabase free tier: 50,000 MAU — effectively unlimited
