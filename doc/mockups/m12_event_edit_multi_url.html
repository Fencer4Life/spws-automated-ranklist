Założenia projektowe — co się NIE zmienia

  • ✔ url_event = URL #1 — nazwa kolumny, kontrakt RPC, znacznik 🔗 na karcie kalendarza, ścieżka discovery — bez zmian.
  • ✔ tbl_tournament.url_results — bez zmian. Pokazywany w drilldown rankingu.
  • ✔ fn_create_event i fn_update_event — istniejące parametry zachowane; cztery nowe parametry p_url_event_2..p_url_event_5 dodane jako nullable defaults, więc wszystkie istniejące wywołania (frontend, scraper, promote.py, seed dump) działają bez modyfikacji.
  • ✔ vw_calendar — kontrakt karty kalendarza nie wymaga ekspozycji slotów #2–#5 (są to dane techniczne dla discovery, nie dla użytkownika końcowego). Można ekspozycję dodać później bez wpływu na obecne komponenty.
  • ✔ ADR-028 NULL-only refresh — niezmiennik trzyma się trywialnie: scraper EVF zapisuje tylko sloty obecnie NULL/empty.
  • ✔ ADR-036 schema-driven export PROD — automatycznie obejmie nowe kolumny.

Co się zmienia

  • ⚠ Migracja: ALTER TABLE tbl_event ADD COLUMN url_event_2..5 TEXT (4 nullowalne kolumny).
  • ⚠ EventManager.svelte: 4 dodatkowe inputy w sekcji URL-e wyników; toggle "Pokaż / Ukryj dodatkowe sloty"; serializacja przy zapisie.
  • ⚠ populate-urls.yml / python/url_discovery/: pętla po slotach url_event, url_event_2..5, pomija nulle, deduplikuje rezultaty po url_results.
  • ⚠ Klucze i18n: event_results_url_label (istniejący — "URL #1"), event_results_url_extra_label (nowy — "URL #2..#5"), event_results_url_show_extras / event_results_url_hide_extras.

Decyzje UI

  • Slot #1 = primary, oznaczony cyjanową obwódką. To jest url_event i pełni dotychczasową rolę (link 🔗 w kalendarzu, seed dla auto-population dla 95% wydarzeń).
  • Sloty #2–#5 ukryte za disclosure domyślnie (zwinięte), aby formularz nie urósł wizualnie dla typowego wydarzenia. Disclosure pokazuje licznik wypełnionych — np. (3 wypełnionych) — żeby admin nie przegapił istnienia dodatkowych URL-i.
  • Brak limitu typu / etykiety na slot. Sloty są równorzędne — każdy może zawierać URL Engarde, FTL, 4Fence albo inny. Discovery rozpoznaje platformę po URL-u (już to robi).
  • Limit 5 jest hardkodowany w schemacie. Pokrywa Budapest (3), zostawia headroom (×2). Jeśli kiedyś pojawi się event z 6+ URL-ami, zostaje migracja na tbl_event_url child table — to jest moment, kiedy bardziej skomplikowane rozwiązanie zaczyna się opłacać. Na dziś — overkill.
