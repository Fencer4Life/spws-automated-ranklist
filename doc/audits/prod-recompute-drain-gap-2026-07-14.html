Investigation record · ADR-072 amendment (2026-07-14) · CERT vs PROD

The PROD recompute-drain gap, explained

A birth-year edit doesn't just change a number — it can move a fencer's result into a different age-category bracket, which has to be re-scored. That re-scoring was automatic on CERT and not on PROD. This is the investigation that found the gap, sized the actual risk, and the amendment that closed it.

Resolved 2026-07-14 — recompute-drain-prod.yml shipped See ADR-072 amendment Verified against live workflow/migration source

The three moving parts

Trigger → queue → worker

The self-heal loop (ADR-072, built 2026-06-15) has three pieces, and they live at three different layers. Understanding the gap meant understanding that only two of the three were guaranteed to exist everywhere — the third was the one missing on PROD.

1. The trigger — fires everywhere, automatically, by construction

trg_fencer_change_enqueue (supabase/migrations/20260615000001_cdc_recompute_dedup.sql:84-88) is a Postgres trigger:

CREATE TRIGGER trg_fencer_change_enqueue
    AFTER UPDATE ON tbl_fencer
    FOR EACH ROW
    EXECUTE FUNCTION fn_fencer_change_enqueue();

It's column-aware — it only acts if int_birth_year or txt_nationality actually changed (a name/alias edit enqueues nothing, since identity is the durable foreign key, not the display name, per ADR-003). When it fires, it calls fn_enqueue_affected_events(id_fencer), which inserts one row into tbl_recompute_queue for every distinct id_event that fencer has a result in.

This trigger is defined once, in a migration. Once that migration is deployed to an environment, the trigger exists in that environment's database and fires on every UPDATE to that environment's tbl_fencer, unconditionally — regardless of whether the UPDATE came from a human clicking "Save" in the admin Birth-Year Review screen, a background script, or a hand-written SQL migration. The trigger was never the gap.

2. The queue — a table, not a process

tbl_recompute_queue just holds rows: id_event, enum_status (PENDING → CLAIMED → DONE), timestamps. A partial unique index keeps at most one PENDING row per event. A row landing in this table does not, by itself, cause anything to happen — it sits PENDING until something claims it.

3. The worker — the only thing that actually acts, and it did not run itself on PROD

python/pipeline/recompute/worker.py's drain_recompute_queue() checks a debounce watermark (only proceeds once quiet ≥120s), claims every non-DONE row, runs RECOMPUTE_DOMESTIC (re-derive V-cat, re-partition results across brackets, re-score, re-count) for each affected event, marks them DONE. Real, working logic — but inert until its main() is invoked from a shell. As of this investigation, nothing invoked it against PROD.

Walking it through — the same edit, three environments, before the fix

What would have happened correcting the same fencer's birth year

LOCAL

UPDATE tbl_fencer runs → trg_fencer_change_enqueue fires → the affected event lands in LOCAL's tbl_recompute_queue as PENDING → it sits there until a developer runs the worker CLI by hand. Accepted as normal for a dev sandbox.

CERT

Same trigger fires; within at most 15 minutes recompute-drain.yml claims the row, recomputes, marks it DONE. The scenario ADR-072 was built for.

PROD (before this amendment)

Same trigger fires — PROD has the identical schema. The affected event lands in PROD's tbl_recompute_queue as PENDING, and then nothing drains it. The raw int_birth_year column would be correct; the event's derived V-cat bracket, counts and scores would stay stale on the live public ranklist until someone noticed and ran a recompute by hand.

Why the asymmetry existed — a design choice, not an oversight

PROD wasn't originally meant to be where master-data corrections originate

  • ADR-072 framed the self-heal loop as solving a CERT problem: "Birth years will be wrong on first ingest and corrected repeatedly during cleanup" — cleanup is a CERT-side activity (identity resolution, alias review, dedup sweeps against CERT's live roster during ingestion review).
  • The environments-and-release handbook page's promotion-contracts table lists exactly three things that move CERT→PROD: season skeleton/config, calendar events, event results. No "fencer master data" row. The implied original model: identity gets corrected and self-healed on CERT, and PROD receives already-recomputed event results through normal promotion — not an independent copy of tbl_fencer corrections plus its own recompute cycle.

Directly editing PROD's tbl_fencer — which the organizer's 2026-07 master-list reconciliation does, writing confirmed birth years to LOCAL, CERT and PROD in one migration rather than staging on CERT and waiting for a later promotion cycle — was a genuinely different shape of operation than the original architecture assumed. Schema-permitted (the trigger doesn't care which environment or how the UPDATE arrives); not, until this amendment, operationally supported (nothing drained what it enqueued on PROD).

Why this mattered concretely, not just in theory

9 of 10 birth-year corrections are neutral; 1 genuinely re-brackets, in a past season

RECOMPUTE_DOMESTIC's entire reason for existing is re-bracketing: moving a result from one V-cat bracket to another when a corrected birth year crosses a threshold (V0↔V1↔V2↔V3↔V4 at ages 30/40/50/60/70). Checking each of the 10 real corrections in the birth-year batch against every season the fencer actually has a result in — not only the current season — surfaced one real case:

! SAMECKA-NACZYŃSKA Martyna — V0 → V1 in SPWS-2024-2025, invisible in the current season

BY 1986 → 1985. At SPWS-2024-2025's season end (2025) that's age 39 (old, V0) → age 40 (new, V1) — a genuine boundary crossing. At the current SPWS-2025-2026 season she's already V1 either way, which is exactly why a current-season-only check would have missed it. 8 of her SPWS-2024-2025 results — including IMEW-V0-F-EPEE-2024-2025 (European Veterans Championships 2025) — sit in V0 brackets and move to V1. Recompute is not a no-op for this fencer, on any environment.

The other 9 corrections check out clean across every season they have results in — a recompute for them touches the same bracket they're already in. But Martyna's case alone was enough to make "PROD queue entries that never drain are harmless" false for this exact batch, not just as a hypothetical future risk.

Resolution — what was actually built

ADR-072 amended; recompute-drain-prod.yml shipped 2026-07-14

Rather than holding PROD's birth-year corrections back or leaving a manual-drain contingency, the self-heal loop was extended to PROD as an amendment to ADR-072 — see the full amendment text. In short:

  • New workflow .github/workflows/recompute-drain-prod.yml — structurally identical to CERT's (same 15-minute cron, same worker invocation, same manual-dispatch debounce input, same Telegram-on-failure step), pointed at secrets.SUPABASE_PROD_URL / secrets.SUPABASE_PROD_SERVICE_ROLE_KEY (both already-provisioned — no new credential).
  • Concurrency group is the existing prod-write, not a new dedicated group. promote-season.yml, evf-sync.yml and promote.yml already share prod-write so PROD writers never overlap; the recompute drain joins that same group rather than getting its own, which is what actually prevents it from racing a live season/event-result promotion.
  • No new escalation code needed. A below-minimum-bracket auto-drop (ADR-074, no-halt) already surfaces via the POST_COMMIT reactor's ON_LOSS Telegram escalation whenever RECOMPUTE_DOMESTIC reaches Commit — verified to fire identically regardless of which environment's database the flow ran against, so a dropped bracket on PROD alerts exactly like one on CERT does today.
  • CERT's job, cadence and concurrency group are unchanged.
  • Cataloged in the workflow catalog and the operator runbooks' "Repair after identity/master-data change" procedure.
