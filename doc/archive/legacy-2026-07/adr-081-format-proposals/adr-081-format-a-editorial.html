Why this exists

Context

The CERT→PROD calendar-promotion path (python/pipeline/promote.py::promote_calendar → fn_import_evf_events, the ADR-026 amendment) was written as an EVF special-case and carried three coupled defects, all from the assumption that this path only ever sees EVF:

  1. Hardcoded organizer. The function selected organizer SPWS once and stamped it on every promoted event. Seven live Samorin rows were mis-tagged SPWS instead of EVF.
  2. Insert-or-refresh only; no DELETE. It could introduce divergence from CERT but never remove it. After CERT collapsed seven Samorin duplicates to one, six dead PROD rows had no route to removal.
  3. PEW/MEW prefix filter. The only incremental event path excluded domestic PPW/MPW, so a domestic event added after season go-live could never reach PROD.

A fourth upstream defect fed the duplicates: fn_allocate_evf_event_code gated location matching on a non-empty location. A venue-less event skipped reuse and minted a fresh code on every scrape.

Root realization. One-time season-skeleton copy and incremental calendar promotion are the same operation at different points in time: reconcile PROD’s event set to CERT’s.

Chosen architecture

Decision

Replace fn_import_evf_events with fn_mirror_events_to_prod(p_creates, p_updates, p_deletes JSONB): a complete Create / Update / Delete reconciler over the whole active-season event set, keyed by txt_code. promote_calendar becomes the diff engine.

1

Organizer-agnostic, verified from source

id_organizer is never hardcoded. CERT’s organizer code is resolved to a PROD id by txt_code through shared _prod_code_to_id; raw ids may diverge across environments. SQL raises if organizer or prior-event codes cannot resolve: fail loud, never guess.

2

Whole-set diff, no code filter

_read_cert_promotable_events reads every active-season event and joins its organizer code. Diff by txt_code: CERT∖PROD → CREATE, CERT∩PROD → UPDATE, and PROD∖CERT → DELETE.

3

Field-ownership split

  • Source-owned identity—name, dates, location, country, organizer, weapons, EVF id and slug—is overwritten from CERT. Empty values keep current values through NULLIF, preventing empty unique slugs.
  • Admin-owned URLs, fees and registration are fill-blank-only; hand-entered values are never clobbered.
  • Status, tournaments and results are never touched; they belong to the results lifecycle and promote_event.
4

Guarded DELETE

Delete only when the PROD event is PLANNED and has zero result rows. Empty tournaments are dropped first in the same transaction. Results-bearing events absent from CERT are returned in delete_skipped for investigation and are never erased.

5

CREATE is childless

New events are created without tournament children. Results promotion creates tournaments on demand; skeleton-promoted domestic events are already childless.

6

Active-season mismatch guard

Reconciliation refuses to run when CERT and PROD have different active seasons. Without this guard, new-season CERT events could be assigned to PROD’s outgoing season while the outgoing set appeared eligible for wholesale deletion.

7

Human-readable run log

Every reconcile writes a run log to doc/staging/reconcile/{season}.{ts}Z.md and the CERT staging-reports bucket. A column-agnostic row diff separates Changes applied from Divergences NOT synced, with RPC counts and convergence in the summary.

What changes

Consequences

  • Self-healing. The first PROD run deleted the Samorin orphans and re-tagged the survivor EVF.
  • Domestic parity. PPW/MPW events now reach PROD incrementally.
  • Structural organizer safety. No organizer literal survives.
  • One owner. fn_import_evf_events is retired; event C/U/D belongs to the reconciler.
  • Regression guards. Season mismatch and empty-slug failures are tested.
  • Bounded cost. Full-set reads are acceptable because season tables contain only tens of rows.
