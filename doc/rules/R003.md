# R003: Age category by birth year

## Statement
A fencer's V-cat for a given season is `fn_age_category(int_birth_year,
season_end_year)` — single SQL implementation. Categories are V0 (30–39),
V1 (40–49), V2 (50–59), V3 (60–69), V4 (70+) where age is measured at
`EXTRACT(YEAR FROM tbl_season.dt_end)`. A fencer enters a category if
they turn the minimum age during the season's end year. Python's
`birth_year_to_vcat` is deleted in Phase 1 — there is exactly one
implementation, in SQL.

## Why
Two implementations (SQL `fn_age_category` vs Python `birth_year_to_vcat`)
drift over time. The 2026-03-26 season-end-year fix landed in SQL but
wasn't ported to Python for weeks. Single SQL implementation kills the
class.

## Trigger
- Per-row: every `tbl_result` insert/update goes through the V-cat
  invariant trigger (ADR-047) which calls `fn_age_category`.
- Per-pool: Stage 4 of the unified pipeline calls
  `fn_age_categories_batch(birth_years[], end_year)` for the splitter.
- Per-fencer query: ranking views call `fn_age_category` to compute
  the fencer's home V-cat for the active season.

## Edge cases
- NULL birth year → fencer's home category falls back to the
  tournament's `enum_age_category` in ranking views; the V-cat invariant
  trigger relaxes for fencers with NULL birth year.
- Estimated birth year (`bool_birth_year_estimated = TRUE`) participates
  in V-cat assignment normally, but admin should verify.
- A fencer aging into a new V-cat mid-season keeps their home V-cat for
  the whole season (driven by `dt_end`, not real-time age).
