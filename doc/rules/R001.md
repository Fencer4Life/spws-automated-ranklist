# R001: Combined-pool split

## Statement
When a tournament records a combined-V-cat pool (e.g. v0v1 or v0v1v2),
results MUST be split per V-cat at parse time, not stored as a single
multi-cat tournament. Each per-V-cat slice gets its own re-ranked
placements and its own `tbl_tournament` row, so ranking views see one
tournament-per-(weapon,gender,V-cat).

## Why
Without splitting, a v0v1v2 pool collapses three rankings into one and
fencers get scored against opponents from a different age category.
Historically this manifested as bug-class "missing scores" (combined-pool
splitter bug, 2026-04-29) where the splitter ran in the XML path but
not in FTL/Engarde/4Fence/Dartagnan/CSV — so 5 of 8 source paths produced
silently wrong rankings.

## Trigger
Stage 3 (combined-pool detection) of the unified pipeline:
fires when `ParsedTournament.raw_pool_size > number-of-fencers-in-one-V-cat`
or when source carries an explicit combined-V-cat marker (e.g. `v0v1v2`,
`(0)`/`(1)`/`(2)` placement annotations, EVF `categoryId` collapsing).

## Edge cases
- Joint pool (R002) is a different story: same physical pool, but the
  pool size IS the fencer count — no split needed; only the joint-pool
  flag fires.
- Empty V-cat slices (e.g. v0v1 with zero V0 entrants) MUST produce a
  zero-row `tbl_tournament` row, not omit the slice — downstream
  carry-over logic needs the row.
- V-cat invariant trigger (ADR-047) blocks any `tbl_result` row whose
  fencer's V-cat doesn't match the parent tournament's V-cat. R001 fires
  upstream of that trigger.
