# R004: Cross-gender scoring

## Statement
Tournaments where men and women fence together (some PEW/MEW/MSW
configurations) are scored as separate per-gender rankings — fencers are
only ever ranked against the same gender, regardless of pool/DE
co-fencing. The ingester routes results to the correct
`(weapon, gender, V-cat)` `tbl_tournament` slot.

## Why
Mixed-gender DEs happen on the strip but the ranking system is gendered.
Without explicit per-gender routing, women's results would land in
men's tournaments (or vice versa), polluting both rankings.

## Trigger
- Stage 2 (event resolution by date+weapon+gender) of the unified
  pipeline disambiguates by gender of the fencer, not the source's
  composite tournament.
- The V-cat invariant trigger (ADR-047) and parent-tournament gender
  must match the result's fencer gender.

## Implementation
- SQL view-level enforcement: `vw_score` and ranking views filter by
  `tbl_tournament.enum_gender` matching `tbl_fencer.enum_gender`.
- Reused — no rebuild changes; existing implementation passes
  pgTAP 14_cross_gender_scoring.sql.

## Tests
- pgTAP existing: 14.CG1–14.CG9 (cross-gender scoring suite).

## Edge cases
- A fencer with NULL `enum_gender` cannot be cross-gender-routed —
  they're scored against the tournament's declared gender; admin
  should backfill the gender via Identity Manager.
- Combined-pool splitter (R001) operates BEFORE gender routing — so a
  v0v1 mixed-gender pool produces 4 slices: V0-M, V0-F, V1-M, V1-F.
