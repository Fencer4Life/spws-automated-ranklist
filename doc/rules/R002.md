# R002: Joint-pool flag

## Statement
A "joint pool" is one physical pool that produces multiple
`tbl_tournament` rows (one per V-cat slice from R001) where every slice
shares the same `int_participant_count` (= the physical pool size, NOT
the per-slice fencer count). When this happens, every sibling row gets
`tbl_tournament.bool_joint_pool_split = TRUE`.

## Why
Carry-over and scoring formulas use `int_participant_count` for log-formula
math. If joint-pool slices report only their own slice size (e.g. 3 V0 + 4
V1 = 7 → V0 row reports `count=3`), scores diverge from what fencers see
in real-world standings (everyone fenced N=7 opponents). The flag exists
so views and scoring functions know "this `count` is shared across
sibling rows" without needing to re-derive it.

## Trigger
Stage 5 (joint-pool sibling detection) of the unified pipeline:
fires when ≥2 `tbl_tournament` rows share the same `(id_event, source_url,
weapon, gender)` after Stage 4 splitting.

## Edge cases
- Standalone V-cat tournament (one URL → one `tbl_tournament` row): flag
  stays FALSE.
- Empty sibling slice produced by R001: gets the flag too — its zero-row
  state plus `bool_joint_pool_split = TRUE` is the canonical "this V-cat
  was at the event but fielded no entrants" signal.
- PPW5-class events (no DB signal exists for retroactive detection per
  memory project_split_combined_pool_fix.md) are corrected by re-scrape,
  not backfill.
