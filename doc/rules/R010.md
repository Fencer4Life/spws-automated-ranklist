# R010: Frozen snapshot

## Statement
An event with `tbl_event.txt_source_status = 'FROZEN_SNAPSHOT'` is
ingested by **copying rows verbatim from cert_ref** instead of parsing
a live source. The unified pipeline detects FROZEN_SNAPSHOT at Stage 0
and routes the event around stages 1–7 entirely; the resulting
`tbl_*_draft` rows are byte-identical to `cert_ref.tbl_*` for that
event.

## Why
Some events have no live source (results lost, source platform shut
down, only PDF available, etc.). Re-deriving them from the
`seed_prod_<latest>.sql` snapshot is the only honest answer — they are
historical data, not re-ingestable. The flag makes "no source"
explicit and auditable, rather than masquerading as a normal event
with stale URLs.

## Trigger
- Stage 0 (pre-parse) of the unified pipeline: when
  `tbl_event.txt_source_status = 'FROZEN_SNAPSHOT'`, branch to the
  copy path (`python/pipeline/copy_from_prod.py`) instead of parsing.

## Edge cases
- An event flipped to FROZEN_SNAPSHOT after partial ingest: the copy
  path TRUNCATEs draft tables for that run, then copies; previous
  draft state is discarded.
- cert_ref rows for the event don't exist (e.g. event was created
  post-seed): admin must either provide a source or accept that the
  event has no historical baseline. Pipeline halts.
- Phase 6 drops cert_ref schema — at that point all FROZEN_SNAPSHOT
  events are committed and live in `public.*`; the flag becomes
  reference-only (preserved for audit, no behavioral impact).
