# R009: URL‚Üídata validation

## Statement
On every write of `tbl_event.url_event*` or `tbl_tournament.url_results`,
the pipeline MUST scrape the URL and confirm that the date / event name
/ weapon / category in the scraped page matches the row being written.
On mismatch, REJECT the write with a clear error ‚Äî never silently
overwrite. pgTAP/pytest tests assert this enforcement.

## Why
URLs are admin-managed (per data-integrity hard rule in conventions.md);
typos and copy-paste mistakes are common. Without validation, an admin
can paste a URL pointing to a different event into a row, then later
the cron scrapes the wrong URL and overwrites real results with
unrelated data. URL‚Üídata validation is the bottleneck where the wrong
URL gets caught.

## Trigger
- On admin Event/Tournament edit: pre-save validation against the
  scraped page.
- On `fn_ingest_tournament_results` for a row whose
  `url_results` was just changed: re-validate before the ingest runs.
- On EVF refresh path (`fn_refresh_evf_event_urls`): only fills NULL
  slots; never overwrites ‚Äî this rule covers the explicit-write paths.

## Edge cases
- Scrape failure (network, source temporarily down): admin must be
  able to override with a clear "I know this is right" override path ‚Äî
  but the override is auditable in `tbl_audit_log`.
- Multi-slot URLs (ADR-040): each slot validates independently; only
  the changed slot triggers re-validation.
- EVF organizer events: URL‚Üídata tolerates the well-known EVF API
  date conventions (e.g. start vs end date may differ from
  `dt_start`/`dt_end` by ¬±1 day).
