# R011: Source priority by organizer (EVF backup-only)

## Statement
For events organized by SPWS or FIE, the recorded `url_event*` /
`url_results` URLs are the **primary** ingestion source. EVF API is a
**backup** source — used only when the recorded URL is unreachable, OR
for events where the EVF API is the recorded source itself. After every
EVF-organized event commits, a parity gate (ADR-053) cross-checks the
EVF API against the committed `tbl_result` rows and routes failures to
`enum_status = 'SCORED'` with notes in `tbl_event.txt_parity_notes`.

## Why
The recorded URL is what the admin verified. EVF API is a convenient
fallback but is not authoritative for SPWS/FIE events — its data may
diverge from on-site posting. By treating the EVF API as backup-only,
we avoid drifting away from the admin-curated source. The parity gate
on EVF events validates the engine itself: if the unified pipeline
agrees with EVF API on EVF-organized events, we have transitive trust
that it also handles SPWS/FIE events correctly.

## Trigger
- Stage 0 (source dispatch): the operator's source choice (interactive
  CLI prompt) drives source priority; default is "use recorded URL".
- Post-commit: parity gate fires automatically for events where
  `tbl_event.id_organizer = EVF`.

## Edge cases
- EVF API rate limits (open risk #3): batch the parity calls and
  back off on 429.
- An EVF-organized event missing from EVF API (deleted upstream):
  parity gate produces a "no API data" annotation; commit still
  proceeds. Admin reviews.
- A parity failure on a single fencer's score (within ±0.5 tolerance):
  passes. Larger drift fails the gate.
