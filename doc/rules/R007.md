# R007: Rolling-score 366-day cap

## Statement
Rolling carry-over for the active season looks back at most
`tbl_season.int_carryover_days` days (default 366). Results older than
the cap are excluded from the active season's rolling ranking, even
if they would otherwise carry over via the rules-based engine. The
column exists today; cap **enforcement** lands in Phase 7 (ADR-054)
alongside the FK-based carry-over engine.

## Why
Without a cap, a fencer's results from 5+ seasons ago could weight
the current ranking. 366 days is "the most recent full season cycle
plus one buffer day for late results imports". Configurable per
season because some categories (IMEW biennial) need 2-year windows.

## Trigger
- At ranking compute time: `fn_ranking_ppw` / `fn_ranking_kadra` with
  `p_rolling = TRUE` filter joined results by `dt_tournament >=
  CURRENT_DATE - tbl_season.int_carryover_days`.

## Implementation
- Column (existing): `tbl_season.int_carryover_days INT NOT NULL DEFAULT 366`.
- Enforcement: TBD — Phase 7 (ADR-054 — adds the date filter to
  ranking functions).

## Tests
TBD — Phase 7 (assertion: a rolling ranking with a 366-day cap drops
results from > 366 days ago even if they're declared counterparts).

## Edge cases
- IMEW biennial (R008): some seasons run with `int_carryover_days =
  730+` to allow IMEW (every 2 years) to carry forward the alternate
  season. Admin sets per-season.
- Future-dated tournaments (`dt_tournament > today`): excluded by the
  cap formula's lower bound; no changes needed.
