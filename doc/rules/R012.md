# R012: Engarde multilingual handling

## Statement
The Engarde HTML parser MUST be locale-agnostic — it parses results
correctly from EN / FR / ES / IT / DE / PL / HU exports without
locale-specific code branches. The parser uses positional column order
and language-independent markers (numerical placements, ISO date
formats where present), not translated header strings.

## Why
SPWS imports Engarde results from many countries (EVF events
especially). Each Engarde install can be set to a different language
and exports follow that locale. A parser that depends on English
header strings ("Place", "Name", "Birth Year") would silently break
on a French ("Place", "Nom", "Année de naissance") or German
("Platz", "Name", "Geburtsjahr") export.

## Trigger
- Engarde parser (`python/scrapers/engarde.py`): every parse call.

## Edge cases
- Polish `ł`/`Ł` not decomposed by NFD: matcher's
  `fold_diacritics` handles this explicitly (R006 implementation).
  Engarde parser preserves source bytes; folding happens at match time.
- Non-Latin scripts (Greek, Cyrillic) — out of scope; Engarde events
  pulling from these locales aren't currently in SPWS calendar.
- Engarde "team" exports (DMEW): different table structure, not
  parsed by individual-results path.
