# R008: Carry-over FK linkage

## Statement
Carry-over from prior seasons MUST be driven by `tbl_event.id_prior_event`
(an FK to the prior season's event row), not by `txt_code` prefix
matching. Each season has exactly one declared "prior counterpart" event
per recurring kind; ranking carry-over walks that FK chain to find
results to surface as carried-over scores.

## Why
The legacy `EVENT_CODE_MATCHING` engine matches events by `txt_code`
prefix — fragile against renames, missing for one-off events, and
inconsistent for biennial kinds (IMEW). FK linkage is explicit and
admin-auditable. The dispatcher pattern (ADR-042) lets a season opt
into FK matching while older seasons keep the legacy engine until
Phase 7 default-flips to FK.

## Trigger
- At ranking compute time: when `tbl_season.enum_carryover_engine =
  'EVENT_FK_MATCHING'`, the rolling-score path walks `id_prior_event`
  to find counterparts.
- Pre-Phase-7 seasons stay on `EVENT_CODE_MATCHING` (default). New
  seasons created post-Phase-7 default to `EVENT_FK_MATCHING`.

## Edge cases
- Manual FK relinking (admin-corrected `id_prior_event`) MUST be
  preserved — the seed import idempotently updates the FK column.
- Skeleton events (`enum_status = 'CREATED'`) carry an
  `id_prior_event` pointer set at season-init wizard time
  (ADR-044 atomic transaction).
- DMEW/IMEW biennial alternation: see project memory
  `project_dmew_imew_alternation.md` — only IMEW scores results, but
  both events appear on the calendar.
